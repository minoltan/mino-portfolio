Use Case 1 : Simple SNS

🔹 Step 1: Create an SNS Topic

  1. Log in to the AWS Console, and go to the Simple Notification Service (SNS).
  2. Click on “Topics” in the left sidebar.
  3. Click “Create topic”.
  4. Choose the Topic type:
  • Select Standard (high throughput, best-effort ordering, at-least-once delivery).
  • You could also select FIFO if you need strict ordering and exactly-once delivery (requires .fifo suffix).

5. Enter a name for your topic — e.g., MyFirstTopic.

6. Optional:

  • Enable encryption using AWS KMS if needed.
  • Access policy: Leave this as default unless you need cross-account or service-to-service access.

7. Click “Create topic”.

✅ Done! Your topic is ready.

🔹 Step 2: Create a Subscription (Using Email)

  1. In your newly created topic page, click “Create subscription”.
  2. Choose Protocol: Select Email.
  3. Enter Endpoint: Input the email address you want to subscribe (e.g., issackpaul95@gmail.com).
  4. Optional:
  • Subscription filter policy: This lets you filter messages that specific subscribers receive — skip it for now.

5. Click “Create subscription”.

✅ Your subscription is created, but it’s in “Pending confirmation” status.

🔹 Step 3: Confirm the Subscription

  1. Open your email and access the inbox.
  2. Open the confirmation email from AWS SNS.
  3. Click “Confirm subscription”.

🔄 Go back to the AWS Console and refresh the subscription page. You should now see the status as “Confirmed”.

🔹 Step 4: Publish a Message

Now let’s send a message to test the SNS topic!

  1. In the MyFirstTopic page, click “Publish message”.
  2. Fill in:
  • Subject: e.g., Test Notification
  • Message body: e.g., Hello, world!

3. Leave other options as default.

4. Click “Publish message”.

💡 Within seconds, you should receive the message in the email inbox you subscribed.

🔹 Step 5: Clean Up (Optional but Recommended)

Once done testing, it’s good practice to clean up:

  1. Delete the subscription:
  • Go to your topic → Subscriptions tab.
  • Select the subscription and click “Delete”.

2. Delete the topic:

  • Go back to Topics.
  • Select MyFirstTopic.
  • Click “Delete” and confirm by typing delete me.

✅ Clean and tidy!

Use Case 2 : SNS + SQS — Fan Out Pattern

This pattern allows you to broadcast a message to multiple SQS queues through a single SNS topic. It’s great for microservices and loosely coupled systems.

✅ What You’ll Build:

  • One SNS topic
  • Two (or more) SQS queues
  • Both queues subscribed to the SNS topic

🔸 Steps:

  1. Create SNS Topic
     If not already created, repeat Use Case 1 from Part 1 (e.g., MyFanoutTopic).
  2. Create SQS Queues
  • Go to Amazon SQS → Create queue
  • Choose Standard queue
  • Name the queue (e.g., QueueA, then repeat for QueueB)
  • Leave settings as default and Create queues
  • For Queue Hands-On — Click Here

3. Subscribe Queues to SNS Topic

  • Go back to your SNS topic (MyFanoutTopic) → Create subscription
  • Protocol: Amazon SQS
  • Endpoint: Select ARN of QueueA
  • Repeat for QueueB

4. Confirm Subscription Permissions
 SNS needs permission to publish to SQS:

  • SNS sets this up automatically with access policies, but if it fails, manually edit the SQS access policy to allow the SNS topic ARN.

5. Publish a Test Message

  • Go to MyFanoutTopic → Publish message
  • Enter Hello Fan Out! and publish
  • Check QueueA and QueueB — both should receive the message

✅ Success! You’ve implemented the SNS → SQS fan-out pattern.

Use Case 3: Create an SNS FIFO Topic

FIFO (First-In-First-Out) topics guarantee strict message ordering and exactly-once delivery — ideal for financial or transactional systems.

🔸 Steps:

  1. Go to SNS → Topics → Create topic
  2. Select FIFO topic
  3. Name it (must end with .fifo, e.g., MyFIFOTopic.fifo)
  4. Choose a Content-Based Deduplication option:
  • ON = deduplication based on message body
  • OFF = must provide MessageDeduplicationId on publish

5. Click Create topic

🧠 Note: Only SQS FIFO queues can subscribe to SNS FIFO topics.

Use Case 4: SNS FIFO + SQS FIFO — Fan Out

Now let’s connect a FIFO SNS Topic to multiple FIFO SQS Queues for an ordered fan-out system.

🔸 Steps:

  1. Create FIFO SQS Queues
  • Go to SQS → Create queue
  • Type: FIFO
  • Name: Must end with .fifo, e.g., Queue1.fifo, Queue2.fifo
  • Enable content-based deduplication if preferred

2. Create FIFO SNS Topic

  • Repeat Use Case 3 above to create MyFIFOTopic.fifo

3. Subscribe FIFO Queues

  • Go to MyFIFOTopic.fifo → Create subscription
  • Protocol: Amazon SQS
  • Endpoint: Use ARN of Queue1.fifo, then repeat for Queue2.fifo

4. Publish a FIFO Message

  • Go to MyFIFOTopic.fifo → Publish message
  • Provide:
  • Message group ID (required, e.g., group1)
  • Message deduplication ID (if required)
  • Body: FIFO message test
  • Click Publish

🎯 Check the SQS FIFO queues — the order and uniqueness will be preserved.

Use Case 5: SNS — Message Filtering

With message filtering, subscribers only receive messages they care about, reducing noise and increasing efficiency.

✅ What You’ll Build:

  • One SNS topic
  • Two SQS queues
  • Subscriptions with filter policies (e.g., messages tagged by eventType)

🔸 Steps:

  1. Create SNS Topic (e.g., FilteredTopic)

2. Create Two SQS Queues (check previous steps): OrderQueue, ShippingQueue

3. Subscribe Both Queues to Topic

  • Go to FilteredTopic → Create subscription
  • Protocol: Amazon SQS
  • Select OrderQueue ARN
  • Click “Create subscription”

4. Add Filter Policy

  • Click on the OrderQueue subscription
  • Choose Edit subscription filter policy
  • Add filter:
{
  "eventType": ["order"]
}
  • Repeat for ShippingQueue:
{
  "eventType": ["shipping"]
}

5. Publish a Message with Attributes

  • Go to FilteredTopic → Publish message
  • Enter a message body: Order created
  • Add Message Attribute:
  • Key: eventType
  • Type: String
  • Value: order
  • Click Publish

🧪 Only OrderQueue should receive this message.

Repeat with eventType: shipping — now only ShippingQueue should get it.
