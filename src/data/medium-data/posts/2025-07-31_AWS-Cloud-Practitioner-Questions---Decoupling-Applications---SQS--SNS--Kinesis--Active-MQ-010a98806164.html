Question 1:

You have an e-commerce website and you are preparing for Black Friday which is the biggest sale of the year. You expect that your traffic will increase by 100x. Your website already using an SQS Standard Queue, and you’re running a fleet of EC2 instances in an Auto Scaling Group to consume SQS messages. What should you do to prepare your SQS Queue?

Correct Answer : (4) Amazon SQS (Simple Queue Service) is designed to handle varying volumes of requests seamlessly without requiring manual adjustments or pre-warming, making it ideal for significant traffic spikes like those expected on Black Friday. This understanding aligns with the learning objective of recognizing the scalability features of AWS services.

🏗️ Imagine a Factory (Your Serverless App)

  • Customers (Users) place orders (events) very quickly (traffic spike like Black Friday).

Without SQS

  • Orders go directly to the workers (Lambdas).
  • If too many orders come at once, workers may get overwhelmed or you may hit concurrency limits.
  • Some orders could be lost or delayed.

With SQS

  • You add a warehouse (SQS Queue) in between.
  • Customers put orders into the warehouse first (SQS stores messages durably).
  • Workers (Lambdas) pick up orders from the warehouse at a steady rate.
  • If more orders come than workers can handle, the warehouse just stacks them up safely (buffering).
  • Workers can scale up (Lambda scaling) when needed and clear the backlog.

✅ So:

  • SQS = the warehouse buffer ensuring no orders are lost and workers are not overwhelmed.
  • Lambda = the workers that can scale up quickly, but still rely on the warehouse to handle unexpected traffic surges.

Question 2:

You have an SQS Queue where each consumer polls 10 messages at a time and finishes processing them in 1 minute. After a while, you noticed that the same SQS messages are received by different consumers resulting in your messages being processed more than once. What should you do to resolve this issue?

Correct Answer : (3) By increasing the Visibility Timeout, you ensure that once a consumer processes a message, it remains hidden from other consumers for a longer period, reducing the chances of duplicate message processing. This adjustment allows the consumer ample time to complete the processing without risk of the message being returned to the queue prematurely.

📌 Scenario

You have an online store.

  • Customers place orders (messages).
  • Orders go into SQS (the queue).
  • Workers (consumers) pick up orders to process them (packaging & shipping).

1️⃣ Without Visibility Timeout Problem

  1. Customer A places an order → message Order#123 is added to SQS.
  2. Worker 1 picks Order#123 to process (SQS hides it from others).
  3. Worker 1 takes 2 minutes to process the order.
  4. But the Visibility Timeout was set to 30 seconds.
  5. After 30 seconds, SQS thinks Worker 1 failed → Order#123 becomes visible again.
  6. Worker 2 also picks Order#123 and processes it → Order shipped twice! 🚚🚚

2️⃣ With Correct Visibility Timeout

  1. The same Order#123 is picked by Worker 1.
  2. This time, the Visibility Timeout is set to 5 minutes (longer than processing time).
  3. Order#123 stays hidden from other workers while Worker 1 is processing it.
  4. Worker 1 completes the process and sends a DeleteMessage to SQS.
  5. SQS permanently deletes Order#123.
  6. No other worker can process it → ✅ no duplicate shipments.

🟢 Why This Matters

  • SQS does not assign messages to a single consumer permanently.
  • It only temporarily hides the message for the duration of Visibility Timeout.
  • If the consumer does not confirm (delete) in time, SQS gives the same order to another worker to ensure reliability.

Question 6:

You have a website where you want to analyze clickstream data such as the sequence of clicks a user makes, the amount of time a user spends, and where the navigation begins and how it ends. You decided to use Amazon Kinesis, so you have configured the website to send these clickstream data all the way to a Kinesis data stream. While you checking the data sent to your Kinesis data stream, you found that the users’ data is not ordered and the data for one individual user is spread across many shards. How would you fix this problem?

Correct Answer : (3) By adding a partition key that represents the user’s identity, you ensure that all records related to a single user are routed to the same shard in Kinesis. This preserves the order of each user’s actions, which is crucial for accurately analyzing clickstream data.
