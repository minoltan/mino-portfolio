In this hands-on demo, we’ll explore how to use AWS App Runner, a fully managed service that simplifies deploying web applications and APIs. Since this is outside the free tier and incurs costs (e.g., per vCPU and GB of memory), it’s best to follow along conceptually unless you’re ready to pay. Let’s walk through deploying a basic HTTP server to see App Runner in action.

Step 1: Creating an App Runner Service

  1. Access the Console: Navigate to the AWS App Runner service in the AWS Management Console.

2. Select Deployment Source: Choose the source for your deployment. Options include a Container Registry (e.g., Amazon ECR) or Source Code (e.g., GitHub). For this demo, we’ll use a public container image.

  • Visit the Amazon ECR Public Gallery, search for httpd (a simple Apache HTTP server), and copy the image URL (e.g., public.ecr.aws/httpd:latest).
  • Paste the URL into the App Runner console under Container Registry.

3. Deployment Settings: Opt for Manual deployment since we’re using a pre-built image (automatic deployment works with source code or ECR triggers). Click Next.

Step 2: Configuring the Service

  1. Service Name: Name it DemoHTTP.
  2. Resource Settings: Set the minimum resources — 1 vCPU and 2 GB memory (the lowest tier available).
  3. Port Configuration: Check the httpd documentation (or assume port 80 for HTTP). Set the port to 80 to match the container’s default.

4. Auto Scaling: Enable the default configuration, allowing scaling from 1 to 25 instances with 100 concurrent requests per instance. This ensures the app handles traffic spikes.

5. Health Checks: Leave as default to monitor the app’s health.

6. Security and Networking: Skip instance roles and use public access (no custom VPC for simplicity). Disable AWS X-Ray tracing for now.

7. Review and Deploy: Verify settings (public httpd image, port 80) and click Create and Deploy. App Runner handles the rest — deploying an auto-scaling group, scaling triggers, target containers, a load balancer, and a domain.

Step 3: Monitoring and Accessing

  • Once deployed (this may take a few minutes), click the service to find the default domain (e.g., *.awapp.io).
  • Open the URL in a browser to see the “It works!” message, confirming the HTTP server is live.
  • Explore the Event Log to track deployment, health checks, and scaling events.
  • Use the interface for logs, metrics, observability, and custom domain setup — all in one place.

Step 4: Cleanup

  • To avoid costs, select the service, type delete, and confirm. This removes the deployment.

Key Takeaways

App Runner automates the heavy lifting — building, deploying, and scaling your container — making it incredibly easy to launch production-ready web apps. Its all-in-one dashboard provides visibility and control, and auto-scaling ensures performance during traffic surges. This simplicity is perfect for rapid deployments without infrastructure expertise.

Conclusion

AWS App Runner proves its power with this quick setup of an httpd server, showcasing its ability to handle containerized apps seamlessly. While costs apply, its ease of use and robust features make it a standout for developers. Experiment cautiously, and stay tuned for more AWS container insights in the next lecture!
