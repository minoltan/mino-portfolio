Introduction

In this hands-on blog post, we’ll walk through using Amazon Athena to query data stored in an Amazon S3 bucket, focusing on analyzing server access logs. Athena is a serverless query service that lets you run standard SQL queries on data in S3 without moving it or managing infrastructure. We’ll set up Athena, create a database and table, and run queries to extract insights from S3 access logs, such as HTTP status counts and unauthorized access attempts. This guide is perfect for AWS Certified Solutions Architect Associate (SAA-C03) exam preparation or anyone looking to analyze S3 data effortlessly.

Prerequisites

  • AWS account with permissions for S3, Athena, and IAM.
  • AWS CLI installed (optional for manual steps).
  • An S3 bucket with server access logs enabled (we’ll create one for results).

Step 1: Create and Upload Sample Log File to S3

Athena requires an S3 bucket to store query results and another bucket for the data to query. We’ll create a bucket for access logs, upload a sample server access log file, and set up a bucket for query results.

  1. Create a Sample Log File:
  • Create a file named access_log_sample.txt on your local machine with the following content:

2. Create S3 Buckets:

  • Go to the S3 Console.
  • Create a bucket for logs named mino-access-logs-v1 in your preferred region (e.g., ap-southeastl-1).

3. Upload the Log File:

  • In the S3 Console, navigate to my-access-logs and upload access_log_sample.txt to the root of the bucket.

Step 2: Set Up Query Result Location in S3

Athena requires an S3 bucket to store query results. Let’s create one and configure Athena to use it.

  1. Go to the S3 Console and create a bucket:
  • Name: aws-athena-results-<your-name>-<region> (e.g., aws-athena-results-mino-ap-southeast-1-v1).
  • Region: Choose your region (e.g., eu-central-1).
  • Keep default settings and create the bucket.

2. In the Athena Console, open the Query Editor.

3. Go to Settings > Manage > Query result location.

4. Enter the S3 bucket path: s3://aws-athena-results-<your-name>-<region>/.

  • Alternatively, click Browse S3 to select the bucket and avoid typos.

5. Save the settings. All query results will now be stored in this bucket.

Step 3: Create a Database in Athena

Before querying data, we need to create a database in Athena to organize our tables.

  1. In the Athena Query Editor, select the default database.
  2. Run the following SQL query to create a database:
CREATE DATABASE s3_access_logs_db;

3. Verify the database appears in the left-hand panel under Databases. Select s3_access_logs_db for subsequent queries.

Step 3: Create a Table for S3 Access Logs

We’ll create a table in Athena to represent server access logs stored in an S3 bucket (e.g., s3://mino-access-logs-v1/). The logs contain fields like bucket owner, request date, IP address, and HTTP status.

  1. Identify your S3 bucket with access logs (e.g., s3://mino-access-logs-v1/). Ensure server logging is enabled on the bucket.
  2. In the Athena Query Editor, use the following SQL query from the AWS S3 Athena documentation to create a table. Replace the LOCATION with your bucket path, ensuring a trailing slash:
CREATE EXTERNAL TABLE s3_access_logs_db.mybucket_logs(
         BucketOwner STRING,
         Bucket STRING,
         RequestDateTime STRING,
         RemoteIP STRING,
         Requester STRING,
         RequestID1 STRING,
         RequestID2 STRING,
         Operation STRING,
         Key STRING,
         RequestURI STRING,
         HTTPstatus STRING,
         ErrorCode STRING,
         BytesSent BIGINT,
         ObjectSize BIGINT,
         TotalTime STRING,
         TurnAroundTime STRING,
         Referrer STRING,
         UserAgent STRING,
         VersionId STRING,
         HostId STRING,
         SigV STRING,
         TLSVersion1 STRING,
         Dash STRING,
         EndPoint STRING,
         TLSVersion2 STRING
) 
ROW FORMAT SERDE 'org.apache.hadoop.hive.serde2.RegexSerDe'
WITH SERDEPROPERTIES (
         'serialization.format' = '1', 
         'input.regex' = '([^ ]+) ([^ ]+) \\[([^]]+)\\] ([^ ]+) ([^ ]+) ([^ ]+) ([^ ]+) ([^ ]+) ([^ ]+) "([^"]+)" ([^ ]+) ([^ ]+) ([^ ]+) ([^ ]+) ([^ ]+) ([^ ]+) "([^"]*)" "([^"]*)" ([^ ]+) ([^ ]+) ([^ ]+) ([^ ]+) ([^ ]+) ([^ ]+) ([^ ]+).*'
)
LOCATION 's3://mino-access-logs-v1/';

3. Run the query. Verify the table access_logs appears under s3_access_logs_db in the left panel.

Step 4: Preview the Table

Let’s explore the data by previewing the table to confirm it’s set up correctly.

  1. In the Athena Query Editor, find the access_logs table in the left panel.
  2. Click the three dots next to the table and select Preview Table.

3. Athena runs a query to fetch 10 rows:

SELECT * FROM "s3_access_logs_db"."mybucket_logs" limit 10;

4. Check the results in the lower panel. You should see columns like bucketowner, bucket_name, requestdatetime, remoteip, httpstatus, etc.

5. Verify the data matches your S3 access logs. This confirms the table is correctly mapped to your S3 bucket.

Step 5: Run Analytical Queries

Now, let’s perform analytics on the access logs to gain insights, such as counting requests by HTTP status or identifying unauthorized access attempts.

  1. Create a new query tab in the Athena Query Editor.
  2. Run an aggregation query to count requests by HTTP status and operation:
SELECT httpstatus, operation, COUNT(*) as request_count
FROM "s3_access_logs_db"."mybucket_logs" 
GROUP BY httpstatus, operation;

3. Check the results. Example output:

  • 404 (Not Found): 142 requests for GET operations.
  • 200 (OK): 101 requests for HEAD operations.
  • Empty rows: ~200,000 (indicating incomplete logs). Note the data scanned (e.g., 30 MB) in the query details.

4. Investigate unauthorized access (HTTP 403 errors):

SELECT requestdatetime, remoteip, requesturi
FROM "s3_access_logs_db"."mybucket_logs" 
WHERE httpstatus = '403';

5. Review results to identify IPs or URIs causing unauthorized access. This helps detect potential security issues or misconfigurations.

6. Save results to S3 (automatically stored in s3://aws-athena-results-<your-name>-<region>/) for further analysis.

Example: Analyzing Access Logs

Imagine you’re running a website, and your S3 bucket (s3://my-access-logs/) stores server access logs. You notice users reporting “Not Found” errors. Using Athena:

  • You run the HTTP status query and find 142 404 errors, indicating broken links or missing files.
  • You query 403 errors and discover unauthorized access attempts from a specific IP, prompting you to update your bucket policy.
  • You connect Athena to Amazon QuickSight to visualize error trends in a dashboard, helping your team monitor site health.

This serverless approach makes it easy to analyze logs without manually parsing files or setting up servers.

Tips for Optimization

  • Use Parquet/ORC: Convert logs to columnar formats with AWS Glue to reduce data scanned.
  • Partition Data: Organize logs by date (e.g., s3://my-access-logs/year=2025/month=01/) for faster queries.
  • Compress Files: Use GZIP or Snappy to minimize file size.
  • Larger Files: Combine small log files into larger ones (e.g., 128 MB) to reduce overhead.

Conclusion

Amazon Athena makes analyzing S3 data, like server access logs, incredibly simple and serverless. By setting up a query result bucket, creating a database and table, and running SQL queries, you can extract insights like HTTP status trends or unauthorized access attempts in minutes. This hands-on guide walked you through querying S3 access logs, a common use case for the AWS Certified Solutions Architect Associate (SAA-C03) exam. With Athena, you avoid the hassle of manual file parsing or server management, making it a go-to tool for ad hoc analytics and log analysis.
