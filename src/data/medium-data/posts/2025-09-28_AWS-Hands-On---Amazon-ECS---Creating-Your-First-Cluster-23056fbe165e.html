In this hands-on guide, we’ll walk through setting up your first Amazon Elastic Container Service (ECS) cluster using the AWS Management Console. By the end, you’ll have a functional ECS cluster configured with both Fargate and EC2 launch types, ready to run containerized workloads. Let’s dive in!

Step 1: Accessing the ECS Console

  1. Log in to the AWS Management Console.
  2. Navigate to the Amazon ECS service.

Step 2: Creating an ECS Cluster

  1. In the ECS console, go to Clusters and click Create Cluster.
  2. Name the cluster, e.g., DemoCluster, and leave the default namespace as is.
  3. Configure the infrastructure options. ECS offers three choices:
  • AWS Fargate: A serverless option where AWS manages the compute resources for your containers. You simply define your tasks, and AWS runs them on demand without visible infrastructure.
  • Amazon EC2 Instances: You provision and manage your own EC2 instances to provide compute for your containers.
  • External Instances (ECS Anywhere): Run ECS containers on your own on-premises infrastructure or data center.

For this demo, enable both Fargate and Amazon EC2 Instances to explore both launch types.

Step 3: Configuring EC2 Instances

Since we selected the EC2 launch type, we need to configure the EC2 instances:

  1. Auto Scaling Group: Create a new auto-scaling group for the ECS cluster.
  2. Operating System: Choose Amazon Linux 2 (or Amazon Linux 2023 if preferred).
  3. Instance Type: Select t2.micro (free-tier eligible) for cost efficiency.
  4. Capacity Settings:
  • Desired Capacity: Set to 0 initially.
  • Minimum Capacity: 0.
  • Maximum Capacity: 5.

5. SSH Key Pair: Skip for now (no SSH access needed).

6. Root EBS Volume: Leave the default size.

7. Network Settings:

  • Use the default VPC.
  • Select all available subnets (e.g., across three Availability Zones).
  • Choose the default security group.
  • Set Auto-assign Public IP to use the subnet’s default setting.

8. Skip monitoring and tags for simplicity.

9. Click Create to provision the cluster.

Step 4: Understanding the Auto Scaling Group

While the cluster is being created, check the Auto Scaling Groups in the EC2 console:

  1. Navigate to EC2 > Auto Scaling Groups.
  2. Look for a group named something like Infra-ECS-Cluster.
  3. Verify the settings:
  • Desired Capacity: 0.
  • Minimum Capacity: 0.
  • Maximum Capacity: 5.
  • Spans multiple Availability Zones for high availability.

This auto-scaling group was automatically created by ECS to manage EC2 instances for your cluster.

Step 5: Exploring the ECS Cluster

Once the cluster creation is complete:

  1. Click on DemoCluster in the ECS console.
  2. Check the Services and Tasks tabs (both should show 0, as no tasks are running yet).
  3. Go to the Infrastructure tab to view the capacity providers:
  • FARGATE: Allows running serverless tasks.
  • FARGATE_SPOT: Runs Fargate tasks on cost-effective Spot Instances.
  • ASGProvider: Manages EC2 instances via the auto-scaling group.

Step 6: Scaling Up the EC2 Capacity

To demonstrate EC2 instance registration:

  1. In the Infrastructure tab, select the ASGProvider and click Details.
  2. Navigate to EC2 > Auto Scaling Groups.
  3. Look for a group named something like Infra-ECS-Cluster.
  4. Edit the Desired Capacity to 1.

5. This triggers the creation of a single t2.micro EC2 instance, which registers itself with the DemoCluster.

6. Once the instance is in the InService state, refresh the ECS console.

7. Under Container Instances, you’ll see the new instance with:

  • CPU Available: 1,024 units.
  • Memory Available: 982 MB.
  • Running Tasks: 0 (no tasks launched yet).

This instance is now ready to host ECS tasks, alongside the Fargate and Fargate Spot capacity providers.

Key Takeaways

  • You’ve created an ECS cluster (DemoCluster) with both Fargate and EC2 launch types.
  • The cluster includes three capacity providers: Fargate, Fargate Spot, and an EC2 auto-scaling group.
  • EC2 instances register automatically with the cluster via the ECS Agent.
  • You can launch tasks on either Fargate (serverless) or EC2 instances, depending on your needs.

Hands-On with Amazon ECS: Creating and Scaling an ECS Service

In this hands-on guide, we’ll build on our previous setup of an Amazon ECS cluster (DemoCluster) and create an ECS service using a task definition. We’ll deploy a simple Nginx-based application using AWS Fargate, integrate it with an Application Load Balancer (ALB), and demonstrate scaling the service. Let’s get started!

Step 1: Creating a Task Definition

Before launching an ECS service, we need a task definition to specify how our container should run.

  1. In the ECS console, navigate to Task Definitions and click Create new Task Definition.

2. Name the task definition, e.g., nginxdemos-hello. This name reflects the Docker image we’ll use: nginxdemos/hello from Docker Hub.

3. Infrastructure Requirements:

  • Choose AWS Fargate for serverless compute. (You could also select EC2 instances, but we’ll stick with Fargate for simplicity.)
  • Select Linux as the operating system and architecture.

4. Task Size:

  • Set vCPU to 0.5 and memory to 1 GB to keep costs low. (Fargate supports up to 16 vCPUs and 120 GB of memory for larger workloads.)

5. Task Role:

  • Skip the task role for now, as our Nginx container doesn’t need to make AWS API calls. (Use a task role if your container needs access to AWS services like S3 or DynamoDB.)

6. Task Execution Role:

  • Leave as default. If the ECS task execution role doesn’t exist, AWS will create it automatically.

7. Container Details:

  • Name: nginxdemos-hello.
  • Image URL: nginxdemos/hello (pulled from Docker Hub).
  • Port Mappings: Map port 80 (host) to port 80 (container) for HTTP traffic.
  • Leave other settings (e.g., resource limits, environment variables, logging) as default.

9. Storage:

  • Use the default 21 GB of ephemeral storage provided by Fargate.
  1. Click Create to generate the task definition (version 1 for new setups).

Step 2: Creating an ECS Service

With the task definition ready, let’s create a service to run it on our DemoCluster.

  1. In the ECS console, go to Clusters, select DemoCluster, and click Create under the Services tab.

2. Service Details:

  • Task Definition Family: Select nginxdemos-hello and the latest revision (e.g., version 2 if you’ve recreated it, otherwise version 1).
  • Service Name: Keep the default or customize (e.g., nginxdemos-hello-service).
  • Desired Tasks: Set to 1 to start with one container (we’ll scale later).

3. Compute Configuration:

  • Use the default capacity provider strategy (Fargate).
  • Use custom model, select FARGET

4. Deployment Configuration:

  • Choose Replica to maintain a fixed number of tasks across the cluster.
  • Leave Availability Zone (AZ) rebalancing enabled to ensure tasks are distributed evenly across AZs.

5. Networking:

  • Select the default VPC and all available subnets.
  • Create a new security group (e.g., nginxdemos-sg) and allow HTTP traffic (port 80) from anywhere (0.0.0.0/0).
  • Enable public IP assignment for Fargate tasks.

6. Load Balancing:

  • Enable Application Load Balancer (ALB).
  • Create a new ALB named DemoALBForECS with a listener on port 80.
  • Create a new target group named nginxdemosTG targeting port 80.

7. Skip VPC Lattice, auto-scaling, and volume configurations for now.

8. Click Create to deploy the service.

Step 3: Verifying the Service

Once the service is deployed:

  1. Go to the Services tab in DemoCluster and select the new service.
  2. Check the status:
  • Desired Tasks: 1.
  • Running Tasks: 1.
  • Status: Active.

3. Verify the target group (nginxdemosTG):

  • It’s linked to the ALB (DemoALBForECS).
  • One IP address (the private IP of the Fargate task) is registered as a healthy target.

4. Copy the DNS name of the ALB, open it in a browser, and confirm the Nginx welcome page loads successfully.

5. Explore the service’s Tasks tab:

  • Click the running task to view its configuration, private IP, and container logs.

6. Check the Events tab to see service events, such as task registration and deployment completion.

Step 4: Scaling the ECS Service

Scaling an ECS service with Fargate is straightforward:

  1. Go to the service and click Update.
  2. Change the Desired number of tasks to 3 (one task per AZ for high availability).
  3. Keep the task definition, compute configuration, and load balancing settings unchanged.
  4. Click Update to apply the changes.

5. Monitor the Tasks tab:

  • New tasks transition from Pending to Activating to Running.
  • AWS provisions the necessary resources automatically in the Fargate engine.

6. Refresh the ALB’s DNS name in your browser:

  • The IP address changes with each refresh, confirming the ALB is distributing traffic across the three tasks.

Step 5: Cleaning Up

To minimize costs:

  1. Update the service and set the Desired number of tasks to 0. This stops all running containers while keeping the service definition.
  2. Navigate to the EC2 > Auto Scaling Groups console, select the Infra-ECS-Cluster group, and set the Desired Capacity to 0 to terminate any EC2 instances (if used).
  3. Verify in the Tasks and Events tabs that all tasks are stopped.

Key Takeaways

  • We created a task definition (nginxdemos-hello) to run an Nginx container from Docker Hub.
  • We deployed an ECS service on Fargate, integrated it with an ALB, and exposed it via HTTP.
  • Scaling the service to multiple tasks was seamless with Fargate’s serverless compute.
  • We cleaned up resources to avoid unnecessary costs.

Next Steps

You’ve now set up an ECS cluster, created a task definition, and deployed a scalable service with Fargate. Experiment with adding task roles for AWS API access, integrating persistent storage (e.g., EFS), or enabling auto-scaling based on CloudWatch alarms. Stay tuned for more ECS tutorials, including advanced configurations and EKS integration!

Conclusion

This comprehensive, two-part guide successfully walked through the setup and scaling of a foundational Amazon ECS cluster (DemoCluster). We first established a versatile cluster configured for both the serverless Fargate and self-managed EC2 launch types, confirming that instances in the managed Auto Scaling Group seamlessly register via the ECS Agent. Building on this base, we then created a Task Definition for a simple Nginx application and deployed it as a resilient ECS Service using Fargate. Crucially, the service was integrated with an Application Load Balancer (ALB), enabling public access and demonstrating the effortless, on-demand scaling of containerized workloads from one to three tasks across multiple Availability Zones. By the end, we achieved a fully functional, scalable, and cost-optimized container deployment platform, culminating in the proper cleanup of resources to conclude the hands-on demonstration.
