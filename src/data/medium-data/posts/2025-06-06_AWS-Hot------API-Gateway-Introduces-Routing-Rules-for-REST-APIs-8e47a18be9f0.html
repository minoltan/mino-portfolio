AWS Hot 🔥 | API Gateway Introduces Routing Rules for REST APIs

AWS Posted on: Jun 3, 2025

To understand the new routing rules for REST APIs in Amazon API Gateway, let’s consider a real-world example comparing how HTTP requests would have been handled before this feature, and how they can be handled now with routing rules.

Scenario: A/B Testing for a New Feature

Imagine you have an e-commerce website and you’re launching a new recommendation engine. You want to A/B test this new engine with 10% of your users to see its impact before rolling it out to everyone.

Before Routing Rules (Traditional Approach)

Without API Gateway routing rules, you’d typically handle this in one of the following ways:

Client-Side Logic

Your client application (e.g., mobile app, web browser) would be responsible for determining which version of the API to call. This could involve:

  • The client making an initial call to a “feature flag” service to determine if it should use the new recommendation engine.
  • Based on the flag, the client then constructs the appropriate URL for either the old or new API.

Example:

  • User A (Control Group): Client calls api.example.com/recommendations/v1
  • User B (Test Group): Client calls api.example.com/recommendations/v2

Drawbacks:

  • Increased client complexity: The client needs to manage the A/B testing logic.
  • Slower updates: If you change the A/B test criteria or percentages, you might need to update and redeploy client applications.
  • Less control: Less control over which specific users are routed without additional server-side logic.

Proxy Layer/Load Balancer

You might have an intermediary proxy server (like Nginx, HAProxy, or a dedicated load balancer service) in front of API Gateway or your backend services. This proxy would inspect incoming requests and route them to different API Gateway endpoints or backend services based on configured rules.

Example:

  • All requests go to api.example.com/recommendations.
  • The proxy server inspects an HTTP header like X-User-Segment: A or X-User-Segment: B.
  • If X-User-Segment: A, the proxy routes to api.example.com/v1_recommendations.
  • If X-User-Segment: B, the proxy routes to api.example.com/v2_recommendations.

Drawbacks:

  • Additional infrastructure: You need to manage and maintain the proxy layer.
  • Increased latency: An extra hop in the request path.
  • Complexity: Configuring and managing routing rules on the proxy can be complex.

After Routing Rules (Traditional Approach)

Now, with API Gateway’s new routing rules for custom domain names, you can achieve the same A/B testing scenario much more efficiently, directly within API Gateway.

How it works

  1. Custom Domain Name: You have a custom domain name, e.g., api.example.com, configured in API Gateway.
  2. Multiple REST APIs/Stages: You have two different REST APIs or stages:
  • Recommendations-v1-API (for the current recommendation engine, associated with stage prod)
  • Recommendations-v2-API (for the new recommendation engine, associated with stage beta)

3. Routing Rules: You define routing rules on your custom domain in API Gateway.

Example Implementation

Rule 1 (Lower Priority):

  • Condition: HTTP Header: X-User-Segment is B (or X-User-Segment exists and random_number < 0.1 if you use a lambda for complex logic).
  • Action: Route to Recommendations-v2-API (stage beta).
  • Note: This rule would typically be for the specific segment of users you want to test the new feature with. You could have a Lambda authorizer or a pre-request Lambda that sets this header based on user IDs or other criteria to ensure 10% of users get X-User-Segment: B.

Rule 2 (Higher Priority):

  • Condition: (No specific conditions, or a catch-all condition)
  • Action: Route to Recommendations-v1-API (stage prod).
  • Note: This is the default rule that catches all other requests.

How a Request is Handled

  1. A user’s request comes to api.example.com/recommendations.
  2. API Gateway receives the request.
  3. API Gateway evaluates Rule 1:
  • If the X-User-Segment header in the request is B, the conditions are met. API Gateway immediately routes the request to Recommendations-v2-API (stage beta). The user gets recommendations from the new engine.

4. If Rule 1’s conditions are not met (e.g., X-User-Segment is not B, or the header is absent), API Gateway proceeds to evaluate Rule 2.

5. Rule 2 (the default/catch-all) matches, and the request is routed to Recommendations-v1-API (stage prod). The user gets recommendations from the current engine.

Benefits of the New Approach

  • Simplified Architecture: No need for external proxy layers or complex client-side routing logic. The routing is handled directly within API Gateway.
  • Reduced Latency: Fewer hops in the request path.
  • Centralized Control: All routing logic is managed within API Gateway, making it easier to configure, update, and monitor.
  • Dynamic Backend Selection: Easily switch between different API versions or backend implementations based on request attributes.
  • Granular Control: Route based on various HTTP headers, URL paths, or a combination, providing fine-grained control over API traffic.
  • Faster Deployment: Changes to routing rules can be deployed quickly without impacting client applications or requiring additional infrastructure changes.

In essence, the new routing rules in API Gateway allow you to move sophisticated routing logic directly into your API Gateway configuration, streamlining your architecture and providing more control over how your API traffic is directed.
