Introduction

In this blog post, we’ll explore a microservices architecture for MyBlog.com, transitioning from a serverless focus to a flexible, scalable design where services interact via REST APIs. Each microservice will have its own architecture, codebase, and scaling strategy to support a lean development lifecycle. We’ll also discuss synchronous and asynchronous patterns, challenges, and hands-on steps to implement the architecture using AWS services.

Requirements

  • Implement a microservices architecture with REST API interactions.
  • Allow each microservice to have a unique architecture, codebase, and scaling strategy.
  • Support lean development cycles and independent scaling for each service.
  • Enable services to interact with each other (e.g., Service 2 calls Service 1, Service 3 calls Service 2).
  • Use a mix of serverless and traditional architectures (e.g., ECS, EC2, Lambda).
  • Host services with DNS names (e.g., service1.example.com).
  • Address microservices challenges like server utilization and client-side complexity.

Prerequisites: AWS account, AWS CLI installed, Route 53 hosted zone for example.com, basic IAM permissions.

Architecture Overview

We’ll design three microservices for MyBlog.com, each with a distinct architecture:

  • Service 1: Elastic Load Balancer (ELB) → ECS (Docker) → DynamoDB.
  • Service 2: API Gateway → Lambda → ElastiCache (serverless).
  • Service 3: ELB → EC2 Auto Scaling → RDS (traditional). Services interact via REST APIs (synchronous) or could use asynchronous patterns (SQS/SNS). We’ll use Route 53 for DNS resolution and include hands-on steps for each service.

Step 1: Service 1 — ECS with DynamoDB

Service 1 uses an Elastic Load Balancer (ELB) to route traffic to Amazon ECS (Docker containers) with DynamoDB as the backend. It’s accessible via service1.example.com.

Hands-On:

  1. Create DynamoDB Table: In DynamoDB Console, create table BlogContent, Partition Key: postId (String), On-Demand capacity.
aws dynamodb create-table 
--table-name BlogContent 
--attribute-definitions AttributeName=postId,AttributeType=S 
--key-schema AttributeName=postId,KeyType=HASH 
--billing-mode PAY_PER_REQUEST

2. Set Up ECS Cluster: ECS Console > Create Cluster: BlogCluster, Fargate.

3. Create Task Definition: Define container (e.g., Node.js app) with port 80.

{
  "family": "blog-service1",
  "containerDefinitions": [{
    "name": "blog-app",
    "image": "node:16",
    "portMappings": [{"containerPort": 80, "hostPort": 80}]
  }],
  "requiresCompatibilities": ["FARGATE"],
  "cpu": "256",
  "memory": "512"
}

4. Create ECS Service: Link to Application Load Balancer (ALB), Target Group: Port 80.

5. Set Up ALB: ELB Console > Create ALB: BlogALB, Listener: HTTPS (port 443), attach SSL certificate.

6. Route 53 DNS: Create alias record service1.example.com pointing to ALB.

aws route53 change-resource-record-sets 
--hosted-zone-id <ZONE-ID> 
--change-batch '{"Changes": [{"Action": "CREATE", "ResourceRecordSet": {"Name": "service1.example.com", "Type": "A", "AliasTarget": {"HostedZoneId": "<ALB-ZONE-ID>", "DNSName": "<ALB-DNS>", "EvaluateTargetHealth": true}}}]}'

7. Deploy Node.js App:

const express = require('express');
const AWS = require('aws-sdk');
const app = express();
const ddb = new AWS.DynamoDB.DocumentClient();
app.get('/posts/:postId', async (req, res) => {
  const result = await ddb.get({ TableName: 'BlogContent', Key: { postId: req.params.postId } }).promise();
  res.json(result.Item);
});
app.listen(80);

8. CLI:

aws ecs create-cluster --cluster-name BlogCluster
aws ecs register-task-definition --cli-input-json file://task-def.json
aws ecs create-service --cluster BlogCluster --service-name BlogService1 --task-definition blog-service1 --launch-type FARGATE --load-balancers targetGroupArn=<TG-ARN>,containerName=blog-app,containerPort=80

9. Test: curl https://service1.example.com/posts/post1.

Step 2: Service 2 — Serverless with Lambda and ElastiCache

Service 2 uses API Gateway → Lambda → ElastiCache (Redis) for a serverless architecture. It interacts with Service 1 via REST API (service1.example.com).

Hands-On:

  1. Set Up ElastiCache: ElastiCache Console > Create Cluster: Redis, BlogCache, Node Type: cache.t3.micro, VPC: Default.
aws elasticache create-cache-cluster 
--cache-cluster-id BlogCache 
--cache-node-type cache.t3.micro 
--engine redis

2. Create Lambda Function: Lambda Console > Name: BlogService2, Runtime: Node.js 22.

const redis = require('redis');
const axios = require('axios');
exports.handler = async (event) => {
  const client = redis.createClient({ url: 'redis://<ELASTICACHE-ENDPOINT>:6379' });
  await client.connect();
  const cacheKey = event.pathParameters.postId;
  let data = await client.get(cacheKey);
  if (!data) {
    const response = await axios.get(`https://service1.example.com/posts/${cacheKey}`);
    data = JSON.stringify(response.data);
    await client.set(cacheKey, data, { EX: 300 });
  }
  await client.quit();
  return { statusCode: 200, body: data };
};

3. Create API Gateway: API Gateway Console > REST API > Resource: /posts/{postId}, Method: GET, Integration: Lambda Proxy.

aws apigateway create-rest-api --name BlogService2API
aws apigateway create-resource --rest-api-id <API-ID> --parent-id <ROOT-ID> --path-part posts/{postId}
aws apigateway put-method --rest-api-id <API-ID> --resource-id <RESOURCE-ID> --http-method GET --authorization-type NONE
aws apigateway create-deployment --rest-api-id <API-ID> --stage-name prod

4. IAM Role: Grant Lambda access to ElastiCache (VPC) and internet for REST calls.

{
  "Statement": [
    {"Effect": "Allow", "Action": ["elasticache:Connect"], "Resource": "*"},
    {"Effect": "Allow", "Action": ["ec2:CreateNetworkInterface", "ec2:DescribeNetworkInterfaces", "ec2:DeleteNetworkInterface"], "Resource": "*"}
  ]
}

5. Route 53 DNS: Create alias record service2.example.com to API Gateway.

aws route53 change-resource-record-sets 
--hosted-zone-id <ZONE-ID> 
--change-batch '{"Changes": [{"Action": "CREATE", "ResourceRecordSet": {"Name": "service2.example.com", "Type": "A", "AliasTarget": {"HostedZoneId": "<API-GW-ZONE-ID>", "DNSName": "<API-GW-DNS>", "EvaluateTargetHealth": false}}}]}'

6. Test: curl https://service2.example.com/prod/posts/post1. Verify cache hits in ElastiCache metrics.

Step 3: Service 3 — EC2 with RDS

Service 3 uses ELB → EC2 Auto Scaling → RDS (MySQL) for a traditional architecture. It calls Service 2 (service2.example.com) for data.

Hands-On:

  1. Create RDS Instance: RDS Console > Create Database: MySQL, BlogDB, Instance: db.t3.micro, VPC: Default.
aws rds create-db-instance 
--db-instance-identifier BlogDB 
--db-instance-class db.t3.micro 
--engine mysql 
--allocated-storage 20 
--master-username admin 
--master-user-password <PASSWORD>

2. Set Up EC2 Auto Scaling: EC2 Console > Auto Scaling Group: BlogASG, Launch Template: t3.micro, AMI: Amazon Linux 2.

3.Create ALB: ELB Console > Name: BlogALB2, Listener: HTTPS (port 443).

4. Deploy Node.js App on EC2:

const express = require('express');
const mysql = require('mysql2/promise');
const axios = require('axios');
const app = express();
app.get('/data/:id', async (req, res) => {
  const service2Data = await axios.get(`https://service2.example.com/prod/posts/${req.params.id}`);
  const conn = await mysql.createConnection({ host: '<RDS-ENDPOINT>', user: 'admin', password: '<PASSWORD>', database: 'blog' });
  const [rows] = await conn.execute('SELECT * FROM posts WHERE id = ?', [req.params.id]);
  res.json({ service2: service2Data.data, rds: rows });
  await conn.end();
});
app.listen(80);

5. Create RDS Table:

CREATE TABLE posts (id VARCHAR(50), title VARCHAR(255), PRIMARY KEY (id));
INSERT INTO posts (id, title) VALUES ('post1', 'Sample Post');

6. CLI:

aws autoscaling create-auto-scaling-group --auto-scaling-group-name BlogASG --launch-template LaunchTemplateName=BlogTemplate --min-size 1 --max-size 3
aws elbv2 create-load-balancer --name BlogALB2 --subnets <SUBNET1> <SUBNET2> --security-groups <SG-ID>

7. Route 53 DNS: Create alias record service3.example.com to ALB.

aws route53 change-resource-record-sets 
--hosted-zone-id <ZONE-ID> 
--change-batch '{"Changes": [{"Action": "CREATE", "ResourceRecordSet": {"Name": "service3.example.com", "Type": "A", "AliasTarget": {"HostedZoneId": "<ALB-ZONE-ID>", "DNSName": "<ALB-DNS>", "EvaluateTargetHealth": true}}}]}'

8. Test: curl https://service3.example.com/data/post1.

Step 4: Asynchronous Pattern (Optional)

For asynchronous communication, services could use SQS, SNS, or Kinesis. For example, Service 1 could send messages to an SQS queue, processed by Service 2.

Hands-On:

  1. Create SQS Queue: SQS Console > Create Queue: BlogQueue, Standard Queue.
aws sqs create-queue --queue-name BlogQueue

2. Update Service 1 to Send to SQS:

const AWS = require('aws-sdk');
const sqs = new AWS.SQS();
await sqs.sendMessage({
  QueueUrl: '<SQS-QUEUE-URL>',
  MessageBody: JSON.stringify({ postId: 'post1', action: 'process' })
}).promise();

3. Create Lambda for SQS: Name: ProcessQueue, Trigger: SQS BlogQueue.

exports.handler = async (event) => {
  for (const record of event.Records) {
    console.log('Processing:', record.body);
  }
};

4. CLI:

aws lambda create-function --function-name ProcessQueue --runtime nodejs18.x --role <ROLE-ARN> --handler index.handler --zip-file fileb://queue.zip
aws lambda create-event-source-mapping --function-name ProcessQueue --event-source-arn <SQS-QUEUE-ARN>

5. Test: Send message from Service 1, check Lambda logs in CloudWatch.

Final Architecture

The architecture:

  • Service 1: ELB → ECS → DynamoDB (service1.example.com).
  • Service 2: API Gateway → Lambda → ElastiCache (service2.example.com), calls Service 1.
  • Service 3: ELB → EC2 Auto Scaling → RDS (service3.example.com), calls Service 2.
  • Route 53: Resolves DNS for all services.
  • Optional Asynchronous: SQS/SNS/Kinesis for decoupled communication.

Each service scales independently with its own codebase and architecture.

Conclusion

The MyBlog.com microservices architecture demonstrates flexibility with ECS, Lambda, and EC2-based services, connected via REST APIs and Route 53. Synchronous (REST) and asynchronous (SQS) patterns enable robust communication, while serverless components reduce operational overhead. The hands-on steps help you deploy this in under an hour.
