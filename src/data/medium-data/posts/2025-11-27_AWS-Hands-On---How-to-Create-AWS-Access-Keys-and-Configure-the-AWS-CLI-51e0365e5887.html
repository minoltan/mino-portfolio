Introduction

You’ve just installed the AWS CLI — now let’s make it actually work by giving it credentials. In this short guide we’ll walk through the classic (and still very common) method: creating IAM access keys and using aws configure to set everything up in under 2 minutes.

Getting started with AWS is exciting, but before you can launch your first S3 bucket, spin up an EC2 instance, or test a Bedrock model from your laptop, you need one essential thing: properly configured credentials. In just a few clicks and one simple command, you can create IAM access keys and connect the AWS CLI to your account. This short, hands-on process bridges the friendly web console you already know with the full power of the command line — giving you the ability to script, automate, and manage your entire AWS environment from any terminal, on any machine, in seconds. Let’s walk through it step by step.

Step 1: Create an IAM Access Key (One-Time Only!)

  1. Log into the AWS Management Console with your IAM user (not root — never use root for daily work).
  2. Click your username (top-right) → Security credentials.
  3. Scroll down to Access keys → Create access key.

4. Choose Command Line Interface (CLI).

5. You’ll see a big yellow warning box with better alternatives (CloudShell, IAM Identity Center SSO, etc.). → That’s excellent long-term advice, but for learning and many local/dev workflows, access keys are still perfectly valid.

6. Tick the checkbox “I understand the above recommendation…” → Next.

7. (Optional) Add a description like “My laptop — 2025” → Create access key.

IMPORTANT: This is the only time you will ever see the Secret Access Key. → Click Show → Copy both the Access key ID and Secret access key (or download the CSV).

Never commit these keys to GitHub or share them. Treat them like passwords.

Step 2: Configure the AWS CLI (Takes 20 Seconds)

Open your terminal (PowerShell, CMD, Terminal, iTerm, etc.) and run:

aws configure

You will be prompted for four things:

AWS Access Key ID [None]:     AKIAxxxxxxxxxxxxxxxx
AWS Secret Access Key [None]: xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
Default region name [None]:   us-east-1        # or eu-west-1, ap-south-1, etc.
Default output format [None]: json            # just press Enter (json is fine)

That’s it! The CLI just created (or updated) two files in your home directory:

  • ~/.aws/credentials → stores your keys securely
  • ~/.aws/config → stores region & output preferences

Step 3: Test That Everything Works

aws sts get-caller-identity

You should instantly get back something like:

{
    "UserId": "AIDAXYZ...",
    "Account": "123456789012",
    "Arn": "arn:aws:iam::123456789012:user/stephane"
}

Or try listing IAM users:

aws iam list-users --query "Users[*].UserName"

Permissions Are Identical in Console & CLI

If you remove your IAM user from the AdministratorAccess group (or any policy), the CLI will immediately start returning:

An error occurred (AccessDenied) when calling the ListUsers operation...

Add the user back → CLI works again instantly. The CLI does not have “magic” permissions — it respects exactly the same IAM policies as the web console.

Quick Recap & Best Practices (2025)

  • Access keys are still 100% supported and widely used for local development, CI/CD, and scripts.
  • Never use root user keys — always create an IAM user with the permissions you actually need.
  • Store keys only in ~/.aws/credentials (never hard-code them in scripts).
  • Rotate keys every 90 days (AWS will remind you).
  • When possible, upgrade later to IAM Roles Anywhere or SSO — but access keys remain a perfectly valid starting point.

You now have full programmatic access to every AWS service from your laptop. Run aws s3 ls, aws ec2 describe-instances, aws bedrock list-foundation-models — whatever you want.

Go build something awesome! ☁️⚡

Conclusion

And that’s all there is to it! With your access keys created and aws configure complete, your local AWS CLI is now fully authenticated and ready to talk to every AWS service on your behalf. Whether you’re uploading files to S3, deploying Lambda functions, querying Bedrock, or managing dozens of resources across regions, you now have a fast, repeatable, scriptable way to get work done. Keep those keys safe, consider rotating them regularly, and when you’re ready, you can level-up to IAM Identity Center or CloudShell — but for now, you’ve just unlocked the true command-line superpower of AWS. Happy building!
