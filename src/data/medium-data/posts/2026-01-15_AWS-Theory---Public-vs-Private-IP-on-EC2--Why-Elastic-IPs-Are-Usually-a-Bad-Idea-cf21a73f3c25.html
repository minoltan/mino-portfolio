Introduction

Every EC2 instance comes with two IP addresses, and confusing them is responsible for about 90% of those “why can’t I SSH into this thing?!” moments. It seems quite frustrating. In this blog, I will break down—easily and painlessly—the difference between a public IP, a private IP, and the often misunderstood Elastic IP. We’ll also cover a rule experienced AWS engineers live by: almost never use Elastic IPs in real-world architectures. By the end, you’ll know exactly which IP to use in each situation and why your public IP seems to vanish every time you stop and start an instance.

The Two IPs Every EC2 Instance Gets (Automatically)

Think of it like a house:

  • Private IP = your apartment number inside a giant building (the VPC)
  • Public IP = the building’s street address that delivery drivers see

Real-World Example You Already Experienced

  • When you SSH from your laptop, you use the public IP (you’re outside the building).
  • When two EC2 instances in the same VPC talk to each other, they use private IPs (they’re in the same building).

Elastic IP: The “Fixed” Public IP (and Why Pros Hate It)

An Elastic IP is a static public IPv4 address you own. You can move it from one instance to another like a sticky label.

Sounds great, right? Here’s why it’s usually a code smell in 2025:

  • You’re limited to 5 per region (hard limit for most accounts)
  • Costs money if not attached to a running instance
  • Real architectures use DNS names (Route 53) or load balancers instead—both scale infinitely and survive instance failures automatically
  • Elastic IPs hide failures instead of fixing them

Golden Rule in Modern AWS: Never use Elastic IP : use a random public IP + DNS name (or better: put a load balancer in front and keep instances 100% private).

Quick Demo You Can Try Right Now

  1. Launch any EC2 instance (default settings)
  2. Note the Public IPv4 address → SSH works with this
  3. Stop the instance → Start it again → Public IP changed!
  4. Private IP stays exactly the same
