🌟 Introduction

In the complex ecosystem of AWS security, assigning permissions is often a balancing act between functionality and safety. While identity-based policies define what a user can do, IAM Permission Boundaries define the maximum possible permissions that user is allowed to have. Think of it as a “security ceiling.” Even if you grant a user “AdministratorAccess,” if their permission boundary only allows S3 access, they are effectively restricted to S3 alone. This advanced feature is essential for maintaining the principle of least privilege, especially when delegating authority to others, ensuring that no matter what permissions are added later, the user can never overstep the predefined boundary.

How Permission Boundaries Work: The “Intersection” Logic

The most important rule to remember is that a user can only perform an action if it is allowed by both the identity-based policy and the permission boundary.

  • Identity-Based Policy: What you are “given” (e.g., Allow EC2).
  • Permission Boundary: The “outer limit” (e.g., Allow S3 and EC2).
  • Effective Permission: The overlap between the two.

If you give a user an identity-based policy to iam:CreateUser but their Permission Boundary only allows s3:*, the result is Access Denied. The action must exist in both places to be successful.

Real-World Example: Empowering the “Junior Admin”

Imagine you are a Senior Cloud Architect at a startup. You want to hire a Junior Administrator to help manage the team. You want them to be able to create new IAM users for incoming interns, but you are terrified they might accidentally create an “Admin” account for themselves and take over the company.

The Solution:

  1. Identity Policy: You give the Junior Admin a policy that allows iam:CreateUser.
  2. Permission Boundary: You attach a boundary to the Junior Admin that only allows s3:* and ec2:*.
  3. Result: The Junior Admin can create a new user (the intern), but when they try to give that intern “AdministratorAccess,” the intern will only actually be able to use S3 and EC2. The Junior Admin has effectively been “boxed in” — they can delegate work, but they cannot escalate privileges beyond the ceiling you set.

IAM Permission Boundaries

  • Can be used in combinations of AWS Organizations SCP

Use cases

  • Delegate responsibilities to non administrators within their permission boundaries, for example create new IAM users
  • Allow developers to self-assign policies and manage their own permissions, while making sure they can’t “escalate” their privileges (= make themselves admin)
  • Useful to restrict one specific user (instead of a whole account using Organizations & SCP)

The AWS Policy Evaluation Logic

When you make a request to AWS, the system follows a strict logical flow to decide if you are allowed to proceed. Understanding this flow is key to troubleshooting “Access Denied” errors.

  1. Explicit Deny: If any policy says “Deny,” the request is immediately rejected. Deny always wins.
  2. Organizations SCP: Does the Organization allow this action for this account?
  3. Resource-based Policies: Does the S3 bucket or SQS queue specifically allow this user?
  4. Identity-based Policies: Does the user’s own policy allow the action?
  5. Permission Boundaries: Is the action within the user’s allowed boundary?

Permission Logic Quiz: Test Your Knowledge

Consider a policy with these two statements:

  1. Can you perform sqs:CreateQueue?
  • No, Because there is a "Deny All."

2. Can the user delete a queue?

  • No. Even though there is an explicit allow for DeleteQueue, the explicit deny on sqs:* (which includes DeleteQueue) takes precedence. In AWS, a "No" from any corner of the room cancels out all the "Yes" votes.

3. Can you perform ec2:DescribeInstances?

  • No. You can see nothing on EC2 Instances.

🧾 Conclusion

IAM Permission Boundaries are a sophisticated tool for any organization looking to scale without sacrificing security. By setting a hard limit on what users and roles can achieve, you create a fail-safe environment where developers can move fast and manage their own permissions without the risk of privilege escalation. Whether you are protecting your production environment from accidental “Admin” creations or delegating tasks to junior staff, boundaries provide the peace of mind that your security “ceiling” is ironclad. Mastering this concept, alongside SCPs and Identity policies, marks the transition from a basic cloud user to a true AWS Security expert.
