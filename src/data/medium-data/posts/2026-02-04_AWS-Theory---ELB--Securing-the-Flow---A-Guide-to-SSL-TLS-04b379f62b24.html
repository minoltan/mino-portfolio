🌟 Introduction

In today’s web, security isn’t just an option — it’s the standard. When you see that little green padlock in your browser, you’re looking at SSL/TLS in action. For your AWS load balancer, these certificates are what stand between your users’ private data and the “wild west” of the public internet.

What are SSL and TLS?

We nearly always use TLS (Transport Layer Security), even though most of the people still refers to SSL (Secure Sockets Layer). Simply put, TLS is SSL’s more recent and secure version.

  • The Goal: To provide In-Flight Encryption. This ensures that data traveling from a user’s laptop to your load balancer is scrambled and can only be read by the two of them.
  • The Trust: Certificates are issued by Certificate Authorities (CA) like Digicert or Let’s Encrypt. In AWS, we use AWS Certificate Manager (ACM) to handle this for us for free.

How it Works: SSL Termination

One of the best features of an AWS Load Balancer is SSL termination.

  1. The user sends encrypted traffic (HTTPS) over the public internet.
  2. The load balancer terminates (decrypts) the traffic using its certificate.
  3. The load balancer sends the traffic to your EC2 instances as plain HTTP over the private, secure AWS network.

This frees up your servers to concentrate on executing your application by relieving them of the complex math involved in decryption.

The “Apartment Building” Problem: What is SNI?

Let’s imagine a single load balancer hosting three different websites. Such as coffee.com, tea.com, and water.com. If the user connects via HTTPS, how does the load balancer know which certificate to show?

The simple answer is Server Name Indication, or SNI. I mean, it will work like this. “Hey, I’m specifically looking for coffee.com,” the client says to the server at the beginning of the handshake, which is an extension of the TLS protocol. This enables the load balancer to retrieve the appropriate certificate for that particular domain by reaching into its key cabinet.

Step-by-Step Hands-On: Adding an SSL Certificate

Step 1: Get a Certificate in ACM

Before you touch the load balancer, you need a certificate.

  1. Go to AWS Certificate Manager (ACM).
  2. Click Request a certificate and choose Request a public certificate.
  3. Enter your domain name (e.g., myapp.example.com) and complete the DNS validation.

Step 2: Add an HTTPS Listener

  1. Go to EC2 > Load Balancers and select your ALB.
  2. Under the Listeners and rules tab, click Add listener.
  3. Select HTTPS for the protocol (Port 443).
  4. In the Default actions section, choose Forward to target group and select your backend instances.

Step 3: Attach the Certificate

  1. Scroll down to the Secure listener settings.
  2. For the Default SSL/TLS certificate, select From ACM and pick the certificate you created in Step 1.
  3. (Optional) Under Security policy, stick with the latest recommended policy (e.g., ELBSecurityPolicy-2016-08).
  4. Click Add.

🧾 Conclusion

Implementing SSL/TLS is the foundation of a professional web development standard. You can make your architecture more affordable and secure by using SNI to host multiple sites and ACM for your certificates. But don’t forget for the AWS exam, if the question mentions multiple SSL certificates on one load balancer, the answer is almost always ALB or NLB using SNI.
