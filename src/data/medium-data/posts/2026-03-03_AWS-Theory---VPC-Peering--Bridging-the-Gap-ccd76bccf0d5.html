🌟 Introduction

In the world of AWS, isolation is the default setting. Your Virtual Private Cloud (VPC) is a walled garden. Keeping your resources safe and separate from everything else. But in the real world, businesses grow. You might acquire another company with its own AWS account, or you might need to connect your production environment in Northern Virginia to a backup site in Tokyo. VPC peering is the networking handshake that creates a direct, private bridge between two VPCs. It allows them to communicate using private IP addresses as if they were part of the same giant network. All without ever sending traffic over the public internet.

1. The One-to-One Connection

VPC Peering is a direct relationship between exactly two VPCs. While it is incredibly powerful. The one golden rule is, it is not transitive. Imagine you have three VPCs: A, B, and C.

  • You peer VPC A with VPC B.
  • You peer VPC B with VPC C.
  • The Catch: Even though B can talk to both A and C, VPC A cannot talk to VPC C through B. If A needs to talk to C, you must create a direct peering connection between A and C.

2. The Golden Rule: No Overlapping CIDRs

Before you even send a peering request, you must check your address books. VPC Peering will only work if the two VPCs have distinct, non-overlapping IP address ranges (CIDR blocks). If both VPC A and VPC B are using 10.0.0.0/16, your network routers won't know where to send a packet. It’s like having two houses on the same street with the exact same address. The mailman (the AWS router) would just get confused and drop the letter.

3. Updating the Map (Route Tables)

Creating the peering connection is only half the battle. Think of the peering connection as a new bridge. Even though the bridge exists, your instances don’t know it’s there until you update their maps.

  • In VPC A: You must add a route that says, “If you want to reach VPC B’s IP range, go through this peering connection."”
  • In VPC B: You must do the same for VPC A’s range.
  • Without these route table updates, your instances will still try to look for those IPs locally and fail to find them.

4. Security Groups Across the Bridge

One of the coolest features of VPC peering (when both VPCs are in the same region) is that you can reference security groups from the peered VPC.

Instead of opening a port to a wide range of IP addresses (like 10.1.0.0/16), you can tell your database security group, "Only allow incoming traffic from the web server security group in the other VPC." This keeps your security tight and dynamic, even across different AWS accounts!

🧾 Conclusion

VPC Peering is the simplest and most cost effective way to link your isolated cloud networks. It provides high performance, private connectivity that scales across accounts and regions. Just remember, it’s a direct bridge, not a transit hub. When your network grows to dozens or hundreds of VPCs, you might find the mesh of peering connections gets messy, and that’s exactly when you’ll want to look into AWS Transit Gateway.
