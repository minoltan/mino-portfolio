🌟 Introduction

In many high-security environments, instances in private subnets are completely isolated from the internet to minimize the attack surface. However, these instances still often need to interact with AWS services like Amazon S3. Traditionally, this required a NAT Gateway and an Internet Gateway, which effectively opened a path to the public web. VPC Endpoints solve this by providing a private connection within the AWS backbone. By using a Gateway Endpoint for S3, we can modify our subnet’s route table so that traffic destined for S3 stays internal. This not only increases security by removing the need for internet access but also reduces data transfer costs and improves network performance.

Step-by-Step Guide: Accessing S3 via VPC Endpoints

1. Set Up Permissions (IAM Role)

Before an EC2 instance can talk to S3, it needs the “keys” to do so.

  • Go to the IAM Console and create a new Role.
  • Service: EC2.
  • Policy: AmazonS3ReadOnlyAccess.
  • Name: DemoRoleEC2-S3ReadOnly.
  • Attach this role to your Private EC2 Instance via Actions > Security > Modify IAM Role.

2. Verify the “Internet” Connection

Initially, your instance might still have internet access via a NAT Gateway.

  • SSH into your Private Instance (via the Bastion Host).
  • Run aws s3 ls. It should work.
  • Run curl google.com. It should also work.

Cut the Internet Cord

To prove the VPC Endpoint is working, we must first disable the internet.

  • Go to VPC > Route Tables and select your Private Route Table.
  • Edit Routes: Delete the route for 0.0.0.0/0 (the one pointing to the NAT Gateway).
  • Test: Try aws s3 ls and curl google.com. Both should now fail and hang indefinitely.

4. Create the S3 Gateway Endpoint

Now we build the private tunnel.

  • Go to VPC > Endpoints and click Create endpoint.
  • Service Category: AWS Services.
  • Services: Search for “s3” and select the one with the Type Gateway (e.g., com.amazonaws.us-east-1.s3).
  • VPC: Select your Demo VPC.
  • Route Tables: Select your Private Route Table. AWS will automatically inject the route for you.
  • Policy: Full Access.
  • Click Create endpoint.

5. Verify Private Connectivity

Return to your terminal on the Private Instance.

Gateway vs. Interface Endpoints

🧾 Conclusion

By implementing a VPC Gateway Endpoint, we have successfully enabled our private instance to “reach out” to Amazon S3 without ever touching the public internet. This architecture fulfills the “Principle of Least Privilege” at a network level: our instance has exactly the connectivity it needs (S3) and none that it doesn’t (the broad internet). As you scale your AWS environment, remember that Gateway Endpoints for S3 and DynamoDB are free and highly recommended, while Interface Endpoints for other services provide a robust, IP-based way to keep your service-to-service communication entirely within the wire.
