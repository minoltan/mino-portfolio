Here’s a step-by-step guide to demonstrating S3 Event Notifications, based on the provided content:

Objective: Learn how to configure S3 Event Notifications to send messages to an SQS queue whenever an object is created in an S3 bucket.

Prerequisites:

✔ An AWS account.

✔ A sample file (e.g., coffee.jpeg) to upload.

Step 1: Create an S3 Bucket

  1. Navigate to S3: Go to the Amazon S3 console.
  2. Create Bucket: Click “Create bucket”.
  3. Configure Bucket Details:
  • Bucket name: mino-v3-events-notifications (or a unique name of your choice).
  • AWS Region: Choose a region, e.g., ap-southeast-1 (Singapore).
  • Leave other settings as default for this demo.

4. Create Bucket: Click “Create bucket”.

Step 2: Create an SQS Queue

  1. Navigate to SQS: Open a new tab or window and go to the Amazon SQS console.
  2. Create Queue: Click “Create queue”.
  3. Configure Queue Details:
  • Name: DemoS3Notification (or a unique name).
  • Leave other settings as default (Standard queue is sufficient).

4. Create Queue: Click “Create queue”.

Step 3: Update SQS Queue Access Policy to Allow S3 to Publish Messages

  1. Select SQS Queue: In the SQS console, select your DemoS3Notification queue.
  2. Go to “Access policy”: Scroll down to the “Access policy” section and click “Edit”.
  3. Use Policy Generator: Under “Policy”, click “Generate policy”.
  4. Configure Policy Generator:
  • Select Type of Policy: Choose SQS Queue Policy.
  • Effect: Allow.
  • Principal: * (This is very permissive for demonstration; in production, you would use the S3 service principal s3.amazonaws.com or the bucket ARN).
  • Actions: Select SendMessage.
  • Amazon Resource Name (ARN): Go back to your SQS queue’s details page, copy its ARN, and paste it here.
  • Click “Add Statement”.
  • Click “Generate Policy”.

5. Copy and Paste Policy: Copy the generated JSON policy.

6. Apply Policy to SQS Queue: Go back to the SQS queue’s “Edit access policy” page, delete any existing policy text in the “Policy” box, and paste the new JSON policy.

7. Save Changes: Click “Save”.

Explanation: This policy grants S3 permission to send messages to this SQS queue. Without it, S3 cannot publish events.

Step 4: Configure S3 Event Notification to SQS

  1. Navigate to S3 Bucket: Go back to your mino-v3-events-notifications S3 bucket.
  2. Go to “Properties” Tab: Click on the “Properties” tab.
  3. Create Event Notification: Scroll down to “Event notifications” and click “Create event notification”.
  4. Configure Notification Details:
  • Event name: DemoEventNotification (or a descriptive name).
  • Prefix/Suffix: Leave blank for now (applies to all objects).
  • Event types: Select “All object create events” (this includes Put, Post, Copy, CompleteMultipartUpload).
  • Optional: Explore other event types like Object removals, Object restores, etc.
  • Destination: Select “SQS queue”.
  • Choose from your queues: Select your DemoS3Notification queue from the dropdown.

5. Save Changes: Click “Save changes”.

Observation: If the SQS queue policy was set up correctly, the notification will be saved successfully. S3 will immediately send a test message to the SQS queue.

Step 5: Verify the Test Message in SQS

  1. Go to SQS Console: Navigate back to your DemoS3Notification SQS queue.
  2. Send and Receive Messages: Click “Send and receive messages”.
  3. Poll for Messages: Click “Poll for messages”.
  • Observation: You should see a message with the eventName s3:TestEvent from Amazon S3. This confirms the connection is working.

4. Delete Test Message: Select the test message and click “Delete”.

Step 6: Test S3 Event Notification with an Object Upload

  1. Upload a File to S3: Go back to your mino-v3-events-notifications S3 bucket.
  2. Upload Object: Click “Upload”, then “Add files”, and select coffee.jpeg.
  3. Complete Upload: Click “Upload”.
  4. Poll for Messages in SQS: Go back to your DemoS3Notification SQS queue in the SQS console, and click "Poll for messages".
Observation: You should now see a new message.
Inspect Message: Click on the message to view its details. Look for:
1. "eventName": "ObjectCreated:Put" (or similar ObjectCreated event).
 2. The key under s3 indicating coffee.jpeg as the uploaded object.

Conclusion:

You have successfully configured and demonstrated S3 Event Notifications. You’ve learned how to:

  • Set up an S3 bucket and an SQS queue.
  • Grant S3 permissions to publish messages to SQS using an access policy.
  • Create an S3 Event Notification rule to trigger on object creation.
  • Verify that messages are sent to the SQS queue upon object upload, including a test event and a real event. This capability is fundamental for building event-driven architectures on AWS.
