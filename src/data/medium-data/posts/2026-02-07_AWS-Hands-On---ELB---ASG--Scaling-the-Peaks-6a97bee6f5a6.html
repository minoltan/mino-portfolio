🌟 Introduction

Think of an Auto Scaling Group (ASG) as the cruise control for your web fleet. If you’re running a news site and a story goes viral, you don’t want to be manually clicking Launch Instance at 3:00 AM. A real-world example is a retail shop during a flash sale, the sudden traffic spikes for two hours. The ASG adds 10 servers to handle the load, and once the sale ends, it terminates them to save you money.

In this guide, we’ll walk through creating an ASG from scratch and linking it to a load balancer. Finally watching it self-heal and scale in real time.

Step 1: Creating a Launch Template

Before the ASG can launch a server, it needs to know what to launch. We use a launch template for this.

  1. Navigate to EC2 -> Launch Templates and click Create launch template.

2. Name: DemoTemplate.

3. AMI: Select Amazon Linux 2 (Free Tier eligible).

4. Instance Type: Select t3.micro. (Free tier eligible)

5. Key Pair: Select your existing .pem key or don’t include.

6. Security Group: Choose one that allows HTTP (Port 80) traffic.

7. Advanced Details: Scroll to the bottom and paste your User Data script (the script that installs your web server).

#!/bin/bash
# Use this for your user data (script from top to bottom)
# install httpd (Linux 2 version)
yum update -y
yum install -y httpd
systemctl start httpd
systemctl enable httpd
echo "<h1>Hello World from $(hostname -f)</h1>" > /var/www/html/index.html

8. Click Create launch template.

Step 2: Creating the Auto Scaling Group

Now, let’s build the group that will use that blueprint.

  1. Go to Auto Scaling Groups in the left menu and click Create Auto Scaling group.

2. Name: Demo-ASG. Select the DemoTemplate you just made.

3. Network: Select your VPC and at least three subnets (different Availability Zones) for high availability.

4. Load Balancing: Choose Attach to an existing load balancer and select your Target Group (e.g., demo-tg-alb).

5. Health Checks: Turn on ELB Health Checks. This allows the ASG to replace instances if the Load Balancer says they aren’t responding.

6. Group Size: Set Desired: 1, Min: 1, and Max: 1.

7. Click Create Auto Scaling group.

Step 3: Verification & The Scaling Test

Once created, the ASG sees its “Desired Capacity” is 1, but it has 0 instances. It will immediately launch one.

  • Activity History: Click your ASG and look at the Activity tab. You’ll see a status like “Launching a new EC2 instance.”
  • Target Group: Go to your Target Group > Targets. You’ll see the new instance appearing. Once it passes health checks (status: healthy), your Load Balancer will start sending it traffic.

The Scaling Demo

  1. Scale Out: Edit your ASG and change the Desired Capacity to 2.

2. Watch: In the Activity History, you’ll see a second instance launch. Refresh your Load Balancer DNS in the browser; you’ll now see traffic alternating between two different IPs!

3. Scale In: Change the Desired Capacity back to 1.

4. Watch: The ASG will pick one instance and terminate it. This is “Scaling In” to save costs.

🧾 Conclusion

You’ve just built a self-healing, elastic web fleet. By linking the ASG to a target group, you’ve automated the hardest part of cloud engineering. Ensuring your app stays online and balanced without manual intervention. Whether you’re scaling up for a surge or scaling down to save pennies, the ASG handles the heavy lifting for you.
