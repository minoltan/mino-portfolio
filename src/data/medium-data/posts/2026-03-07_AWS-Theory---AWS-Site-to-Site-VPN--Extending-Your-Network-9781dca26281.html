🌟 Introduction

Connecting your office to the cloud shouldn’t feel like building a bridge across an ocean. AWS Site-to-Site VPN is the tool that makes your corporate data center and your AWS VPC feel like they are sitting in the same room. It creates a secure, encrypted tunnel over the public internet, allowing your on-premises servers to talk to your EC2 instances using their private IP addresses. It’s like having a private extension cord that stretches from your desk all the way to an AWS data center.

1. The Two Pillars: VGW and CGW

To build this bridge, you need an anchor on both sides:

  • Virtual Private Gateway (VGW): This is the “VPN Concentrator” on the AWS side. You create it, attach it to your VPC, and it acts as the gateway for all VPN traffic entering or leaving the cloud.
  • Customer Gateway (CGW): This is the “Anchor” on your side (on-premises). It can be a physical firewall (like a Cisco or Juniper device) or even a software appliance.

2. Establishing the Connection

When setting up the connection, the IP address of your Customer Gateway matters:

  • Public CGW: If your device has a direct public IP, use that to establish the tunnel.
  • Private CGW (Behind NAT): If your device is behind a router, use the Public IP of the NAT device. Ensure “NAT-T” (NAT Traversal) is enabled on port UDP 4500 so the encrypted traffic can pass through.

3. Two Gotchas for the Exam

Even if the tunnel is technically up, traffic might not flow. There are two common reasons for this:

  1. Route Propagation: You must go into your VPC Route Table and enable Route Propagation. This tells your subnets, “Hey, if you see traffic for the office’s IP range, send it through the VGW!”
  2. Security Group Rules: If you are trying to ping an instance from your office to test the connection, you must explicitly allow ICMP traffic in the EC2 instance’s Security Group. Without this, the instance will simply ignore the request.

4. AWS VPN CloudHub: Connecting the Spokes

If you have multiple branch offices (e.g., New York, London, and Tokyo) and you want them to talk to each other as well as to AWS, you use AWS VPN CloudHub.

Instead of building a messy web of separate VPNs between every office, you connect every office to a single Virtual Private Gateway. This creates a “hub-and-spoke” model where the VGW acts as the central hub, relaying encrypted traffic between all your branch offices securely.

🧾 Conclusion

AWS Site-to-Site VPN is a reliable, low-cost way to create a hybrid cloud environment. By using encrypted IPsec tunnels, you get the security of a private network with the convenience of using the existing public internet. Whether you are connecting one office or building a global network with CloudHub, the key is always in the routing and the gateways.
