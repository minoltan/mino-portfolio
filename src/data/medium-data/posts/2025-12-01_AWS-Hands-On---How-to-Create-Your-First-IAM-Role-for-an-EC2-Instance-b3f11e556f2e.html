Introduction

You now understand that IAM Roles are the secure, recommended way to give AWS services permissions. It’s time to get your hands dirty and actually create one! In this quick guide we’ll build a real IAM Role that an EC2 instance can use — the most classic and common example you’ll encounter (and one that appears on every certification exam). By the end, you’ll have a reusable role ready to attach the moment we launch our first virtual server.

Step-by-Step: Create an IAM Role for EC2 (Takes Less Than 2 Minutes)

  1. In the AWS Management Console, go to IAM → Roles → Create role.

2. Select trusted entity type

  • AWS service
  • Choose EC2 (under “Common use cases” or search for it)
  • Use case: EC2
  • Next

This automatically sets the trust policy so only the EC2 service can assume this role.

3. Attach permissions For this demo, let’s give the instance read-only access to IAM (super safe for learning):

  • Search and check IAMReadOnlyAccess
  • (Later you’ll attach things like AmazonS3FullAccess, CloudWatchAgentServerPolicy, etc.)
  • Next

4. Add a clear, descriptive name Role name: DemoRoleForEC2 (max 64 characters, no spaces) Optional description: “Demo role to attach to EC2 instances — IAM read-only”

  1. Review & Create You’ll see:
  • Trusted entity: EC2 service
  • Attached policy: IAMReadOnlyAccess → Click Create role

Done! Your role instantly appears in the Roles list with a green “Successfully created role” banner.

Quick Verification

Click the new role name → you’ll see:

  • Trust relationship: ec2.amazonaws.com is allowed to sts:AssumeRole
  • Permissions: IAMReadOnlyAccess (or whatever you attached)

That’s literally all there is to it.

What Happens Next?

Right now the role just sits there waiting. The magic happens when we launch an EC2 instance and select DemoRoleForEC2 in the “IAM instance profile” dropdown. From that moment on, any code or CLI command running inside the instance can securely call IAM APIs (or S3, DynamoDB, etc., once you add more policies) without ever seeing a secret key.

Conclusion

You’ve just created your first production-ready IAM Role — the same pattern you’ll repeat hundreds of times in real projects. Whether it’s an EC2 instance reading from S3, a Lambda function publishing to SNS, or a Bedrock agent calling other services, the process is always identical: pick the service, attach the least-privilege policies, give it a clear name, and you’re done. Roles are the backbone of secure AWS architecture, and now you can create them confidently in under two minutes. In the next section we’ll attach this exact role to a live EC2 instance and watch temporary credentials appear automatically. Get ready — things are about to get really fun! 🚀
