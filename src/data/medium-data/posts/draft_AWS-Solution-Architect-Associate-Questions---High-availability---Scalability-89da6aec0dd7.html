Question 1:

Domain
Design High-Performing Architectures

The engineering team at a data analytics company has observed that its flagship application functions at its peak performance when the underlying Amazon Elastic Compute Cloud (Amazon EC2) instances have a CPU utilization of about 50%. The application is built on a fleet of Amazon EC2 instances managed under an Auto Scaling group. The workflow requests are handled by an internal Application Load Balancer that routes the requests to the instances.

As a solutions architect, what would you recommend so that the application runs near its peak performance state?

Overall explanation

✅Correct option:

Configure the Auto Scaling group to use target tracking policy and set the CPU utilization as the target metric with a target value of 50%

An Auto Scaling group contains a collection of Amazon EC2 instances that are treated as a logical grouping for the purposes of automatic scaling and management. An Auto Scaling group also enables you to use Amazon EC2 Auto Scaling features such as health check replacements and scaling policies.

With target tracking scaling policies, you select a scaling metric and set a target value. Amazon EC2 Auto Scaling creates and manages the CloudWatch alarms that trigger the scaling policy and calculates the scaling adjustment based on the metric and the target value. The scaling policy adds or removes capacity as required to keep the metric at, or close to, the specified target value.

For example, you can use target tracking scaling to:

Configure a target tracking scaling policy to keep the average aggregate CPU utilization of your Auto Scaling group at 50 percent. This meets the requirements specified in the given use-case and therefore, this is the correct option.

Target Tracking Policy Overview:

❌Incorrect options:

  1. Configure the Auto Scaling group to use step scaling policy and set the CPU utilization as the target metric with a target value of 50%

Simple Scaling (The “Basic Switch”)

Think of this as an “If X, then Y” toggle. It is the oldest and least flexible method.

  • How it works: You create a CloudWatch Alarm. For example: “If CPU > 60% for 5 minutes.”
  • The Action: When the alarm fires, the ASG adds a fixed number of instances (e.g., “Add 2 instances”).
  • The Downside (Cooldown): After it scales, it waits for a “Cooldown” period (e.g., 5 minutes) before it can do anything else. Even if your CPU spikes to 100% during that wait, it stays frozen.
  • Verdict: Too slow for modern, “bursty” analytics apps.

2. Configure the Auto Scaling group to use simple scaling policy and set the CPU utilization as the target metric with a target value of 50%

Step Scaling (The “Ladder”)

This is a more advanced version of Simple Scaling that stays active even while scaling.

  • How it works: You define “Steps” based on the size of the alarm breach.
  • Step 1: If CPU is 50%–60% -> Add 1 instance.
  • Step 2: If CPU is 60%–80% -> Add 3 instances.
  • Step 3: If CPU is > 80% -> Add 5 instances.
  • The Benefit: It responds more aggressively to big spikes. Unlike Simple Scaling, it doesn’t wait for a long cooldown; it can add more instances immediately if the CPU jumps to a higher “Step.”
  • The Configuration: You still have to manually define the thresholds ($>$ 50, $>$ 60, etc.). It doesn’t “aim” for a target; it just reacts to the ladder you built.

With step scaling and simple scaling, you choose scaling metrics and threshold values for the Amazon CloudWatch alarms that trigger the scaling process. Neither step scaling nor simple scaling can be configured to use a target metric for CPU utilization, hence both these options are incorrect.

3. Configure the Auto Scaling group to use a Amazon Cloudwatch alarm triggered on a CPU utilization threshold of 50% — An Auto Scaling group cannot directly use a Cloudwatch alarm as the source for a scale-in or scale-out event, hence this option is incorrect.

The Missing Link: The Scaling Policy

An ASG cannot “listen” to a CloudWatch Alarm directly. There is a middleman called a Scaling Policy.

  • Wrong Flow: CloudWatch Alarm -> Auto Scaling Group (Does not work).
  • Correct Flow: CloudWatch Alarm -> Scaling Policy -> Auto Scaling Group.

Think of the Alarm as a “Siren” and the Scaling Policy as the “Instructions.” The ASG is the “Engine.” The engine doesn’t do anything just because a siren goes off; it needs the instructions (the policy) to tell it exactly how much to rev up or down.

How it Works (Step-by-Step)

Here is the mechanical process of how a scaling event actually happens:

  1. The Metric: CloudWatch monitors the CPU of your EC2 instances.
  2. The Alarm: You create an alarm (e.g., “CPU > 50%”). This alarm has no power on its own; it just flips from OK to ALARM state.
  3. The Action: Inside the Alarm configuration, you must define an Action. This action sends a notification to the Scaling Policy.
  4. The Policy Execution: The Scaling Policy receives the signal and says, “Okay, the instructions say to add 2 instances when this alarm fires.”
  5. The ASG Update: The Scaling Policy then sends a command to the ASG to change its “Desired Capacity.”

Where to Find it in the Console

  1. Navigate to EC2: Open the Amazon EC2 Console.
  2. Find your Group: In the left-hand navigation pane, under the Auto Scaling section, click on Auto Scaling Groups.
  3. Select your ASG: Click the name of the Auto Scaling group you want to configure.
  4. Open the Correct Tab: Look at the bottom pane (the tabs) and select Automatic scaling.
  5. Create Policy: In the Dynamic scaling policies section, click the Create dynamic scaling policy button.

How to Fill Out the Form

Once the “Create dynamic scaling policy” window opens, configure it as follows:

  • Policy type: Keep the default Target tracking scaling.
  • Scaling policy name: Give it a name like Maintain-50-Percent-CPU.
  • Metric type: Select Average CPU utilization.
  • Target value: Enter 50.
  • Instance warmup: (Optional but recommended) Enter the number of seconds it takes for your photo-sharing app to fully boot up and be ready for traffic.
  • Why? This prevents AWS from scaling out again while the first new instance is still “warming up.”

Question 2

A digital event-ticketing platform hosts its core transaction-processing service on AWS. The service runs on Amazon EC2 instances and stores finalized transactions in an Amazon Aurora PostgreSQL database. During periods of high user activity — such as flash ticket sales or holiday promotions — the application begins timing out, causing failed or delayed purchases. A solutions architect has been asked to redesign the backend for scalability and cost-efficiency, without reengineering the database layer.

Which combination of actions will meet these goals in the most cost-effective and scalable manner? (Select two)

Overall explanation

✅Correct options:

Implement Amazon RDS Proxy between the application and the Aurora PostgreSQL cluster. Deploy EC2 instances in an Auto Scaling group to retry transactions as needed

Amazon RDS Proxy helps manage connection pooling and throttling, which are common causes of Aurora timeouts during high-load scenarios. It maintains a pool of database connections and reuses them across clients, reducing the overhead of frequent connection creation. Pairing this with an Auto Scaling group of EC2 instances ensures that application compute can scale dynamically based on demand. This combination improves application resiliency, reduces Aurora connection stress, and addresses peak usage efficiently and cost-effectively.

Modify the application to publish purchase events to an Amazon SQS queue. Launch an Auto Scaling group of EC2 workers that poll the queue and process purchases asynchronously

Introducing Amazon SQS decouples the web tier from the processing logic, allowing the system to buffer bursts in transaction load without overwhelming the database. EC2 workers in an Auto Scaling group can be dynamically scaled to consume and process queue messages based on throughput needs. This pattern ensures durability, resiliency, and elastic scalability, especially during peak usage. It’s also cost-effective because compute usage grows and shrinks with actual demand.

❌Incorrect Options:

Use an Amazon ElastiCache cluster to cache database queries. Configure the application to store purchase transactions in the cache before writing to the database — While ElastiCache is effective for read-heavy workloads, it’s not designed for durable transactional writes. Caching transactional data — especially before it is persisted in a database — risks data loss if the cache node fails. In addition, caching will not prevent database connection saturation under write-heavy load. This approach does not solve the scalability issues related to timeouts during concurrent write operations.

Deploy an Amazon API Gateway with throttling and usage plans to slow down incoming purchase requests during peak times and maintain application stability — While API Gateway usage plans and throttling can help protect backend services from overload, they do so by delaying or rejecting traffic, which could worsen the customer experience during peak demand. This does not address the need for backend scalability, and it doesn’t improve throughput or processing capacity. Instead, it suppresses demand rather than meeting it.

Deploy read replicas for the Aurora database in another Region and configure EC2 instances to read and write from the nearest replica based on latency — This option acts as a distractor. While read replicas improve read scalability, they are read-only and cannot handle write traffic.
