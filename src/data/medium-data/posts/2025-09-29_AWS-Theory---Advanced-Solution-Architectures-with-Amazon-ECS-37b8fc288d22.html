Beyond auto scaling, Amazon ECS supports several innovative solution architectures, particularly when integrated with other AWS services. Here are a few examples:

1. ECS Tasks Invoked by Event Bridge

  • Setup: An ECS cluster backed by Fargate with S3 buckets. Users upload objects to S3, triggering events sent to Amazon Event Bridge via a rule.
  • Process: Event Bridge invokes ECS tasks on demand. Each task has an ECS task role with permissions to access S3 and DynamoDB.
  • Use Case: The task retrieves the S3 object, processes it (e.g., image resizing), and stores the results in DynamoDB.
  • Benefit: This creates a fully serverless architecture for event-driven object processing using Fargate, S3, and DynamoDB.

2. Event Bridge Scheduled Tasks

  • Setup: An ECS cluster on Fargate integrated with Event Bridge, configured with a rule to trigger every hour.
  • Process: The rule launches an ECS task hourly. The task, equipped with an ECS task role for S3 access, performs batch processing (e.g., analyzing files in S3).
  • Benefit: Ideal for scheduled, serverless batch jobs without managing infrastructure.

3. ECS with SQS Queue Integration

  • Setup: An ECS service with two tasks, pulling messages from an SQS queue.
  • Process: As messages accumulate in the queue, ECS Service Auto Scaling increases the task count based on queue depth, processing messages efficiently.
  • Benefit: Scales dynamically with workload, perfect for message-driven applications.

4. Event Bridge for ECS Task Lifecycle Monitoring

  • Setup: Event Bridge intercepts ECS task state changes (e.g., “stopped” events) within the cluster.
  • Process: A rule triggers an action, such as sending an alert via an SNS topic to notify administrators of task exits with reasons.
  • Benefit: Provides visibility into container lifecycle events, enhancing operational monitoring.
