In this hands-on overview, we’ll walk through the process of setting up an Amazon Elastic Kubernetes Service (EKS) cluster on AWS. Note that this is a demonstration for educational purposes and may incur costs, so it’s recommended to follow along conceptually rather than executing it unless you’re prepared for the expense. Let’s explore the steps to create and manage an EKS cluster.

Step 1: Creating an EKS Cluster

  1. Access the Console: Navigate to the Amazon EKS service in the AWS Management Console.

2. Cluster Creation: Click Create Cluster and name it, e.g., demoEKS. Select the default Kubernetes version for simplicity.

3. Cluster IAM role: EKS requires a cluster IAM role to manage resources. Create this role via the IAM console:

  • Go to IAM > Roles > Create Role.
  • Choose EKS as the service, select EKS — Cluster to allow access to related AWS services.
  • The policy is pre-selected; name the role EKSClusterRole (avoid leading spaces) and create it.
  • Refresh the IAM console to confirm the role is available.
  • Attach the require policies by click on Edit in IAM console.
  • Add Tags

4. Node IAM role :

  • Go to IAM > Roles > Create Role.
  • On the Select trusted entity page, do the following:

I. In the Trusted entity type section, choose AWS service.

II. Under Use case, choose EC2.

III. Choose Next.

  • On the Add permissions page, attach a custom policy or do the following:

I. In the Filter policies box, enter AmazonEKSWorkerNodePolicy.

II. Select the check box to the left of AmazonEKSWorkerNodePolicy in the search results.

III. Choose Clear filters.

IV. In the Filter policies box, enter AmazonEC2ContainerRegistryPullOnly.

VI. Choose Next.

  • On the Name, review, and create page, do the following:

I. For Role name, enter a unique name for your role, such as AmazonEKSNodeRole.

II. For Description, replace the current text with descriptive text such as Amazon EKS - Node role.

III. Under Add tags (Optional), add metadata to the role by attaching tags as key-value pairs. For more information about using tags in IAM, see Tagging IAM resources in the IAM User Guide.

IV. Choose Create role.

4. Security and Networking:

  • Skip KMS encryption for secrets to keep it simple (though it’s a security option).
  • Select a VPC and subnets for high availability across multiple Availability Zones.
  • Use the default security group and set the cluster endpoint access to public for external access.
  • Stick with default networking add-ons, proxy, and DNS settings.

5. Logging and Review: Skip control plane logging for now. Review settings (security groups, networking, public API access) and click Create. The cluster will be provisioned.

Step 2: Adding Compute Capacity

Once the cluster is created, you need to add compute capacity:

  • Managed Node Group: To add EC2-based nodes:
  1. Go to Compute > Node Groups > Add Node Group and name it demoNodeGroup.
  2. Select an IAM role for the node group:
  • AmazonEKSNodeRole

3. Configure the node group:

  • Use Amazon Linux 2 as the OS.
  • Choose instance types like t3.medium or t3.micro (adjust based on needs).
  • Set disk size and node scaling (e.g., desired capacity for the ASG).
  • Select subnets and create the node group.
  • AWS manages these nodes, making deployment easier.
  • Fargate Profile: Alternatively, use Fargate for serverless compute:
  • Go to Compute and add a Fargate profile (skipped here but shown as an option).
  • This eliminates the need to provision EC2 instances.

Step 3: Exploring Add-Ons

  • EBS CSI Driver: Under Add-ons, you can install the Amazon EBS CSI driver to enable EBS volumes for persistent storage in your EKS cluster. This also supports EFS and other storage options via CSI drivers.

Step 4: Cleanup

  • Since this is a demo, delete the cluster to avoid costs:
  • Go to the cluster, select demoEKS, and choose Delete.
  • Delete node groups first (skipped in the video for brevity).

Key Takeaways

  • Creating an EKS cluster involves setting up a service role, configuring networking, and adding compute capacity via managed node groups or Fargate.
  • Kubernetes requires specialized knowledge, and EKS offers flexibility with EC2 and Fargate launch modes.
  • Add-ons like the EBS CSI driver enhance storage capabilities.

Conclusion

This hands-on walkthrough highlights the basics of setting up an EKS cluster, though Kubernetes deserves a dedicated course for mastery. Watching this process helps you understand AWS’s managed Kubernetes offering. Experiment cautiously, and consider Fargate for a serverless experience.
