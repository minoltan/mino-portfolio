Introduction

Hey cloud enthusiasts! If you’re building web or mobile applications and need a secure, scalable way to manage user authentication and authorization, Amazon Cognito is your go-to AWS service. Designed to provide identities for external users (outside your AWS account), Cognito simplifies user management and integrates seamlessly with serverless architectures. In this post, we’ll dive into Cognito’s two main components — User Pools and Identity Pools — their features, integrations, and real-world use cases. Whether you’re prepping for an AWS certification or building a user-facing app, this guide has you covered. Let’s get started!

What Is Amazon Cognito?

Amazon Cognito is a fully managed service that provides user identity and access management for web and mobile applications. It’s designed for external users (e.g., app users, not AWS IAM users) and offers two sub-services:

  • Cognito User Pools: A serverless user directory for authentication (sign-in/sign-up).
  • Cognito Identity Pools (formerly Federated Identities): Grants temporary AWS credentials for direct access to AWS resources.

Why Cognito?

Unlike IAM, which manages users and permissions within your AWS account, Cognito is built for external users, such as mobile app users or website visitors. Look for keywords like “hundreds of users,” “mobile users,” or “SAML authentication” in exam questions to identify Cognito as the solution.

Cognito User Pools: Your Authentication Hub

Cognito User Pools provide a serverless user directory for managing sign-in and sign-up for your web and mobile applications. Here’s what they offer:

Features

  • Simple Login: Support for username/email and password authentication.
  • Password Reset: Built-in functionality for users to reset passwords.
  • Email/Phone Verification: Ensure user identities with verification codes.
  • Multi-Factor Authentication (MFA): Add an extra layer of security.
  • Third-Party Logins: Integrate with social providers like Facebook, Google, or SAML/OpenID Connect for federated logins.

Integrations

  • Amazon API Gateway: Authenticate API requests.
  • Application Load Balancer (ALB): Secure access to web applications.

How User Pools Work

  1. A user signs in via the Cognito User Pool (e.g., using email/password or a social provider).
  2. Cognito issues a JSON Web Token (JWT) upon successful authentication.
  3. The application passes the JWT to:
  • API Gateway: Verifies the token and passes the user identity to a Lambda function backend.
  • ALB: Verifies the token and forwards the request to the backend with user identity headers.

4. The backend (e.g., Lambda or EC2) processes the request, knowing the authenticated user’s identity.

Benefits:

  • Offloads authentication logic from your backend to API Gateway or ALB.
  • Ensures secure, verified user identities.
  • Scalable and serverless, with no infrastructure to manage.

Example:

A mobile app user logs in via Cognito User Pool, receives a JWT, and sends it to an API Gateway endpoint. The API Gateway verifies the token, passes the user’s identity to a Lambda function, which then retrieves user-specific data from DynamoDB.

Cognito Identity Pools: Temporary AWS Credentials

Cognito Identity Pools (also known as Federated Identities) provide temporary AWS credentials for authenticated users to directly access AWS resources (e.g., S3, DynamoDB) without going through API Gateway or ALB.

Features

  • Temporary Credentials: Issues AWS IAM credentials with customizable policies.
  • Fine-Grained Access: Policies can be tailored based on user IDs.
  • Default IAM Role: Assign a default role for guest or authenticated users without specific policies.

Sources of Identity

  • Cognito User Pools.
  • Third-party providers (e.g., Facebook, Google, SAML, OpenID Connect).
  • Guest users (unauthenticated access).

How Identity Pools Work

  1. A user authenticates via a Cognito User Pool or a third-party provider and receives a token.
  2. The application sends the token to the Cognito Identity Pool.
  3. The Identity Pool verifies the token and issues temporary AWS credentials with an associated IAM policy.
  4. The user uses these credentials to directly access AWS resources (e.g., read/write to an S3 bucket or DynamoDB table).

Key Use Case: Row-Level Security in DynamoDB

  • The IAM policy generated by the Identity Pool can include a condition, e.g., restricting DynamoDB access to items where the partition key matches the user’s Cognito identity ID.
  • This ensures users only access their own data, not the entire table.

Example

A web app user logs in via Cognito User Pool, exchanges the JWT for temporary credentials via an Identity Pool, and uses those credentials to upload a file directly to an S3 bucket.

Cognito Architecture Example

Here’s how Cognito fits into a serverless application:

Scenario: A mobile app needs secure access to a DynamoDB table and an API Gateway endpoint.

Flow:

  1. User Authentication:
  • The user logs in via Cognito User Pool (e.g., email/password or Google).
  • Cognito issues a JWT.

2. API Access:

  • The app sends the JWT to API Gateway, which verifies it and invokes a Lambda function.
  • The Lambda function processes the request, knowing the user’s identity.

3. Direct AWS Access:

  • The app sends the JWT to a Cognito Identity Pool, which returns temporary AWS credentials.
  • The credentials include an IAM policy restricting DynamoDB access to the user’s own records (row-level security).
  • The app uses the credentials to read/write directly to DynamoDB.

Benefits: Secure, scalable, and serverless, with fine-grained access control.

Real-World Use Case: E-Commerce App Authentication

Imagine an e-commerce mobile app:

User Pools:

  • Users sign up/log in with email/password or Google.
  • MFA is enabled for security.
  • The app sends a JWT to API Gateway to access order history via a Lambda function.

Identity Pools:

  • Users get temporary credentials to upload profile pictures directly to an S3 bucket.
  • DynamoDB row-level security ensures users only access their own order data.

Benefits: Secure user management, seamless integration with serverless APIs, and fine-grained access control.
