Step 1: Access the SQS Console

  1. Log in to your AWS Management Console
  2. Navigate to the Amazon SQS service (you can search for “SQS” in the services search bar)

Step 2: Create a Queue

  1. Click the “Create queue” button
  2. Select “Standard” queue type (FIFO queues will be covered separately)
  3. Enter a name for your queue (e.g., “DemoQueue”)

Step 3: Configure Queue Settings

  1. Visibility timeout: Leave default (30 seconds) — this determines how long a message is invisible after being received
  2. Message retention period: Set to 4 days (default is 4 days) — how long messages stay in queue if not deleted
  3. Maximum message size: 256 KB (the maximum allowed in SQS)
  4. Delivery delay: Leave default (0 seconds) — delay before messages become available

Step 4: Configure Encryption (Optional but Recommended)

  1. By default, encryption is enabled with SSE-SQS (Amazon SQS-managed encryption keys)
  2. Alternative option: Use AWS Key Management Service (KMS)
  • Select “KMS” encryption type
  • Choose a Customer Master Key (CMK)
  • Set data key reuse period (e.g., 5 minutes to limit KMS API calls)

Step 5: Set Access Policy (Optional)

  1. Define who can access the queue using the visual editor
  2. Options:
  • Only queue owner can send/receive messages
  • Specify accounts, users, and roles that can send/receive

3. This generates a JSON policy document similar to S3 bucket policies

Step 6: Create the Queue

  1. Review all settings
  2. Click “Create queue” button
  3. Your queue is now ready to use

Step 7: Send Messages to the Queue

  1. From your queue details page, click “Send and receive messages”
  2. In the message body field, enter your message (e.g., “Hello World!”)
  3. Click “Send message”
  4. The message is now in the queue (visible in “Messages available” count)

Step 8: Receive Messages from the Queue

  1. In the same “Send and receive messages” section, scroll down
  2. Click “Poll for messages”
  3. Your message will appear with details including:
  • Message ID
  • Receipt handle
  • Receive count (how many times it’s been received)
  • Size in bytes

4. Click “Message details” to view:

  • Message body content
  • Metadata (sender, timestamps, etc.)
  • Any message attributes

Step 9: Process and Delete Messages

  1. After processing a message, select it
  2. Click “Delete” to remove it from the queue
  3. This signals to SQS that the message was successfully processed
  4. The “Messages available” count will decrement

Advanced Operations

  1. Sending multiple messages:
  • Send several messages in sequence
  • They’ll all be available for receiving

2. Receiving multiple messages:

  • When polling, you can receive up to 10 messages at once

3. Purging the queue:

  • From queue actions, select “Purge queue”
  • Type “purge” to confirm
  • This deletes all messages immediately (use carefully)

Monitoring and Management

  1. View queue metrics in the “Monitoring” tab:
  • Number of messages in queue
  • Age of oldest message
  • Useful for auto-scaling decisions

2. Edit queue configuration anytime:

  • Encryption settings
  • Access policies
  • Retention periods, etc.

Best Practices

  1. Always delete messages after processing to prevent reprocessing
  2. Consider setting up a dead-letter queue for messages that repeatedly fail processing
  3. Use appropriate visibility timeout based on your processing time
  4. For production, consider using KMS encryption for enhanced security

SQS — Message Visibility Timeout

Setting Up the Test (We already done this):

  1. Create a standard SQS queue with default 30-second visibility timeout
  2. Open two browser windows to simulate two consumers
  3. Send a test message (“Hello World”) to the queue

Observed Behavior:

  1. Consumer 1 receives the message — it disappears from view
  2. Consumer 2 polls during timeout — sees no messages
  3. After ~30 seconds without deletion:
  • Message reappears in queue
  • Now Consumer 2 can receive it
  • Receive count increments to 2

Best Practices for Visibility Timeout

  1. Set Appropriate Defaults:
  • Base timeout on 95th percentile of processing time
  • Add 20–30% buffer beyond average processing time

2. Use ChangeMessageVisibility API:

  • Call when processing takes longer than expected
  • Prevents unnecessary message reappearances

3. Monitor and Adjust:

  • Watch ApproximateAgeOfOldestMessage metric
  • Adjust timeout as your processing patterns change

4. Error Handling:

  • Always implement proper error handling
  • Delete messages only after successful processing
  • Consider dead-letter queues for repeated failures

SQS — Sending Messages to FIFO Queues

1. Queue Creation

  1. Navigate to SQS in AWS Console
  2. Click “Create queue”
  3. Select “FIFO” queue type
  4. Name must end with .fifo (e.g., “OrderProcessing.fifo”)

2. Configuration Options

  • Content-based deduplication: Automatically generates deduplication IDs based on message content
  • Visibility timeout: Same as standard queues (default 30 seconds)
  • Message retention: Up to 14 days (same as standard)
