Why Monolithic Architecture? 

If the Application is Relatively Small — Go with Monolithic Architecture

Traditional Problems ?

  • Scalability Problems : Everything scales together, even if only one feature needs more resources (e.g., a “checkout” service gets overloaded, but you’re forced to scale the entire app).
  • Slow Development & Deployment : Tight coupling, A small change requires rebuilding and redeploying the entire app & Team bottlenecks, Developers wait for others to merge code.
  • Technology Lock-In : Stuck with outdated tech, (e.g., Java 8, old frameworks) because upgrading affects the whole app.
  • Reliability Risks : Single point of failure: One bug crashes the entire app.
  • Performance Bottlenecks: Shared database, slow queries in one module slow down everything.

Why Microservice Architecture?

If the Application is Become Big — Go with Microservice Architecture

  • It enables the continuous delivery and deployment of large, complex applications.
  • Services are small and easily maintained.
  • Services are independently deployable.
  • Services are independently scalable.
  • The microservice architecture enables teams to be autonomous.
  • It allows easy experimenting and adoption of new technologies.
  • It has better fault isolation.

Defining an application’s microservice architecture

A three-step process for defining an application’s microservice architecture

Step 1: Identifying the system operations

  1. The first step creates the high-level domain model consisting of the key classes that provide a vocabulary with which to describe the system operations.
  2. The second step identifies the system operations and describes each one’s behavior in terms of the domain model.
  • Nouns — Derived the domain model 
  • Verbs — Derived system operations

Place Order story

Given a consumer
And a restaurant
And a delivery address/time that can be served by that restaurant
And an order total that meets the restaurant's order minimum
When the consumer places an order for the restaurant
Then consumer's credit card is authorized
And an order is created in the PENDING_ACCEPTANCE state
And the order is associated with the consumer
And the order is associated with the restaurant

Nouns : Consumer, Delivery, Order, Restaurant, and Credit Card

Accept Order story

Given an order that is in the PENDING_ACCEPTANCE state
and a courier that is available to deliver the order
When a restaurant accepts an order with a promise to prepare by a particular
time
Then the state of the order is changed to ACCEPTED
And the order's promiseByTime is updated to the promised time
And the courier is assigned to deliver the order

Noun : Courier
Hidden Noun: Menu Item, and Address

Note: 

While each noun could technically be a service, splitting them all would lead to over-fragmentation, high latency, and operational complexity. Instead, group related domains into bounded contexts (logical service boundaries).

DEFINING SYSTEM OPERATIONS

Once you’ve defined a high-level domain model, the next step is to identify the requests that the application must handle.

There are two types of system operations:

  • Commands — System operations that create, update, and delete data
  • Queries — System operations that read (query) data

The queries provide the UI with the information a user needs to make decisions. At this stage, we don’t have a particular UI design for FTGO application in mind, but consider, for example, the flow when a consumer places an order:
1. User enters delivery address and time.
2. System displays available restaurants.

This user scenario suggests the following queries:

  • findAvailableRestaurants(deliveryAddress, deliveryTime) — Retrieves the restaurants that can deliver to the specified delivery address at the specified time
  • findRestaurantMenu(id) — Retrieves information about a restaurant including the menu items

Step 2: Identify Services

1. Defining services by applying the Decompose by business
capability pattern

Definition:
Split the system into services based on distinct business functions (e.g., “Order Management,” “Inventory,” “Payment”).

Key Traits:

  • Alignment with Org Structure: Mirrors how teams are organized (e.g., Finance team owns the “Billing” service).
  • Stable Boundaries: Capabilities rarely change over time (e.g., “Shipping” is always a core function).

When to Use:

  • Ideal for well-established business processes with clear functional boundaries.
  • Best when organizational teams align directly with business functions.

2. Defining services by applying the Decompose by
sub-domain pattern

Definition:
Split the system based on domain-driven design (DDD) subdomains (e.g., “Order Fulfillment,” “Customer Support,” “Product Catalog”).

Key Traits:

  • Problem-Space Focus: Maps to business problems (not just functions).
  • Bounded Contexts: Each subdomain has its own ubiquitous language and rules (e.g., “Order” means different things in “Fulfillment” vs. “Billing”).

When to Use:

  • Complex domains where business logic varies by context (e.g., an “Order” has different rules in shipping vs. accounting).
  • Teams need autonomy over specific problem areas.


Hybrid Approach

Many companies combine both:

  • Use Business Capabilities for high-level services (e.g., User Management).
  • Use Subdomains for nuanced logic (e.g., User Authentication vs. User Analytics).

Step 3 : Defined Service APIs and Collaborations

  • So far, we have a list of system operations and a list of a potential services. 
  • The next step is to define each service’s API: its operations and events. 
  • A service API operation exists for one of two reasons: 
  1. some operations correspond to system operations. They are invoked by external clients and perhaps by other services.
  2. The other operations exist to support collaboration between services. These operations are only invoked by other services.
  • The starting point for defining the service APIs is to map each system operation to a service. 
  • After that, we decide whether a service needs to collaborate with others to implement a system operation. 
  • If collaboration is required, we then determine what APIs those other services must provide in order to support the collaboration. 
  • Let’s begin by looking at how to assign system operations to services.
