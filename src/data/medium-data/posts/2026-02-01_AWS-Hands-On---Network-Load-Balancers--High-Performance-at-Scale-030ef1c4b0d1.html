🌟 Introduction

However, the Application Load Balancer (ALB) is good fitr for HTTP traffic and flexible routing, in some cases your application needs extreme performance. If our system handling millions of requests per second, dealing with volatile traffic spikes, or require static IP addresses for your load balancer, the Network Load Balancer (NLB) is your best option. It is under Layer 4 (Transport Layer). The NLB focuses on raw speed and efficiency by routing TCP, UDP, and TLS traffic without inspecting the application data.

Key Differences: NLB vs. ALB

  • Layer: NLB works at Layer 4 (IP and Port), while ALB works at Layer 7 (HTTP/HTTPS content).
  • Static IPs: NLB provides a fixed/static IP address per Availability Zone. ALB uses a DNS name with dynamic IPs.
  • Performance: NLB is built for ultra-low latency and massive throughput (millions of requests per second).

Step 1. Configure Security Groups

Previously, NLBs did not support security groups, but now they support security groups.

  • Create or select a Security Group (e.g: demo-sg-nlb).
  • Inbound Rules: Allow HTTP (Port 80) from Anywhere (0.0.0.0/0).
  • Outbound Rules: Allows All Traffic.

Step 2. Create a TCP Target Group

The NLB needs to know where to send the packets.

  • Create a new Target Group.
  • Name: demo-tg-nlb
  • Target Type: Instances.
  • Protocol: TCP on Port 80.
  • Health Checks: Even though it’s a TCP load balancer, you can use HTTP health checks if your app is a web server. This ensures the application is healthy, not just the network port.
  • Register Targets: Select your two existing EC2 instances and click Include as pending below.

Step 3. Initiate the NLB Creation

  • Go to Load Balancers > Create Load Balancer > Network Load Balancer.
  • Name: DemoNLB.
  • Scheme: Internet-facing.
  • Network Mapping: Select your VPC and enable all available Availability Zones (AZs).
  • Note: Notice that for each AZ, AWS assigns a fixed IPv4 address. You could also assign your own Elastic IP here if needed.
  • Attach Security group that we created already in the step 1
  • Attach the target group that we already created
  • Finally, create the NLB
  • After few minutes the NLB is active
  • Try in browser

4. Troubleshooting: The Health Check Challenge

Once created, your NLB might show targets as Unhealthy. This is a common point of confusion.

  • The Problem: Your EC2 instances have a security group that currently only allows traffic from the ALB. Since the NLB has a different security group (or a different IP), the instances are blocking the NLB’s health checks.
  • The Fix:
  1. Go to your EC2 Security Group (launch-wizard-1).
  2. 2. Add a new inbound rule: HTTP (Port 80).
  3. 3. For Source, select the NLB Security Group (demo-sg-nlb).
  • Verification: Wait a minute. Your targets should now switch to Healthy.

🧾 Conclusion

The Network Load Balancer is a power-user tool. By setting one up, you’ve learned how to handle traffic at the transport layer and, more importantly, how to manage “layered” security. In a real-world scenario, you might have an NLB for high-performance TCP traffic and an ALB for your complex web routing. Understanding how to allow traffic from both in your backend security groups is a vital skill for any AWS architect.
