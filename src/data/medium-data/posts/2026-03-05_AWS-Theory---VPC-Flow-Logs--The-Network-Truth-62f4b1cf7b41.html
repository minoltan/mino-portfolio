🌟 Introduction

Imagine you’re a detective trying to solve a cold case of missing network data. You know a request was sent, but it never arrived. How do you find out exactly where it was stopped without intercepting every single piece of live mail?

VPC Flow Logs are your surveillance cameras. They don’t record the content of the conversation (the packet payload), but they record the metadata: who called whom, on what port, for how long, and — most importantly — did the “security guard” let them through or kick them out?

1. What are Flow Logs?

VPC Flow Logs capture information about the IP traffic going to and from network interfaces in your VPC. You can enable them at three levels:

  • VPC Level: Monitors every interface in the entire VPC.
  • Subnet Level: Monitors every interface within a specific subnet.
  • Network Interface (ENI) Level: Monitors a specific instance or service.
  • srcaddr & dstaddr — help identify problematic IP
  • srcpor t & dstpor t — help identity problematic ports
  • Action — success or failure of the request due to Security Group / NACL
  • Can be used for analytics on usage patterns, or malicious behavior
  • Query VPC flow logs using Athena on S3 or CloudWatch Logs Insights

2. Troubleshooting: “Accept” vs. “Reject”

The most powerful field in a flow log is the Action. By combining this with what we know about Security Groups (Stateful) and NACLs (Stateless), we can pinpoint exactly where a connection is failing.

Scenario A: Inbound Reject

If you see a REJECT on an incoming request, the problem is simple: either your Security Group or your NACL has a rule blocking that traffic.

Scenario B: Inbound Accept + Outbound Reject

This is the “aha!” moment for many engineers. If the request was accepted on the way in but rejected on the way out, it must be a NACL issue.

  • Why? Because Security Groups are stateful. If they allow the request in, they automatically allow the response out.
  • The Culprit: NACLs are stateless. You likely forgot to allow the “response” traffic on the ephemeral port range in your NACL’s outbound rules.

3. Analyzing the Data: Athena & QuickSight

Raw flow logs are just lines of text. To get real insights, you need a way to query them.

  • Amazon Athena: Allows you to use standard SQL to query your logs stored in S3. You can ask questions like: “Show me the top 10 IP addresses with the most REJECT actions in the last hour.”
  • Amazon QuickSight: Takes those Athena queries and turns them into visual dashboards. You can see heatmaps of traffic or pie charts of the most-used protocols.

4. Real-World Use Cases

  • Detecting Port Scans: If you see a single outside IP hitting 500 different ports on your instance in 10 seconds, you’re being scanned.
  • Identifying “Top Talkers”: Find out which instance is consuming all your bandwidth (and increasing your NAT Gateway bill).
  • Security Alerts: You can send flow logs to CloudWatch Logs, set a “Metric Filter” for sensitive ports (like SSH 22), and trigger an SNS alert if an unauthorized IP tries to connect.

🧾 Conclusion

VPC Flow Logs are the “black box” flight recorder of your cloud network. They provide the evidence you need to move from “I think the network is broken” to “I know exactly which NACL rule is blocking our traffic.” While they aren’t real-time (there is a slight aggregation delay), they are indispensable for security auditing and post-mortem analysis.
