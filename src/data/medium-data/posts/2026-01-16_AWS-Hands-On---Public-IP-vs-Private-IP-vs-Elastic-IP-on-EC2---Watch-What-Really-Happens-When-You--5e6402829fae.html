Introduction

If you’ve ever stopped an EC2 instance and suddenly couldn’t SSH anymore because the public IP changed, this 5-minute hands-on will end the mystery forever. In the demo you’ll see exactly why the private IP never changes, why the public IP disappears on every stop/start, and how an Elastic IP locks the public address in place. In addition, you can notice the hidden $3.50/month surprise that catches everyone on the bill (for Elastic IP). You can use your own console to avoid being confused.

The Experiment—Do This Right Now

  1. Launch any Linux instance (default VPC, public subnet)
  2. Note three things:
  • Public IPv4 → 3.129.218.15 (changes on stop/start)
  • Private IPv4 → 172.31.11.234 (never changes)
  • Elastic IP → none yet

SSH works with the public IP only (private IP will just time out from your laptop).

Stop → Start = Public IP Lottery

  • Stop the instance → Start it again
  • Result: Private IP stays 172.31.11.234, and the Public IP is now something completely different (e.g., 3.137.222.51)
  • Old SSH command → instant timeout (you’re knocking on a door that no longer exists)

Fix It with an Elastic IP (Static Public Address)

  1. EC2 left menu → Elastic IPs → Allocate Elastic IP address

2. Actions → Associate Elastic IP address → pick your instance → Associate

Refresh your instance page → the Public IPv4 field now shows your Elastic IP (and it says “Elastic IP” underneath).

Now stop → start the instance again → Public IP stays exactly the same! Your SSH command works forever, even after 20 reboots.

The Catch You Must Know (2026 Pricing)

  • Every public IPv4 (normal or Elastic) costs ~$0.005/hour when attached (~$3.50/month)
  • Free Tier gives you 750 hours/month of public IPv4 → enough for one instance 24/7
  • If you leave an Elastic IP unattached or have extra ones → you pay for them even when nothing is running
  • Always release Elastic IPs you’re not using!
  • https://aws.amazon.com/vpc/pricing/

The Modern Way (Skip Elastic IPs Entirely)

99% of real production systems do not use Elastic IPs. Instead they:

  • Keep instances 100 % private
  • Put an Application Load Balancer or Route 53 DNS name in front

That way the IP can change a thousand times, and nobody ever notices.
