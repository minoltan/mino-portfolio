Here’s a step-by-step guide to creating and understanding S3 Lifecycle Rules, based on the provided content:

Objective: Learn how to create an S3 Lifecycle Rule to automatically manage current and non-current object versions, transitioning them between storage classes and eventually expiring or permanently deleting them.

Prerequisites:

✔ An AWS account with access to Amazon S3.

✔ An S3 bucket (versioning enabled on the bucket is necessary for managing non-current versions).

Step 1: Navigate to Your Bucket and Access Lifecycle Rules

  1. Select Your Bucket: In the Amazon S3 console, choose the bucket you want to apply the lifecycle rule to.
  2. Go to “Management” Tab: Click on the “Management” tab for your selected bucket.
  3. Create Lifecycle Rule: Under “Lifecycle rules”, click “Create lifecycle rule”.

Step 2: Configure the Basic Rule Settings

  1. Lifecycle rule name: Enter a name, e.g., DemoRule.
  2. Rule scope:
  • Select “Apply to all objects in the bucket”.
  • Check “I acknowledge that this rule will apply to all objects in the bucket”.

3. Click “Create rule” (Don’t create yet, just move to the next screen).

Hands-On : Click Here

Step 3: Define Actions for Current Versions

  1. Enable “Transition current versions of objects between storage classes”: Check this box.

Add Transition 1:

  • Click “Add transition”.
  • Choose storage class: Select Standard-IA.
  • Days after creation: Enter 30.
Explanation: Current versions will move to Standard-IA after 30 days.

Add Transition 2:

  • Click “Add transition”.
  • Choose storage class: Select Intelligent-Tiering.
  • Days after creation: Enter 60.
Explanation: Current versions will move to Intelligent-Tiering after 60 days.

Add Transition 3:

  • Click “Add transition”.
  • Choose storage class: Select Glacier Instant Retrieval.
  • Days after creation: Enter 90.
Explanation: Current versions will move to Glacier Instant Retrieval after 90 days.

Add Transition 4:

  • Click “Add transition”.
  • Choose storage class: Select Glacier Flexible Retrieval.
  • Days after creation: Enter 180.
Explanation: Current versions will move to Glacier Flexible Retrieval after 180 days.

Add Transition 5:

  • Click “Add transition”.
  • Choose storage class: Select Glacier Deep Archive.
  • Days after creation: Enter 365.
Explanation: Current versions will move to Glacier Deep Archive after 365 days.

Step 4: Define Actions for Non-Current Versions

  • Note: This step requires bucket versioning to be enabled.
  1. Enable “Transition noncurrent versions of objects between storage classes”: Check this box.
  2. Add Transition for Non-Current:
  • Click “Add transition”.
  • Choose storage class: Select Glacier Flexible Retrieval.
  • Days after objects become noncurrent: Enter 90.
Explanation: Non-current versions (older versions of an object that have been overwritten) will move to Glacier Flexible Retrieval 90 days after they become non-current.

Step 5: Define Expiration Actions

  1. Enable “Expire current versions of objects”: Check this box.
  • Days after creation: Enter 700.
Explanation: The current version of an object will be permanently deleted 700 days after its creation.

2. Enable “Permanently delete noncurrent versions of objects”: Check this box.

  • Days after objects become noncurrent: Enter 700.
Explanation: Non-current versions of an object will be permanently deleted 700 days after they become non-current.

3. Enable “Delete expired object delete markers or incomplete multipart uploads”: Check this box.

Explanation: This action cleans up residual delete markers from previous object deletions and incomplete multipart uploads, which can incur costs if not removed.

Step 6: Review and Create the Rule

  1. Review the Rule Summary: Look at the “Summary” section, which displays a timeline visual of your chosen transitions and expirations for both current and non-current versions.
  2. Create Rule: If everything looks correct, click “Create rule”.

Conclusion:

You have successfully created a comprehensive S3 Lifecycle Rule that automates the management of your objects, transitioning them between different storage classes for cost optimization and automatically expiring/deleting them based on predefined timeframes. This rule will run in the background, applying to all objects in your bucket.
