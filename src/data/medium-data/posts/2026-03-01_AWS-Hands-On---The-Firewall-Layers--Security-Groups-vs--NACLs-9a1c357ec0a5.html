🌟 Introduction

In this hands-on lab, we explore the critical differences between Security Groups and Network ACLs. While Security Groups act as a virtual firewall for your EC2 instances at the host level, Network ACLs provide an additional layer of security at the subnet level. The most important distinction we’ll see today is statefulness: Security Groups automatically allow return traffic for any authorized connection, but Network ACLs are “stateless,” meaning you must explicitly permit traffic in both directions. Understanding how rule priority and statefulness work is essential for troubleshooting “infinite loading” screens and building a robust, multi-layered security architecture.

Step-by-Step Guide: Testing NACLs and Security Groups

1. Set Up a Web Server (Bastion Host)

To test our rules, we need a live service. We’ll turn our Bastion Host into a simple web server.

  • Connect to your Bastion Host via SSH or EC2 Instance Connect.
  • Install and start Apache (HTTPD):
sudo yum install -y httpd
sudo systemctl enable httpd
sudo systemctl start httpd
  • Create a simple “Hello World” page:
echo "Hello World" | sudo tee /var/www/html/index.html

2. Update the Security Group

  • In the EC2 Console, navigate to the Security Group for your Bastion.
  • Add an Inbound Rule for HTTP (Port 80) from 0.0.0.0/0.
  • Test: Copy the Bastion’s Public IP into your browser. You should see “Hello World.”

3. Experiment with NACL Rule Priority

Network ACLs evaluate rules in numeric order. The lowest number wins.

  • Navigate to VPC > Network ACLs and select the Default NACL.
  • Add Rule 80 (Deny): Create an Inbound Rule with number 80, type HTTP (80), and set action to Deny.
  • Result: Refresh your browser. The page will time out (infinite loading) because Rule 80 (Deny) is processed before the default Rule 100 (Allow).
  • Flip the Priority: Edit Rule 80 and change the number to 140.
  • Result: Refresh the browser. The page loads again because Rule 100 (Allow) now takes precedence.

4. Prove Statelessness (NACL Outbound Test)

NACLs do not remember “state.” If traffic comes in, it still needs permission to go back out.

  • Go to the Outbound Rules tab of your NACL.
  • Edit the rules and change the default Allow rule to Deny.
  • Result: Even though your Inbound rules allow HTTP, the website will fail to load. The server is receiving the request but the NACL is blocking the “response” from leaving the subnet.

5. Prove Statefulness (Security Group Outbound Test)

Security Groups are smarter — they know that if a user was allowed in, they should be allowed to get a response back.

  • Reset your NACL Outbound rules to Allow so the subnet is clear.
  • Go to your Security Group -> Outbound Rules.
  • Delete the Outbound Rule (which usually allows all traffic).
  • Result: Refresh the web page. It still works! Because the Security Group is stateful, it automatically tracks the inbound connection and allows the return traffic regardless of outbound rules.

🧾 Conclusion

This demo highlights why it works in the security group but not in the browser, which is such a common troubleshooting scenario. By testing rule priority, we saw that NACLs allow for granular Deny rules (like blocking a specific malicious IP), which security groups cannot do. However, we also saw that the stateless nature of NACLs makes them significantly more complex to manage, as you must account for temporary ephemeral ports for return traffic. In a production environment, your security groups should be your primary defense for specific application needs, while NACLs should serve as a broad safety net for the entire subnet.
