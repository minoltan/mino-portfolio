🌟 Introduction

You’ve built the walls and the gates, but now it’s time to actually walk through them. In cloud networking, accessing a server in a private subnet is a rite of passage. Since these instances have no public IP and no direct route to the internet, you can’t simply click “Connect” from the AWS console. Instead, you must use your Bastion Host as a staging area, a “jump box," to reach the isolated parts of your VPC. In this guide, we’ll walk through the manual process of creating a private instance and hopping into it. We’ll even tackle some of the common “real-world” frustrations, like SSH key formatting, that often trip up engineers in the field.

Step 1: Preparing Your Private Instance

Before we can jump, we need a landing spot. We’ll launch a new EC2 instance into our private subnet.

  1. Create a Key Pair: If you don’t have one, go to the EC2 Dashboard > Key Pairs and create demo-key-pair.pem. Save this file securely on your computer.

2. Launch the Instance:

  • Name: Private-Server
  • Subnet: Select PrivateSubnetA.
  • Auto-assign Public IP: Keep this Disabled.
  • Security Group: Create a new one called Private-SG.

3. The Security Rule: This is the most important part. Instead of allowing SSH from “Anywhere,” set the source to the Security Group ID of your Bastion Host.

  • Tip: This ensures that only traffic coming from your Bastion “gatekeeper” is allowed in.

Step 2: Connecting via the “Jump”

Now, let’s perform the double-hop. First, connect to your Bastion Host (which is in the public subnet) using EC2 Instance Connect. Once you’re in the Bastion terminal, you’re officially inside your VPC’s perimeter.

From here, you want to SSH into the private instance using its Private IP (e.g., 10.0.16.x). But there's a catch: the Bastion Host doesn't have your private key yet!

Step 3: Handling the SSH Key (The Manual Way)

While security experts prefer “SSH Agent Forwarding” (to avoid putting keys on the Bastion), for learning purposes, we’ll manually create the key file on the Bastion.

  1. Create the file: Run nano demo-key-pair.pem inside the Bastion terminal.
  2. Paste the Content: Open the .pem file you downloaded earlier on your local computer, copy the text (including the BEGIN and END headers), and paste it into the terminal.
  3. Fix Permissions: SSH is picky. If your key file is “too open,” it will refuse to work. Run: chmod 400 demo-key-pair.pem
  4. The Final Jump: Now run the SSH command using the private IP: ssh -i "demo-key-pair.pem" ec2-user@10.0.x.x
Real-World Note: If you get a “passphrase” prompt or a “format error,” check your copy-paste. Extra spaces or missing newlines at the end of the file are the most common causes of SSH failure!

Step 4: The Reality Check (No Internet)

Once you are logged into the private instance, try to run ping google.com. You’ll notice... nothing happens. The cursor just blinks.

Even though you are “inside” the instance, it is still isolated. It can talk to the Bastion (and you), but it has no way to reach the outside world to download software or updates.

🧾 Conclusion

You’ve successfully navigated from the public internet, through a bastion host, and into an isolated private server. This “jump” logic is the standard way to manage secure backend infrastructure. You’ve seen firsthand that while the bastion host provides a way in, it doesn’t provide the private instance a way out.
