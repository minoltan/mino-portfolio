Question 1

An enterprise runs a microservices-based application on Amazon EKS, deployed on EC2 worker nodes. The application includes a frontend UI service that interacts with Amazon DynamoDB and a data-processing service that stores and retrieves files from Amazon S3. The organization needs to strictly enforce least privilege access: the UI Pods must access only DynamoDB, and the data-processing Pods must access only S3.

Which solution will best enforce these access controls within the EKS cluster?

Overall explanation

✅ Correct option:

Create separate Kubernetes service accounts for the UI and data services. Use IAM Roles for Service Accounts (IRSA) to map each service account to an IAM role with only the required permissions. Assign DynamoDB access to the UI Pods and S3 access to the data Pods

This option is correct because it uses IAM Roles for Service Accounts (IRSA), which is the AWS-recommended method for providing fine-grained IAM permissions to individual Pods in an Amazon EKS cluster. By creating separate Kubernetes service accounts — one for the UI service and one for the data-processing service — and associating each with an IAM role scoped to only the required AWS permissions, the company ensures that each service can access only its designated AWS resource (DynamoDB for the UI, S3 for the data service). IRSA works by leveraging the OpenID Connect (OIDC) identity provider that EKS integrates with, allowing Pods to assume roles dynamically without sharing EC2 instance-level credentials. This approach enforces least privilege, ensures Pod-level isolation, and avoids the security and operational drawbacks of broader node-level roles.

❌ Incorrect options:

Create one Kubernetes service account shared across all Pods. Attach a single IAM role to this account with both AmazonS3FullAccess and AmazonDynamoDBFullAccess policies — This option defeats the purpose of enforcing least privilege. Sharing a single Kubernetes service account with a broad IAM role across all Pods gives both the UI and data service components unnecessary access. Even though IRSA could be used here, the identity boundary is not separated, so the UI Pods can access S3 and the data Pods can access DynamoDB — both of which violate the access requirements.

Attach an IAM policy directly to each Pod using Kubernetes annotations. Assign the S3 policy to data-service Pods and the DynamoDB policy to UI Pods — IAM policies cannot be directly attached to Pods using annotations. AWS IAM does not natively support Pod-level identity mappings outside of IAM Roles for Service Accounts (IRSA). Without IRSA, Pods only inherit permissions from the underlying node’s IAM role, meaning all Pods on the same EC2 node will share the same set of permissions. This option is not technically feasible in EKS.

Create IAM policies for DynamoDB and S3 access, and attach both to the EC2 instance profile used by the EKS nodes. Use Kubernetes role-based access control (RBAC) to control service-level permissions within the cluster — Attaching both IAM policies to the EC2 instance profile means that all Pods running on that node will inherit all permissions, violating least privilege. Kubernetes RBAC controls API access within the cluster, not access to AWS services like S3 or DynamoDB. Therefore, RBAC cannot enforce IAM policy-level restrictions, making this approach insecure and non-compliant with the requirements.

Question 2

An organization wants to delegate access to a set of users from the development environment so that they can access some resources in the production environment which is managed under another AWS account.

As a solutions architect, which of the following steps would you recommend?

Overall explanation

✅ Correct option:

Create a new IAM role with the required permissions to access the resources in the production environment. The users can then assume this IAM role while accessing the resources from the production environment

IAM roles allow you to delegate access to users or services that normally don’t have access to your organization’s AWS resources. IAM users or AWS services can assume a role to obtain temporary security credentials that can be used to make AWS API calls. Consequently, you don’t have to share long-term credentials for access to a resource. Using IAM roles, it is possible to access cross-account resources.

❌Incorrect options:

Create new IAM user credentials for the production environment and share these credentials with the set of users from the development environment — There is no need to create new IAM user credentials for the production environment, as you can use IAM roles to access cross-account resources.

It is not possible to access cross-account resources — You can use IAM roles to access cross-account resources.

Both IAM roles and IAM users can be used interchangeably for cross-account access — IAM roles and IAM users are separate IAM entities and should not be mixed. Only IAM roles can be used to access cross-account resources.
