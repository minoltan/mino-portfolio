How a Garage Rope Taught Me the Most Important LeetCode Pattern: Subarray Sum

The Problem That Broke Me

It was a Thursday evening. I had been staring at the same LeetCode problem for two hours.

“Given an array of integers, find all subarrays whose sum equals k.”

Simple enough, right? I wrote the obvious solution — three loops, check every combination. It worked perfectly on small inputs. I submitted confidently. Time Limit Exceeded.

I tried optimizing the loops. Same result. I searched online, saw something called “prefix sum with HashMap” and had absolutely no idea what it meant.

First — What Is a Subarray?

Before anything, forget the code. Think of your array as a rope with sections:

You can cut any continuous piece. That piece is your subarray. You cannot skip sections — it must be one solid chunk.

Our job: count how many pieces have a total length equal to k.

Approach 1–The Brute Force Way : 3 Nested Loops O(n³)

Rope analogy: pick every start point, pick every end point, then measure the piece from scratch every time by adding each section one by one.

public class Solution {
    public int subarraySum(int[] nums, int k) {
        int count = 0;
        for (int start = 0; start < nums.length; start++) {
            for (int end = start + 1; end <= nums.length; end++) {
                int sum = 0;
                for (int i = start; i < end; i++)
                    sum += nums[i];
                if (sum == k)
                    count++;
            }
        }
        return count;
    }
}

Approach 2– Using Cumulative Sum : 2 Nested Loops O(n²)

Rope analogy: pick a start, then extend j forward one section at a time, keeping a running total — no need to re-measure from scratch.

public class Solution {
    public int subarraySum(int[] nums, int k) {
        int count = 0;
        int[] sum = new int[nums.length + 1];
        sum[0] = 0;
        for (int i = 1; i <= nums.length; i++)
            sum[i] = sum[i - 1] + nums[i - 1];
        
        for (int start = 0; start < nums.length; start++) {
            for (int end = start + 1; end <= nums.length; end++) {
                if (sum[end] - sum[start] == k)
                    count++;
            }
        }
        return count;
    }
}

Approach 3 — Prefix Sum Array O(n²)

Rope analogy: first walk the entire rope once and write all marks down in a fixed array. Then use those marks to calculate any piece’s length instantly — but still check every (i,j) pair.

public class Solution {
    public int subarraySum(int[] nums, int k) {
        int count = 0;
        for (int start = 0; start < nums.length; start++) {
            int sum=0;
            for (int end = start; end < nums.length; end++) {
                sum+=nums[end];
                if (sum == k)
                    count++;
            }
        }
        return count;
    }
}

Approach 4 — Prefix Sum + HashMap O(n)

Rope analogy: walk once, write marks in a notebook, and at every position instantly check the notebook instead of ever going back.

public int subarraySum(int[] nums, int k) {
    int count = 0;
    int sum = 0;
    HashMap<Integer, Integer> map = new HashMap<>();
    map.put(0, 1);  // write mark 0 before walking

    for(int i = 0; i < nums.length; i++){
        sum += nums[i];              // walk forward, update total
        int need = sum - k;          // what cut mark am I looking for?

        if(map.containsKey(need)){   // open notebook, check instantly
            count += map.get(need);  // each stored count = one valid subarray
        }

        map.put(sum, map.getOrDefault(sum, 0) + 1); // write new mark
    }
    return count;
}

How This Solves Real Industrial Problems

This isn’t just a LeetCode trick. The same rope-and-notebook idea runs inside systems you use every day.

1. The “Calorie Goal” (Health App)

Imagine you have a fitness goal: Eat exactly 500 calories ($k=500$) in a single sitting (contiguous snacks).

  • The Food Log: [100, 200, 300, 100, 100, 400]
  • The Brute Force Way: Every time you eat a grape, you re-add every combination of previous snacks to see if any group equals 500. (Exhausting!)
  • The Map Way: You just keep a Running Total in your head.
  • You eat 100, then 200, then 300. Your total is 600.
  • You ask: “Did I ever have a total of 100 earlier?” ($600–500 = 100$).
  • Yes! You remember hitting 100 after the first snack.
  • Result: You instantly know the snacks after that first one (the 200 and 300) equal exactly 500.

2. The “Loyalty Spike” Detector (Target $k=500$)

The Goal: Find any sequence of purchases that adds up to exactly $500.

Orders: [200, 300, 50, 0, 150, 350]

The Map:

  • Day 2: Total is $500. (Check: $500–500 = 0$. We have the “0” mark! Count = 1).
  • Day 4: Total is $550 (after a $0 order). Map records $550 exists twice now.
  • Day 5: Total is $700. (Check: $700–500 = 200$). Map has “200” from Day 1. Count = 2.
  • Day 6: Total is $1050. (Check: $1050–500 = 550$).

The Value: You check your map for “550”. It says you hit $550$ twice before (Day 3 and Day 4).

Result: You found 2 more streaks ending today!

The Night It All Clicked

That Thursday when I grabbed the rope, I stopped thinking about code and started thinking about the problem itself. A rope. A cut. A notebook. That’s it.

Every time you see a problem asking for sums over subarrays or ranges, remember:

“Walk the rope once. Write every mark in a notebook. At each step, just check the notebook — never walk back.”

That’s prefix sum with HashMap. That’s the whole idea. And once you see it with a rope, you’ll never forget it.
