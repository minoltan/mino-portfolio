Introduction

In this blog post, we’ll explore how to design a serverless architecture for a mobile application called MyTodoList. This app allows users to manage their to-do lists with high read throughput, secure authentication, and direct interaction with their own Amazon S3 folder. We’ll leverage AWS services to build a scalable, cost-effective, and fully managed solution.

Requirements

The MyTodoList app has the following requirements:

  • Expose a REST API with HTTPS endpoints.
  • Use a serverless architecture.
  • Allow users to interact directly with their own folder in Amazon S3 for data management.
  • Provide user authentication through a managed serverless service.
  • Support high read throughput for to-dos, as they are read more often than written.
  • Ensure the database layer scales efficiently and optimizes for performance and cost.

Architecture Overview

To meet these requirements, we’ll use a combination of AWS serverless services: Amazon API Gateway, AWS Lambda, Amazon DynamoDB, Amazon Cognito, and Amazon S3. We’ll also incorporate caching to optimize read performance and reduce costs.

Step 1: REST API with Amazon API Gateway

The mobile clients will interact with the app via a REST API exposed through Amazon API Gateway. This service provides HTTPS endpoints and integrates seamlessly with other AWS services. API Gateway will invoke AWS Lambda functions to handle requests, ensuring scalability and serverless compute.

Hands-On:

  1. Go to API Gateway Console > Create API > REST API > Build.
  2. Name: MyTodoListAPI. Create Resource: /todos with GET and POST methods.
  3. Set Integration Type to Lambda Proxy (created in Step 4). Deploy API to stage prod.
  4. Enable caching on GET method (TTL: 300 seconds) under Stage Settings.
  5. CLI:
aws apigateway create-rest-api --name MyTodoListAPI
aws apigateway create-resource --rest-api-id <API-ID> --parent-id <ROOT-ID> --path-part todos
aws apigateway put-method --rest-api-id <API-ID> --resource-id <RESOURCE-ID> --http-method GET --authorization-type COGNITO_USER_POOLS --authorizer-id <COGNITO-AUTH-ID>
aws apigateway create-deployment --rest-api-id <API-ID> --stage-name prod

6. Test endpoint with Postman after setting up Lambda and Cognito.

Step 2: Data Storage with Amazon DynamoDB

The to-do data will be stored in Amazon DynamoDB, a serverless NoSQL database that scales automatically and supports high read and write throughput. DynamoDB is ideal for this use case due to its ability to handle large-scale, low-latency queries.

Hands-On:

  1. Go to DynamoDB Console > Create Table.
  2. Table Name: Todos, Partition Key: userId (String), Sort Key: todoId (String). Use On-Demand capacity.
  3. CLI:
aws dynamodb create-table --table-name Todos 
--attribute-definitions AttributeName=userId,AttributeType=S AttributeName=todoId,AttributeType=S 
--key-schema AttributeName=userId,KeyType=HASH AttributeName=todoId,KeyType=RANGE --billing-mode PAY_PER_REQUEST

4. Verify table creation and test sample data insertion via AWS Console or CLI:

aws dynamodb put-item --table-name 
Todos --item '{"userId": {"S": "user1"}, "todoId": {"S": "todo1"}, "task": {"S": "Buy groceries"}}'

Step 3: User Authentication with Amazon Cognito

To secure the application, we’ll use Amazon Cognito for user authentication. Cognito is a managed serverless service that handles user sign-up, sign-in, and access control. Mobile clients will authenticate with Cognito, which will generate temporary AWS credentials. These credentials allow secure access to API Gateway and other AWS services.

API Gateway will verify the authentication tokens provided by Cognito, ensuring only authorized users can access the API.

Hands-On:

  1. Go to Cognito Console > Create User Pool.
  2. Name: MyTodoListUsers. Enable email as username, add attributes (e.g., email). Create App Client (no OAuth, no client secret).
  3. Create Identity Pool: Cognito > Federated Identities > Create new > Link User Pool.
  4. IAM Roles: Auto-create roles. Edit authenticated role:
{
  "Version": "2012-10-17",
  "Statement": [
    {"Effect": "Allow", "Action": ["dynamodb:*"], "Resource": "arn:aws:dynamodb:*:*:table/Todos"},
    {"Effect": "Allow", "Action": ["s3:PutObject", "s3:GetObject"], "Resource": "arn:aws:s3:::mytodolist-bucket/${cognito-identity.amazonaws.com:sub}/*"}
  ]
}

5. CLI:

aws cognito-idp create-user-pool --pool-name MyTodoListUsers
aws cognito-idp create-user-pool-client --user-pool-id <POOL-ID> --client-name MyTodoListClient
aws cognito-identity create-identity-pool --identity-pool-name MyTodoListIdentity --cognito-user-pools UserPoolId=<POOL-ID>

6. Note User Pool ID, App Client ID, Identity Pool ID for mobile integration.

Step 4: Direct S3 Access for Users

Users should be able to manage their data in a dedicated Amazon S3 folder. To achieve this securely:

  • Mobile clients authenticate with Amazon Cognito, which returns temporary AWS credentials.
  • These credentials are scoped to allow access only to the user’s specific S3 folder, following a restricted policy.
  • Users can then upload and retrieve files directly from their S3 folder.

Important Note: Storing AWS user credentials on the mobile client is a security risk and should never be done. Instead, rely on Cognito’s temporary credentials for secure access.

Hands-On:

  1. Go to S3 Console > Create Bucket: mytodolist-bucket. Enable versioning.
  2. Ensure Cognito Identity Pool authenticated role allows S3 access (see Step 3).
  3. In mobile app, use AWS SDK/Amplify to fetch credentials and access S3:
import { Auth, Storage } from 'aws-amplify';
await Auth.signIn('user@example.com', 'Pass123!');
await Storage.put('user-folder/file.txt', 'content', { level: 'protected' });

4. CLI Test:

aws s3 cp test.txt s3://mytodolist-bucket/<USER-ID>/test.txt --profile cognito-user

5. Verify user-specific folder access in S3 Console.

Step 5: Optimizing Read Performance

Since the app has a high read throughput for to-dos that rarely change, we can optimize performance and reduce costs using caching.

DynamoDB Accelerator (DAX)

To improve read performance, we’ll add DynamoDB Accelerator (DAX), a fully managed in-memory caching layer for DynamoDB. DAX caches frequently accessed data, reducing the number of read capacity units (RCUs) required from DynamoDB. This results in:

  • Faster response times for read-heavy workloads.
  • Lower costs by reducing the need for provisioned read capacity.

Hands-On:

  1. Go to DAX Console > Create Cluster.
  2. Cluster Name: TodosDAX, Node Type: cache.t3.small, Subnet Group: Default VPC. Attach to Todos table.
  3. IAM Role: Auto-create with DynamoDB access.
  4. CLI:
aws dax create-cluster --cluster-name TodosDAX --node-type cache.t3.small --iam-role-arn <ROLE-ARN>

5. Update Lambda code to use DAX endpoint:

import boto3
ddb = boto3.resource('dynamodb', endpoint_url='<DAX-ENDPOINT>')
table = ddb.Table('Todos')
response = table.get_item(Key={'userId': 'user1', 'todoId': 'todo1'})

6. Monitor DAX hit rate in CloudWatch (>90% for cost savings).

API Gateway Caching

For API responses that are static or rarely change, we can enable caching at the Amazon API Gateway level. By caching responses for specific API routes, we reduce the number of Lambda invocations and DynamoDB queries, further improving performance and lowering costs.

Hands-On:

  1. In API Gateway Console, go to Stages > prod > Settings.
  2. Enable caching (TTL: 300 seconds) for GET /todos.
  3. CLI:
aws apigateway update-stage 
--rest-api-id <API-ID> --stage-name prod 
--patch-operations op=replace,path=/caching/ttlInSeconds,value=300

4. Test caching: Repeated GET requests should return cached responses (check CloudWatch logs).

Final Architecture

The complete architecture looks like this:

  • Mobile Clients: Authenticate with Amazon Cognito and interact with the REST API via Amazon API Gateway.
  • Amazon API Gateway: Handles HTTPS requests, verifies Cognito tokens, and invokes AWS Lambda functions. Optional caching is enabled for static responses.
  • AWS Lambda: Processes requests and interacts with DynamoDB for to-do data.
  • Amazon DynamoDB: Stores to-do data with high scalability. DAX is used for caching to optimize read performance.
  • Amazon Cognito: Manages user authentication and provides temporary credentials for secure S3 access.
  • Amazon S3: Allows users to store and retrieve files in their own folder using temporary credentials from Cognito.

This architecture is fully serverless, meaning AWS manages all infrastructure, and you only pay for usage. There’s no need to provision or manage servers, databases, or caching layers.

Key Benefits

  • Scalability: All components (API Gateway, Lambda, DynamoDB, Cognito, S3) scale automatically with demand.
  • Cost Efficiency: Pay only for what you use, with caching (DAX and API Gateway) reducing costs for read-heavy workloads.
  • Security: Cognito provides robust authentication and temporary credentials, ensuring secure access to S3 and the API.
  • Performance: DAX and API Gateway caching optimize read throughput for frequently accessed to-dos.
  • Simplicity: AWS handles all infrastructure management, allowing developers to focus on application logic.

Conclusion

The MyTodoList app demonstrates a classic serverless architecture using AWS services. By leveraging Amazon API Gateway, AWS Lambda, Amazon DynamoDB, Amazon Cognito, and Amazon S3, we built a scalable, secure, and cost-effective solution. Adding DynamoDB DAX and API Gateway caching ensures high read performance and cost optimization for read-heavy workloads. This architecture is a great example of how serverless technologies can simplify development while meeting modern application requirements.
