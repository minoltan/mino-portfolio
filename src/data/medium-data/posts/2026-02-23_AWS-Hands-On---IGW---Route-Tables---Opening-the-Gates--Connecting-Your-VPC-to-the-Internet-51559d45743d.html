🌟 Introduction

In our previous steps, we built a beautiful digital house with rooms (subnets) and a solid foundation. But right now, it’s like a house without any phone lines or internet, completely cut off from the world. If you launch an EC2 instance inside it, you won’t be able to SSH into it, and the instance won’t be able to download updates. To fix this, we need to do three things. Give our instances a public identity (Public IP), install a “front door” (Internet Gateway), and write the “directions” (route tables) to tell traffic how to find that door.

Step 1: Enabling “Auto-Assign Public IP”

By default, instances in a custom VPC don’t get public IP addresses. Without a public IP, the internet has no way to “see” your instance.

  1. Go to Subnets in the VPC Console.
  2. Select PublicSubnetA, click Actions, and choose Edit subnet settings.

3. Check the box for Enable auto-assign public IPv4 address.

4. Repeat this for PublicSubnetB. Now, any instance you launch here will automatically get a public IP, saving you from having to enable it manually every single time.

Step 2: Creating the Internet Gateway (IGW)

The Internet Gateway is the bridge between your private VPC and the public internet. It’s highly available and scales automatically. You just need to create it and “plug it in.”

  1. In the left sidebar, click Internet Gateways > Create internet gateway.
  2. Name it DemoIGW and click Create.

3. Once created, it will be “Detached.” Select it, click Actions > Attach to VPC, and choose your DemoVPC.

Step 3: Organizing Your Route Tables

Every VPC comes with a “Main” route table, but using it for everything is risky. It’s better to create explicit tables for clarity.

  1. Create Two Tables: Create one named PublicRouteTable and another named PrivateRouteTable, both assigned to DemoVPC.

2. Associate Subnets: * Assign your two Public Subnets to the PublicRouteTable.

3. Assign your two Private Subnets to the PrivateRouteTable.

This separation ensures that if you change a setting for the public web servers, you don’t accidentally expose your private database.

Step 4: Adding the “Internet Route”

This is the final “Aha!” moment. We need to tell it PublicRouteTable that if it sees traffic destined for the outside world, it should send it to our new gateway.

  1. Select PublicRouteTable and click the Routes tab > Edit routes.
  2. Click Add route.
  3. Destination: 0.0.0.0/0 (This represents "The entire Internet").
  4. Target: Select Internet Gateway and pick your DemoIGW.
  5. Save changes.

The Result: Successful Connectivity

Now, if you try to connect to an EC2 instance in your Public Subnet using EC2 Instance Connect, it will work! You can run a command like ping google.com and see the data flowing. Your instance can now "see" the world, and the world can "see" it.

🧾 Conclusion

You’ve successfully turned your isolated island into a connected hub. By configuring Public IPs, an Internet Gateway, and Route Tables, you’ve mastered the core networking flow of AWS. However, while our public instances are happy, our Private Subnets are still in the dark. They can’t reach the internet even to download security patches because they don’t have a route to the outside world.
