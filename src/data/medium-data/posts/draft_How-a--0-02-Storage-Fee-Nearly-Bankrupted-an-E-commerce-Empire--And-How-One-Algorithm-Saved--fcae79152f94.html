How a $0.02 Storage Fee Nearly Bankrupted an E-commerce Empire (And How One Algorithm Saved Millions)

The $4.7 Million Wake-Up Call

Sarah Chen stared at her laptop screen in disbelief. Her e-commerce company had just closed its best year ever — $12 million in revenue, 50,000 units sold. The champagne was already on ice.

Then her CFO walked in with a single number that changed everything: $4.7 million in warehouse storage fees.

“That’s impossible,” Sarah said. “We’re constantly selling. The inventory is moving.”

Her CFO slid a spreadsheet across the table. “Not fast enough. Look at June through November. You sold 8,000 units, yes. But those same 8,000 units sat in the warehouse for a combined 8,000 storage days before selling. At $0.60 per unit per day, you made exactly… zero dollars.”

Sarah had discovered what data scientists call The Zero-Sum Trap — and what we’re about to solve using the Contiguous Array Algorithm.

Understanding the Invisible Profit Killer

The Problem: When “Busy” Doesn’t Mean “Profitable”

Let’s simplify Sarah’s situation. Imagine tracking your business day-by-day:

  • Day 1: A product sells → You earn $10 profit → Mark this as +1
  • Day 2: No sale, product sits in warehouse → You pay $10 storage → Mark this as -1
  • Day 3: Another sale → Earn $10 → Mark as +1
  • Day 4: No sale → Pay $10 → Mark as -1

After 4 days, your pattern looks like this:

[+1, -1, +1, -1]

Your total profit? Exactly $0.

You worked for 4 days. Your staff packed boxes. Your ads ran. But financially? You ran in place.

The Leetcode Translation

In the classic Contiguous Array problem, you’re given an array of 0s and 1s:

[0, 1, 0, 1, 1, 0]

Your mission: Find the longest subarray where the number of 0s equals the number of 1s.

Why does this matter?

Because in business:

  • 1 = A sale (profit flows in)
  • 0 = A storage day (profit flows out)

When they balance perfectly, you’re working for free.

The Story of Discovery

Sarah’s First Attempt: The Brute Force Method

Sarah’s team built a simple tool. For every possible date range, they counted sales vs. storage days:

Check Days 1-2: 1 sale, 1 storage day → Balanced ✓
Check Days 1-3: 2 sales, 1 storage day → Not balanced
Check Days 1-4: 2 sales, 2 storage days → Balanced ✓
Check Days 2-3: 1 sale, 1 storage day → Balanced ✓
...

The Code (Beginner’s Solution)

public class InventoryAnalysis {

    public static int findBreakEvenStreakNaive(int[] dailyActivity) {
        int maxStreak = 0;
        int n = dailyActivity.length;

        // Try every possible starting point
        for (int start = 0; start < n; start++) {
            int sales = 0;
            int storageDays = 0;

            // Try every possible ending point from this start
            for (int end = start; end < n; end++) {
                // Count what happened on this day
                if (dailyActivity[end] == 1) {
                    sales++;
                } else {
                    storageDays++;
                }

                // Check if balanced
                if (sales == storageDays) {
                    int streakLength = end - start + 1;
                    maxStreak = Math.max(maxStreak, streakLength);
                }
            }
        }

        return maxStreak;
    }
}

Time Complexity: O(n²) | Space Complexity: O(1)

The Problem? With 180 days of data, this approach checked 16,290 possible ranges. The algorithm took 3 seconds to run. For a year of data? Over 2 minutes. Sarah needed faster.

The Breakthrough — The Running Balance Trick

Sarah’s data scientist, Marcus, had an idea.

“What if we don’t count from scratch every time?” he said. “What if we keep a running balance instead?”

The Key Insight: Prefix Sums (The “Bank Account” Analogy)

Think of your business like a bank account:

  • Every sale: +$1
  • Every storage day: -$1

Let’s track the running balance for this sequence:

Day:      1    2    3    4    5    6    7    8
Action:   1    0    1    0    0    1    1    0
Balance:  +1   0   +1   0   -1    0   +1   0

The Magic Moment: Notice Days 2, 4, 6, and 8 all have a balance of 0.

Marcus’s revelation: “When your balance returns to 0, everything from the start until now is break-even!"

But there’s more…

What if your balance is +5 on Day 20, and +5 again on Day 80?

That means between Day 20 and Day 80 (60 days), your net change was zero. You added and subtracted the same amount. You were break-even for those 60 days.

The Optimized Solution — The Hash Map Memory

The Algorithm That Saved Millions

Here’s the optimized approach:

  1. Track your running balance as you go through each day
  2. Remember the first time you see each balance (store it in a hash map)
  3. When you see a balance again, calculate: “How many days passed since I first saw this balance?”

The Code (Optimized Solution)

import java.util.HashMap;
import java.util.Map;

public class OptimizedInventoryTracker {
    public static int findBreakEvenStreakOptimized(int[] dailyActivity) {
        // Hash map: balance -> first day we saw this balance
        Map<Integer, Integer> firstSeen = new HashMap<>();
        
        // Base case: Balance of 0 seen at "day -1" (before we start)
        // This is crucial for catching streaks that start from the very first day.
        firstSeen.put(0, -1);
        
        int runningBalance = 0;
        int maxStreak = 0;
        
        for (int day = 0; day < dailyActivity.length; day++) {
            // Update running balance: 1 is +1, 0 is -1
            if (dailyActivity[day] == 1) {
                runningBalance++;
            } else {
                runningBalance--;
            }
            
            // Have we seen this balance before?
            if (firstSeen.containsKey(runningBalance)) {
                // If the balance is the same as a previous day, 
                // the net change between those two days is zero.
                int firstDay = firstSeen.get(runningBalance);
                int streakLength = day - firstDay;
                maxStreak = Math.max(maxStreak, streakLength);
            } else {
                // First time seeing this balance, remember it.
                // We only store the FIRST time to maximize the streak length.
                firstSeen.put(runningBalance, day);
            }
        }
        
        return maxStreak;
    }
}

Time Complexity: O(n) | Space Complexity: O(n)
