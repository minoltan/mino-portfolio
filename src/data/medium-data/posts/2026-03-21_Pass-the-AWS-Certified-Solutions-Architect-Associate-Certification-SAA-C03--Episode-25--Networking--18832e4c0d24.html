🌟 Introduction

The AWS Certified Solutions Architect Associate (SAA-C03) certification demands a profound understanding of networking. The Virtual Private Cloud (VPC) serves as the fundamental bedrock upon which all cloud architectures are built.

In this series, I have consolidated a high-velocity learning path that transitions from theoretical concepts like CIDR blocks and the “five reserved IP” rule to the hands-on mastery of building secure, multi-tier environments. We explore the critical distinctions between stateful Security Groups and stateless Network ACLs, and evolve from legacy NAT Instances to sophisticated Transit Gateways. Whether you are aiming to secure private data with Bastion Hosts or optimize global connectivity via Direct Connect, these summaries offer a roadmap to mastering the “digital backyard” of the AWS cloud.

Summaries of the Series Posts

  1. AWS Theory | An Introduction to VPC & CIDR

In this blog I explained how an AWS VPC acts like a secure, private office building in the cloud. It provides full control over your isolated network environment. I detailed how CIDR blocks manage IP addresses, allowing you to organize resources into subnets and define how they communicate.

2. AWS Hands-On | VPC: Building Your Own Digital Backyard

In this blog I demonstrated how to create a manual VPC. We defined digital boundaries using a CIDR block (like $10.0.0.0/16$) and chose hardware tenancy. I showed how AWS automatically generates essential security and routing components behind the scenes to establish a secure foundation.

3. AWS Theory | Subnets: Carving Out Your Space in the Cloud

In this blog I explained how subnets divide a large VPC into smaller sections. I highlighted a critical exam detail: AWS always reserves five specific IP addresses in every subnet for internal networking.

4. AWS Hands-On | Subnets: (Public and Private) Dividing the House

In this blog I demonstrated the process of carving out public subnets for web servers and private subnets for data storage. I showed how to distribute these across multiple Availability Zones (AZs) to ensure high availability.

5. AWS Theory | Internet Gateway (IGW): The Front Door to the Web

In this blog I explained that the Internet Gateway (IGW) is the “front door” for your VPC. It is highly available and performs the necessary address translation to allow cloud resources to talk to the global internet.

6. AWS Hands-On | IGW & Route Tables | Opening the Gates: Connecting Your VPC to the Internet

In this blog I demonstrated how to attach an IGW to a VPC. I showed how to update Route Tables to direct traffic from public subnets to the internet using a $0.0.0.0/0$ route, while keeping private subnets isolated.

7. AWS Theory | Bastion Host: The Gatekeeper

In this blog I explained the role of a Bastion Host (or “jump server”). It sits in a public subnet and acts as a single, hardened entry point for administrators to access private instances without exposing them to the internet.

8. AWS Hands-On | Bastion Host: Crossing the Bridge to Your Private Subnet

In this blog I demonstrated how to use a “jump box” to access a private server. This involved configuring specific Security Group rules and managing SSH keys to bridge the gap between public and private instances safely.

9. AWS Theory | NAT Instances: The Cloud’s Legacy Bridge

In this blog I explained NAT Instances, a legacy EC2-based solution. I noted that you must manually disable the “Source/Destination Check” for them to work. While good for learning, they are largely replaced by NAT Gateways.

10. AWS Theory | Scaling with NAT Gateways: The Managed Bridge

In this blog I explained the NAT Gateway, a fully managed service that allows private resources to reach the internet (for updates) while blocking the internet from starting connections back to them.

11. AWS Hands-On | Scaling with NAT Gateways: The Managed Bridge

In this blog I demonstrated how to deploy a NAT Gateway in a public subnet and assign it an Elastic IP. I showed how updating the private route table provides a scalable, maintenance-free path to the web for your backend servers.

12. AWS Theory | The Firewall Layers: Security Groups vs. NACLs

In this blog I explained the two-layer defense: Security Groups (stateful, at the instance level) and Network ACLs (stateless, at the subnet level).

13. AWS Hands-On | The Firewall Layers: Security Groups vs. NACLs

In this blog I demonstrated that Security Groups automatically allow return traffic, whereas NACLs require explicit rules for both directions. I also showed how lower-numbered NACL rules take priority.

14. AWS Theory | VPC Peering: Bridging the Gap

In this blog I explained VPC Peering, a private bridge between two VPCs. I emphasized that it is non-transitive — meaning if VPC A peers with B, and B peers with C, A cannot talk to C unless you create a direct link.

15. AWS Hands-On | VPC Peering: Bridging the Gap

In this blog I demonstrated the peering “handshake” process. I showed how to accept a request and manually update route tables in both VPCs so they can communicate using private IP addresses.

16. AWS Theory | VPC Endpoints: The Secret Tunnel

In this blog I explained how VPC Endpoints keep traffic off the public internet when accessing services like S3 or DynamoDB. This reduces costs and increases security.

17. AWS Hands-On | VPC Endpoints: The Secret Tunnel

In this blog I demonstrated setting up an S3 Gateway Endpoint. I showed how a private instance can access storage buckets without a NAT Gateway, keeping all communication within the AWS backbone.

18. AWS Theory | VPC Flow Logs: The Network Truth

In this blog I explained VPC Flow Logs, which act like “surveillance cameras.” They record metadata about IP requests to help you troubleshoot connectivity or detect malicious activity.

19. AWS Hands-On | VPC Flow Logs: The Network Truth

In this blog I demonstrated how to stream logs to CloudWatch and use Amazon Athena to run SQL queries. I showed how to identify which IP addresses are being rejected by your firewalls.

20. AWS Theory | AWS Site-to-Site VPN: Extending Your Network

In this blog I explained how to connect an on-premises data center to AWS using encrypted tunnels over the internet, utilizing a Virtual Private Gateway and a Customer Gateway.

21. AWS Theory | AWS Direct Connect: The High-Speed Freeway

In this blog I explained Direct Connect (DX), a physical, dedicated link between your office and AWS. It bypasses the internet entirely for low-latency, high-speed performance.

22. AWS Theory | Transit Gateway: The Central Station

In this blog I explained how the Transit Gateway acts as a central hub. It replaces messy peering webs with a “star” architecture, allowing thousands of VPCs to connect through one managed point.

23. AWS Theory | VPC Traffic Mirroring: The Digital Security Camera

In this blog I explained this “digital security camera” that copies raw packets from a network interface for deep inspection by security appliances, without impacting production performance.

24. AWS Theory | IPv6 in AWS: The Next Generation of Networking

In this blog I explained Dual-stack VPCs, which allow resources to use both IPv4 and IPv6. This solves address exhaustion while maintaining backward compatibility.

25. AWS Hands-On | IPv6 in AWS: The Next Generation of Networking

In this blog I demonstrated assigning an Amazon-provided /56 CIDR block to a VPC. I showed how to configure IPv6-specific rules in Route Tables and Security Groups.

26. AWS Theory | AWS Egress-Only Internet Gateways: Keeping the Door Locked

In this blog I explained the Egress-Only Internet Gateway (EOIGW). It provides a secure, one-way path for IPv6 traffic, allowing outbound updates while blocking unsolicited inbound connections.
