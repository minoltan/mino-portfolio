Introduction

Hey cloud adventurers! Ready to get your hands dirty with Amazon DynamoDB, AWS’s fully managed, serverless NoSQL database? In this step-by-step guide, we’ll walk through creating a DynamoDB table, configuring its settings, and adding items — all from the AWS Management Console. This hands-on tutorial is perfect for beginners, AWS certification prep, or anyone curious about DynamoDB’s power. We’ll focus on key concepts like table creation, capacity modes, and schema flexibility, with a sprinkle of exam tips. Let’s dive into the AWS Console and build something awesome!

Why DynamoDB? A Quick Recap

DynamoDB is a serverless NoSQL database that scales effortlessly, delivers single-digit millisecond performance, and requires zero server management. Unlike relational databases (e.g., RDS or Aurora), you don’t create a database instance — just jump straight to creating tables. Its schema-less design lets you evolve data structures dynamically, making it ideal for applications with changing requirements. Plus, it’s cost-efficient with a generous free tier. Let’s see it in action!

Step 1: Navigating to DynamoDB in the AWS Console

  1. Log in to the AWS Management Console: Head to the AWS Console and search for DynamoDB in the services menu.
  2. Access the DynamoDB Dashboard: Click on DynamoDB to enter the dashboard. You’ll see options like Tables, Backups, and Streams. For now, we’re focusing on creating a table, so click Create table.

Step 2: Creating a DynamoDB Table

Let’s create a simple table called DemoTable:

  1. Table Name: Enter DemoTable in the Table name field.
  2. Primary Key:
  • Partition Key: Required for every table. Enter userID as the partition key and set its type to String. This key distributes data across partitions for scalability.
  • Sort Key: Optional for sorting items within a partition. For simplicity, leave this blank (we’ll skip the sort key for now).

3. Customize Settings: By default, DynamoDB uses basic settings, but let’s click Customize settings to explore options critical for exams and real-world use.

Step 3: Configuring Capacity Settings

DynamoDB offers two capacity modes for managing read and write throughput. Let’s explore both:

Option 1: On-Demand Mode

  • What It Is: Pay per read/write request, with automatic scaling based on workload.
  • When to Use: Ideal for unpredictable workloads or apps with sudden spikes (e.g., scaling from 1,000 to 1 million requests in a minute).
  • Cost: 2–3x more expensive than provisioned mode, but no capacity planning needed.
  • Console Setup: Select On-demand under Read/write capacity settings.

Option 2: Provisioned Mode

  • What It Is: Specify Read Capacity Units (RCUs) and Write Capacity Units (WCUs) in advance to control costs.
  • When to Use: Best for predictable workloads where you can estimate traffic and want to optimize costs.
  • Auto-Scaling Option:
  • Enable Auto-scaling to dynamically adjust RCUs and WCUs based on load.
  • Set Minimum capacity (e.g., 1 RCU/WCU), Maximum capacity (e.g., 100 RCU/WCU), and Target utilization (e.g., 70%) to balance performance and cost.
  • Manual Option: Disable auto-scaling and set fixed RCUs/WCUs (e.g., 10 RCUs, 5 WCUs).
  • Console Setup:
  • Select Provisioned under Read/write capacity settings.
  • For this tutorial, disable auto-scaling and set 1 RCU and 1 WCU for simplicity (this keeps costs low, around $0.71/month as shown in the console).

Step 4: Additional Settings

  • Secondary Indexes: Skip for now (we’ll cover these in a future blogs). Indexes are used for alternative query patterns but aren’t needed for this basic setup.
  • Encryption: Enable Encryption at rest (default) to secure your data. You can choose AWS-managed or customer-managed keys.
  • Estimated Cost: With 1 RCU and 1 WCU, the console estimates ~$0.71/month. This gives you a sense of DynamoDB’s affordability.

Step 5: Creating the Table

Click Create table. In a few seconds, your DemoTable is ready! You’ll see it listed in the DynamoDB dashboard with details like status, capacity settings, and ARN.

Step 6: Adding Items to the Table

Now, let’s populate DemoTable with some data to see DynamoDB’s schema flexibility in action:

  1. View Items:
  • Click on DemoTable in the dashboard.
  • Go to the Items tab and click Create item.

2. Add First Item:

  • userID: Enter mino_123 (String).
  • Add attributes (key-value pairs):
  • name: Minoltan_Issack (String).
  • favorite_movie: Avengers (String).
  • favorite_number: 42 (Number).
  • Click Create item to save.

3. Add Second Item:

  • Click Create item again.
  • userID: Enter alice_456 (String).
  • Add attributes:
  • name: Alice Doe (String).
  • favorite_movie: Pocahontas (String).
  • age: 23 (Number).
  • Click Create item to save.

Key Observation: Notice that the two items (mino_123 and alice_456) have different attributes (favorite_number vs. age). DynamoDB’s NoSQL flexibility allows each item to have unique attributes without a predefined schema, unlike SQL databases where columns are fixed

Step 7: Exploring the Table

  • View Items: In the Items tab, you’ll see your two items listed:
  • mino_123: {name: “Minoltan Issack”, favorite_movie: “Avenger”, favorite_number: 42}
  • alice_456: {name: “Alice Doe”, favorite_movie: “Pocahontas”, age: 23}
  • Schema Flexibility: Attributes can be null or missing, and you can add new ones anytime without modifying the table structure.
  • Console Features: Explore tabs like Overview, Metrics, Alarms, and Additional settings to see options for monitoring, streams, backups, and more (we’ll cover these in future lectures).
