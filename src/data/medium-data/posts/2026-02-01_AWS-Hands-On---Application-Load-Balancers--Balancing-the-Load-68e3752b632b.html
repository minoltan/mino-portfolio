🌟 Introduction

Imagine you’ve built a popular website. At first, one server handles everything perfectly. But if traffic grows, that single server starts to sweat. You add a second server, but now you have a problem. How will the users get one single URL that can intelligently split traffic between both servers? Don’t worry, that’s where Application Load Balancer (ALB) comes in. It acts as a smart traffic cop, standing in front of your servers, checking their health, and distributing requests so no single instance gets overwhelmed.

Step 1. Launch Your Backend (The Targets)

First, we need something to load balance!

  • Launch two EC2 instances (Amazon Linux 2, t3.micro).
  • Security Group: Ensure they allow HTTP (Port 80) traffic.
  • Go to User Data: Use a script to install a web server (like Apache) that displays a “Hello World” message along with the instance ID. (Scroll down and go to advanced details.)
#!/bin/bash
# Use this for your user data (script from top to bottom)
# install httpd (Linux 2 version)
yum update -y
yum install -y httpd
systemctl start httpd
systemctl enable httpd
echo "<h1>Hello World from $(hostname -f)</h1>" > /var/www/html/index.html
  • Wait for few minutes for EC2 instance to be ready
  • Use http — http://47.128.235.224/

Step 2. Create the Security Group for the ALB

The load balancer needs its own firewall.

  • Create a new security group (e.g: demo-sg-alb).
  • Inbound Rule: Allow HTTP (Port 80) from Anywhere (0.0.0.0/0).

Step 3. Create a Target Group

The target group is a logical collection of your instances.

  • Go to Target Groups > Create target group.
  • Target group name: demo-tg-alb
  • Target type: Instances.
  • Protocol: HTTP on Port 80.
  • Register Targets: Select your two running EC2 instances and click Include as pending below.

Step 4. Configure the Application Load Balancer

  • Go to Load Balancers > Create Load Balancer > Application Load Balancer.
  • Load balancer name: DemoALB
  • Scheme: Internet-facing.
  • Network Mapping: Select at least two Availability Zones (to ensure high availability).
  • Security Groups: Select the one demo-sg-alb you created in Step 2.
  • Listeners and Routing: Forward traffic from HTTP:80 to your Target Group from Step 3.

The Power of Health Checks

Once your ALB status is Active, copy its DNS name and paste it into your browser. Refresh the page! You’ll notice the “Hello World” message toggles between Instance 1 and Instance 2.

  • The real magic happens when a server fails. If you manually stop Instance 1, the ALB’s health check will quickly mark it as unhealthy. The ALB will then stop sending traffic to the broken server and direct 100% of the load to Instance 2. When you start the server back up and it passes its health checks, the ALB automatically puts it back into the rotation.

Leveling Up: Advanced Security & Routing for Your ALB

So far in the hands-on, we got the traffic flowing. But in a production environment, simply flowing isn’t enough. You need to ensure your backend servers are hidden from the public eye and that your load balancer is smart enough to handle complex traffic patterns. In this section, we’ll tighten our security posture and explore the powerful world of Advanced Listener Rules.

1. Tightening the Screws: Security Group Chaining

Currently, your EC2 instances have public IPs, and their security groups likely allow HTTP traffic from anywhere (0.0.0.0/0). This is a security risk, users could bypass your load balancer and hit your servers directly. We want to ensure the only way into our instances is through the ALB.

Step-by-Step: Restricting Access

  • Navigate to EC2 > Security Groups.
  • Select the security group attached to your EC2 instances (e.g: launch-wizard-1).
  • Click Edit inbound rules.
  • Find the rule for Port 80 (HTTP) and delete the current source (0.0.0.0/0).
  • Add a new HTTP rule, but for the source, start typing the name or ID of your ALB’s security group. Select it from the list.
  • The Result: Try to visit your EC2 public IP in a browser. It will now time out. But visit your ALB’s DNS name, and it still works perfectly! You have successfully chained your security groups.

2. Mastering Traffic: Advanced Listener Rules

By default, an ALB just sends everything to one target group. But what if you want /api to go to one set of servers and /images to another? Or what if you want to block certain requests entirely? This is handled by listener rules.

The Components of a Rule

  • Priority: Rules are evaluated from lowest to highest number. The “Default Rule” is always last.
  • Conditions: The “if” statement (e.g: If the path is /error).
  • Actions: The “Then” statement (e.g: Then return a 404 error).

Step-by-Step: Creating a Custom Error Rule

  • Go to Load Balancers, select your ALB, and click the Listeners and rules tab.
  • Click on the HTTP:80 listener.
  • Click Add rule and give it a name (e.g: Demo-Error-Rule).
  • Add Condition: Select Path and enter /error.
  • Add Action: Select Return fixed response.
  • Response code: 404
  • Body: Not found, custom error.
  • Priority: Set this to 5 (or any number lower than the default).
  • The Result: Visit http://[ALB-DNS-NAME]/error. You will see your custom plain-text error message instead of the default "Hello World" page.

🧾 Conclusion

Setting up an Application Load Balancer is the first step toward building a truly resilient and scalable architecture. By decoupling your website’s URL from your actual server’s IP address, you gain the freedom to add, remove, or even break servers without your users ever noticing a glitch. It’s not just about spreading the traffic between servers, it’s about ensuring our application stays online without thinking about what happens in the background.
