Introduction

In this blog post, we’ll design a serverless architecture for MyBlog.com, a globally scalable website hosting static blog content with a dynamic REST API. The site handles hundreds of thousands of read-heavy users, with blogs added once a day or week. We’ll implement caching for cost and latency optimization, send welcome emails to new subscribers, and generate thumbnails for uploaded images — all in a serverless manner.

Requirements

  • Serve static content globally with low latency.
  • Support a public REST API for dynamic content.
  • Implement caching to optimize read-heavy workloads and reduce costs.
  • Send welcome emails to new subscribers.
  • Generate thumbnails for uploaded images.
  • Use a fully serverless architecture.

Prerequisites: AWS account, AWS CLI installed, basic IAM permissions.

Architecture Overview

We’ll leverage Amazon S3, Amazon CloudFront, Amazon API Gateway, AWS Lambda, Amazon DynamoDB, Amazon SES, and Amazon SQS/SNS to build a scalable, serverless solution. Each step includes hands-on instructions to implement the architecture.

Step 1: Serving Static Content Globally with S3 and CloudFront

Static blog content (HTML, CSS, JS) is stored in Amazon S3, and Amazon CloudFront, a global Content Delivery Network (CDN), distributes it to users via edge locations for low-latency access.

Hands-On:

  1. Go to S3 Console > Create Bucket: myblog-static. Enable static website hosting.
  2. Upload sample index.html:
<!DOCTYPE html>
<html>
<body>
  <h1>Welcome to MyBlog.com</h1>
</body>
</html>

3. Go to CloudFront Console > Create Distribution:

  • Origin: Select myblog-static S3 bucket.
  • Origin Access Control (OAC): Create OAC to restrict bucket access to CloudFront.
  • Default Cache Behavior: Enable caching (TTL: 86400 seconds).

4. Update S3 bucket policy to allow CloudFront:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {"Service": "cloudfront.amazonaws.com"},
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::myblog-static/*",
      "Condition": {"StringEquals": {"AWS:SourceArn": "arn:aws:cloudfront::<ACCOUNT-ID>:distribution/<CLOUDFRONT-ID>"}}
    }
  ]
}

5. CLI:

aws s3 mb s3://myblog-static
aws s3 website s3://myblog-static --index-document index.html
aws cloudfront create-distribution --origin-domain-name myblog-static.s3.amazonaws.com --origin-access-control-id <OAC-ID>

6. Test: Access CloudFront URL (e.g., d12345.cloudfront.net) in a browser. Verify direct S3 access is denied.

Step 2: Public Serverless REST API with API Gateway, Lambda, and DynamoDB

The REST API, hosted via Amazon API Gateway, invokes AWS Lambda to read blog metadata from Amazon DynamoDB. We’ll use DynamoDB Accelerator (DAX) for caching and DynamoDB Global Tables for low-latency global access.

Hands-On:

  1. Create DynamoDB Table: DynamoDB Console > Create Table: Blogs, Partition Key: blogId (String). Enable Global Tables (e.g., us-east-1, eu-west-1).
  2. CLI:
aws dynamodb create-table --table-name Blogs --attribute-definitions AttributeName=blogId,AttributeType=S --key-schema AttributeName=blogId,KeyType=HASH --billing-mode PAY_PER_REQUEST
aws dynamodb create-global-table --global-table-name Blogs --replication-group RegionName=us-east-1 RegionName=eu-west-1

3. Create DAX Cluster: DAX Console > Cluster Name: BlogsDAX, Node Type: cache.t3.small, Subnet Group: Default VPC.

4. CLI:

aws dax create-cluster --cluster-name BlogsDAX --node-type cache.t3.small --iam-role-arn <ROLE-ARN>

5. Create Lambda Function: Lambda Console > Name: BlogHandler, Runtime: Python 3.9.

import boto3
def lambda_handler(event, context):
    ddb = boto3.resource('dynamodb', endpoint_url='<DAX-ENDPOINT>')
    table = ddb.Table('Blogs')
    response = table.get_item(Key={'blogId': event['pathParameters']['blogId']})
    return {'statusCode': 200, 'body': json.dumps(response.get('Item', {}))}

6. Create API: API Gateway Console > REST API > Resource: /blogs/{blogId}, Method: GET, Integration: Lambda Proxy.

7. Enable caching (TTL: 300 seconds) on GET method.

8. CLI:

aws lambda create-function --function-name BlogHandler --runtime python3.9 --role <ROLE-ARN> --handler lambda.handler --zip-file fileb://function.zip
aws apigateway create-rest-api --name MyBlogAPI
aws apigateway create-deployment --rest-api-id <API-ID> --stage-name prod

9. Test: Use Postman to call GET <API-URL>/prod/blogs/blog1. Verify cache hits in CloudWatch.

Step 3: User Welcome Email Flow with DynamoDB Streams and SES

When a user subscribes, a DynamoDB Stream triggers a Lambda function to send a welcome email via Amazon SES.

Hands-On:

  1. Enable SES: SES Console > Verify domain/email (e.g., myblog.com). Exit sandbox mode for production.
  2. Create DynamoDB Table for Users: Table Name: Users, Partition Key: userId (String). Enable Streams.
  3. CLI:
aws dynamodb create-table --table-name Users 
--attribute-definitions AttributeName=userId,AttributeType=S 
--key-schema AttributeName=userId,KeyType=HASH 
--billing-mode PAY_PER_REQUEST 
--stream-specification StreamEnabled=true,StreamViewType=NEW_AND_OLD_IMAGES

4. Create Lambda Function: Name: SendWelcomeEmail, Runtime: Python 3.9.

import boto3
def lambda_handler(event, context):
    ses = boto3.client('ses')
    for record in event['Records']:
        if record['eventName'] == 'INSERT':
            user = record['dynamodb']['NewImage']
            ses.send_email(
                Source='no-reply@myblog.com',
                Destination={'ToAddresses': [user['email']['S']]},
                Message={
                    'Subject': {'Data': 'Welcome to MyBlog.com!'},
                    'Body': {'Text': {'Data': 'Hello, thanks for subscribing!'}}
                }
            )

5. Add Trigger: In Lambda Console, add DynamoDB Stream trigger for Users table.

6. IAM Role: Grant SES permissions:

{
  "Statement": [{"Effect": "Allow", "Action": "ses:SendEmail", "Resource": "*"}]
}

7. CLI:

aws lambda create-function --function-name SendWelcomeEmail 
--runtime python3.9 --role <ROLE-ARN> 
--handler lambda.handler 
--zip-file fileb://email.zip
aws lambda create-event-source-mapping 
--function-name SendWelcomeEmail 
--event-source-arn <DYNAMODB-STREAM-ARN>

8. Test: Insert user in DynamoDB (aws dynamodb put-item — table-name Users — item ‘{“userId”: {“S”: “user1”}, “email”: {“S”: “user@myblog.com”}}’). Check email inbox.

Step 4: Thumbnail Generation for Uploaded Images

When users upload images to S3 (via CloudFront or direct S3 with Transfer Acceleration), a Lambda function generates thumbnails and stores them in a separate S3 bucket.

Hands-On:

  1. Create S3 Buckets: myblog-images (uploads) and myblog-thumbnails (Thumbnails).
  2. Enable Transfer Acceleration on myblog-images for faster uploads (optional).
  3. CLI:
aws s3 mb s3://myblog-images
aws s3 mb s3://myblog-thumbnails
aws s3api put-bucket-accelerate-configuration 
--bucket myblog-images --accelerate-configuration Status=Enabled

4. Create Lambda Function: Name: GenerateThumbnail, Runtime: Python 3.9.

from PIL import Image
import boto3
def lambda_handler(event, context):
    s3 = boto3.client('s3')
    for record in event['Records']:
        src_bucket = record['s3']['bucket']['name']
        src_key = record['s3']['object']['key']
        dst_bucket = 'myblog-thumbnails'
        dst_key = f'thumb-{src_key}'
        with Image.open(s3.get_object(Bucket=src_bucket, Key=src_key)['Body']) as img:
            img.thumbnail((128, 128))
            img.save('/tmp/thumb.jpg')
            s3.upload_file('/tmp/thumb.jpg', dst_bucket, dst_key)

5. Add S3 Trigger: In Lambda Console, add S3 trigger for myblog-images (Event: s3:ObjectCreated:*).

6. IAM Role: Grant S3 read/write:

{
  "Statement": [
    {"Effect": "Allow", "Action": "s3:GetObject", "Resource": "arn:aws:s3:::myblog-images/*"},
    {"Effect": "Allow", "Action": "s3:PutObject", "Resource": "arn:aws:s3:::myblog-thumbnails/*"}
  ]
}

7. CLI:

aws lambda create-function 
--function-name GenerateThumbnail 
--runtime python3.9 
--role <ROLE-ARN> 
--handler lambda.handler 
--zip-file fileb://thumbnail.zip
aws s3api put-bucket-notification-configuration 
--bucket myblog-images 
--notification-configuration '{"LambdaFunctionConfigurations": [{"LambdaFunctionArn": "<LAMBDA-ARN>", "Events": ["s3:ObjectCreated:*"]}]}'

8. Test: Upload image to myblog-images (aws s3 cp image.jpg s3://myblog-images/), verify thumbnail in myblog-thumbnails.

Final Architecture

The complete architecture:

  • Clients: Access static content via CloudFront and dynamic API via API Gateway.
  • Amazon S3: Stores static content and images. Triggers Lambda for thumbnails.
  • Amazon CloudFront: Distributes static content globally with OAC for security.
  • Amazon API Gateway: Hosts public REST API, invokes Lambda, caches responses.
  • AWS Lambda: Handles API logic and thumbnail generation.
  • Amazon DynamoDB: Stores blog metadata with Global Tables and DAX for caching.
  • Amazon SES: Sends welcome emails triggered by DynamoDB Streams.
  • Amazon SQS/SNS: Optional for additional S3 event handling.

This is fully serverless, globally scalable, and pay-per-use.

Conclusion

MyBlog.com showcases a powerful serverless architecture using Amazon S3, CloudFront, API Gateway, Lambda, DynamoDB, and SES. With DAX, Global Tables, and caching, we optimized for global scale and read-heavy traffic. The hands-on steps enable you to deploy this in under an hour.
