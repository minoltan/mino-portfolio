Introduction

Security groups are the simple yet powerful firewalls that protect every EC2 instance, and getting comfortable editing them is one of the most useful skills you’ll learn on AWS. In this quick hands-on guide, we’ll jump into the AWS console, explore the security groups attached to the instance you just launched, add and delete rules in real time, see exactly why a missing rule causes a frustrating timeout (and how to fix it instantly), and learn pro tips for inbound/outbound rules, multiple security groups, and staying secure. Follow along — it takes less than 10 minutes and you’ll never be stuck staring at a blank browser again!

Step 1: Find Your Security Groups

  1. Open the EC2 Dashboard (search “EC2” in the AWS console).
  2. In the left menu, under Network & Security, click Security groups.
  3. You’ll see at least two: • default — automatically created with every VPC • launch-wizard-X — the one created when you launched your first instance

Each security group has a unique ID (like sg-0123456789abcdef0) and a description so you can tell them apart.

Step 2: Check the Rules on Your Instance

  1. Go back to Instances, select your “My First Instance” (checkbox).
  2. Scroll down to the Security tab. You’ll see your launch-wizard security group listed with its inbound and outbound rules summary.

This is the fast way to check what’s attached — perfect for quick troubleshooting.

Step 3: Explore & Edit Inbound Rules (The Ones That Let Traffic In)

  1. Back in Security groups, click the launch-wizard group name.
  2. Click the Inbound rules tab → Edit inbound rules.

You should see:

  • SSH (port 22) — Source: Anywhere-IPv4 (0.0.0.0/0)
  • HTTP (port 80) — Source: Anywhere-IPv4 (0.0.0.0/0)

The HTTP rule is why your “Hello World” webpage worked!

Step 4: See What Happens When You Delete the HTTP Rule

  1. Check the box next to HTTP → Remove → Save rules.
  2. Go to your browser tab with http://your-public-ip
  3. Refresh → You’ll get an endless loading spinner (timeout!).

Key Lesson: Timeout = Security Group Problem When the browser keeps trying but never gets a response, traffic is being blocked before it reaches your instance. 100% of the time, check your security group first!

Step 5: Add the Rule Back (and Try a Few More)

  1. In Inbound rules, click Add rule
  • Type: HTTP (automatically fills port 80)
  • Source: Anywhere-IPv4 (0.0.0.0/0)
  • Description (optional): “Public web access”

2. Click Save rules

3. Refresh your browser → Hello World is back instantly!

Bonus Experiments

  • Add HTTPS (port 443) the same way — great for real websites later.
  • Change Source to My IP — only your computer can connect (safer for testing, but remember your IP can change!).
  • Try a custom port (e.g., Type: Custom TCP, Port range: 8080) if you ever run apps on non-standard ports.

Step 6: Quick Look at Outbound Rules

Click the Outbound rules tab. You’ll see one rule: All traffic → Destination: Anywhere (0.0.0.0/0).

This lets your instance download updates, install packages, and reach the internet. You usually leave this open, but in high-security setups you can lock it down to only what’s needed.

Step 7: Multiple Security Groups — They Stack Like Lego!

You can attach up to 5 security groups to one instance (rules combine). Example:

  • One group for SSH (locked to your IP)
  • One group for web traffic (HTTP/HTTPS from anywhere)
  • One group for database access (from your app servers only)

One security group can also protect dozens of instances — perfect for web fleets or microservices.

Quick Best-Practice Checklist

  • Never leave SSH (port 22) open to Anywhere in production
  • Use clear descriptions (“Public HTTP”, “My laptop SSH”)
  • Name security groups meaningfully (e.g., web-public-sg, ssh-personal-sg)
  • Fix timeout errors → always check security groups first!
  • Stop your instance when you’re done to save Free Tier hours

Conclusion

You’ve just gone from beginner to confident: you can now view, edit, and troubleshoot security groups like a pro. That timeout nightmare? Gone forever. You also understand why your web server works (or doesn’t!) and how to safely open exactly the ports you need. Next time you launch anything on AWS, you’ll set up security groups perfectly on the first try. Keep your instance running for now — we’ll use it in the next hands-on to SSH in and run commands directly. You’re doing awesome — see you there! 🚀
