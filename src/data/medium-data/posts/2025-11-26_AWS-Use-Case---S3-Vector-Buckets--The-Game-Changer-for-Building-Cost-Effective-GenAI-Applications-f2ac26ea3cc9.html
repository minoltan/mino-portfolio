What is Amazon S3 Vectors?

Amazon S3 Vectors is a feature that allows you to store and search vector data directly inside Amazon S3. instead of moving data to a separate database, you can manage vectors natively alongside your other S3 content.

Key Benefits

  • Massive Cost Savings: It is designed to reduce the cost of uploading, storing, and querying vectors by up to 90%.
  • Built for AI: It is optimized for AI agents, inference, and performing semantic searches on your data.
  • High Performance at Scale: You can store billions of vectors and search through them with sub-second speed, utilizing the same proven durability and scale as standard Amazon S3.

Part 1: Why Do We Need Vector Databases in GenAI Applications?

A. The Problem with Pure Foundational Models

Models like Claude, GPT-4o, Gemini, or Llama are trained on internet-scale data. Ask them “How do you make Any Berry Sauce?” and they’ll give you a perfectly fine generic answer.

But what if you own a restaurant with a secret Any Berry Sauce recipe with extra butter, a dash of truffle oil, and a special technique? You don’t want your chatbot giving the generic internet version. You want it to answer using your proprietary data.

This is the core promise of Retrieval-Augmented Generation (RAG): LLM + Your Private Data = Accurate, Hallucination-Free Answers

The Two Types of Private Data

  1. Structured data → Databases (DynamoDB, RDS, etc.) → Queryable with SQL or APIs (e.g., via Lambda tools in Bedrock)
  2. Unstructured data → PDFs, Word docs, images, videos, Notion pages, etc. → Impossible to query with traditional methods

Most company knowledge (recipes, product manuals, policies, support docs) lives in unstructured formats. That’s where vector databases become mandatory.

B. How Semantic Search Actually Works

You can’t run SELECT * FROM recipes WHERE topic = ‘any berry sauce’ on a PDF.

Instead:

  1. Chunk the document into meaningful pieces
  2. Convert each chunk into a vector (embedding) — a long array of numbers that mathematically represents meaning
  3. Store those vectors in a vector database
  4. At query time: convert the user question into a vector → find nearest neighbor vectors → retrieve original text chunks → feed context to the LLM

This “similarity search” is what powers accurate, context-aware GenAI apps.

C. Traditional Vector Database Options (Before S3 Vectors)

  • OpenSearch Serverless + k-NN plugin
  • Amazon RDS for PostgreSQL + pgvector
  • Pinecone, Weaviate, Qdrant, etc.

They all work great… but they come with operational overhead and cost, especially at low to medium scale.

Then AWS said: “What if the world’s most durable, cheapest, and most scalable storage service… just became a vector database?”

Enter Amazon S3 Vector Buckets (currently in public preview).

Part 2: Hands-On Demo — Build a Recipe Chatbot Using S3 Vectors + Bedrock

Let’s implement the restaurant chatbot we talked about.

Step 1: Create a Regular S3 Bucket for Source Documents

This is where your raw PDFs live (not vectors). We created: knowledge-base-search-data-source-bucket

Step 2: Create a Bedrock Knowledge Base Using S3 Vectors

In Amazon Bedrock → Knowledge bases → Create knowledge base

Key settings:

  • Data source: Amazon S3 (point to the regular source bucket)
  • Embedding model: Amazon Titan Text Embeddings v2 (1024 dimensions)
  • Vector store: Choose Existing vector store → S3 Vectors → select your vector bucket + index

Hit “Create” → then upload your cookbook PDF to the source bucket → click Sync

Behind the scenes:

  • Bedrock chunks the PDF (default: ~300 tokens per chunk with overlap)
  • Converts each chunk to a 1024-dimensional vector using Titan
  • Stores vectors + metadata in your S3 Vector Bucket

Sync completed in seconds for a 5 MB PDF.

Step 3: Create a Bedrock Agent

Bedrock → Agents → Create agent

Settings:

  • Name: Restaurant-ChefBot
  • Model: Claude 3.5 Sonnet
  • Instructions (critical!):
You are a friendly chef bot and AI cooking assistant. You have access to a knowledge base of proprietary restaurant recipes. Always consult the knowledge base first when answering cooking-related questions. Only use information returned from the knowledge base. If the answer isn’t there, say you don’t know that specific restaurant recipe, but you can share a general version from public knowledge (and mention the source). Maintain a warm, conversational tone.
  • save it before adding knowledge base

Step 4: Connect the Knowledge Base

  • Instruction
Internal knowledge base that includes information about cooking recipes, storage tips,information about cooking tools
  • save it again
  • click on prepare

Step 5: Test It Live!

Queries we tried:

  • “How do you make Any Berry Sauce?” → Returned exact recipe including 4 cups berries + 1 tbsp cornstarch
  • Follow-up: “How much cornstarch?” → Agent remembered context and answered correctly

You can even view the trace to see exactly which chunks were retrieved from the S3 Vector Bucket.

It just works — with sub-second latency and zero infrastructure to manage.

Why S3 Vector Buckets Are a Big Deal

Conclusion

Amazon S3 Vector Buckets mark a turning point in making production-grade RAG accessible to every AWS developer — no PhD in vector databases required.

With this release you can now:

  • Store proprietary documents in regular S3
  • Turn them into a searchable vector database automatically via Bedrock Knowledge Bases
  • Query them with sub-second latency and extreme durability
  • Pay pennies while doing it

The future of enterprise GenAI is no longer locked behind expensive third-party vector DBs. It now lives in the same S3 bucket you’ve been using for a decade — only now it understands meaning.
