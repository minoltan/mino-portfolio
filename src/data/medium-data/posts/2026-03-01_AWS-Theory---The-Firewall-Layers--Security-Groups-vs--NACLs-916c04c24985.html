🌟 Introduction

In AWS, security isn’t just a single locked door. It’s a series of checkpoints. When a request tries to reach your EC2 instance, it must pass through two distinct types of firewalls: Network Access Control Lists (NACLs) at the subnet level and Security Groups (SGs) at the instance level. Understanding how these two work together is the key to mastering VPC security and passing the AWS exams.

1. The Two-Layer Defense System

Imagine your VPC as a gated apartment complex.

  • The NACL is the security guard at the main gate of the complex (the Subnet). They check everyone coming in and out of the entire neighborhood.
  • The Security Group is the deadbolt on your specific front door (the Instance). Even if the guard at the main gate lets someone in, your personal door lock still has its own rules.

2. Statefulness: The Biggest Difference

This is the concept that trips up most beginners.

  • Security Groups are Stateful: If you allow traffic in (inbound), the return traffic is automatically allowed out (outbound) because the security group remembers the connection. You don’t need to write a rule for the response.
  • NACLs are stateless: NACLs have no memory. If you allow traffic in on port 80, you must explicitly write a rule to allow the response to go back out. If you forget the outbound rule, the connection will time out.

3. Rule Evaluation: Priority Matters

  • NACLs use rule numbers: Rules are evaluated in order from the lowest number to the highest (e.g., 100, 200, 300). As soon as a packet matches a rule, the evaluation stops. This is why it’s best practice to leave gaps (increments of 100) between rules so you can insert new ones later. Newly created NACLs will deny everything. NACL are a great way of blocking a specific IP address at the subnet level
  • Security Groups evaluate ALL rules: Unlike NACLs, security groups look at all your rules before deciding to allow the traffic. There is no priority numbering.

4. NACLs & Default NACLs

Default NACL

  • Accepts everything inbound/outbound with the subnets it’s associated with
  • Do NOT modify the Default NACL, instead create custom NACLs

5. Ephemeral Ports: The Response Path

When a client connects to a server, the server responds on a random, temporary port called an Ephemeral Port.

  • For any two endpoints to establish a connection, they must use ports
  • Clients connect to a defined port, and expect a response on an ephemeral port
  • Different Operating Systems use different port ranges.
  • IANA & MS Windows 10 -> 49152–655
  • Many Linux Kernels -> 32768–60999
  • Because NACLs are stateless, you must allow these high-numbered ports (usually 1024–65535) in your outbound rules.
  • If your NACL allows inbound traffic on Port 80 but denies outbound traffic on high ports, your web server will never be able to send the website data back to the user!

Pro Tip: Use NACLs for “broad strokes” (like blocking a specific malicious IP address) and use Security Groups for “fine-tuned” access (like allowing only your web server to talk to your database).

🧾 Conclusion

Security in AWS is a "defense in depth" strategy. While the default NACL is wide open, you can harden your infrastructure by layering restrictive Security Groups on your instances. By mastering the interaction between stateful SGs and stateless NACLs, you ensure your data is accessible to those who need it — and invisible to those who don’t.
