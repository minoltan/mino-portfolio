Introduction

In AWS, IAM policies are the single most important security control you will ever touch. A tiny mistake can grant full admin access to the wrong person (or worse, the internet), while a well-crafted policy keeps your account safe, compliant, and cost-controlled. This hands-on lab is not just another tutorial — it is the fastest and most effective way to truly feel how IAM policies work in real life. By deliberately removing your own permissions and watching “Access Denied” appear, then attaching read-only access, exploring real JSON policies, and even building your own custom policy with the visual editor, you will move from theory to confidence in minutes. Whether you’re preparing for the SAA-C03 exam, starting your first AWS project, or responsible for your company’s cloud security, completing this lab will give you the practical mastery that reading alone can never provide. Let’s jump in — your secure AWS journey starts right here!

Prerequisites

  • AWS account
  • IAM user minoltan in admin group (from previous lab)
  • Two browser windows:
  • Left: Root user
  • Right: IAM user minoltan

Step 1: Remove User from Admin Group (Lose All Access)

Let’s see what happens when you lose permissions.

  1. In left window (root) → IAM Dashboard
  2. Go to User groups → click admin
  3. Check mino → Remove
  4. Confirm removal

Now switch to right window (mino)

  1. Go to Users
  2. Try to view list → Access Denied
Error: You are not authorized to perform: iam:ListUsers

You just lost all access!
Because mino is no longer in admin group.

Step 2: Attach IAMReadOnlyAccess Directly to User

Let’s give read-only access back.

  1. In left window (root) → Users → click mino
  2. Click Add permissions → Attach policies directly
  3. Search: IAMReadOnlyAccess
  4. Check box → Next → Add permissions

Test in right window

  1. Refresh Users page
  2. You can now see users and groups

3. Try to create a group → Access Denied

Success! You have read-only access to IAM.

Step 3: Create a Developers Group (via Root)

  1. In left window → User groups → Create group
  2. Name: developers
  3. Attach any policy (e.g., CloudFrontFullAccess)
  4. Add user: mino
  5. Click Create group

Step 4: Re-add User to Admin Group

  1. Go to admin group → Add users
  2. Check mino → Add

Step 5: View All Permissions on Mino

  1. In left window → Users → mino→ Permissions tab

Step 6: Explore Managed Policies (JSON View)

Let’s see how AWS defines policies.

1. View AdministratorAccess

  1. IAM Dashboard → Policies
  2. Search: AdministratorAccess → click
  3. JSON tab
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "*",
      "Resource": "*"
    }
  ]
}* = anything → full access to all services

2. View IAMReadOnlyAccess

  1. Search: IAMReadOnlyAccess → JSON tab
{
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "iam:Get*",
        "iam:List*",
        "iam:GenerateCredentialReport"
      ],
      "Resource": "*"
    }
  ]
}
Get* = all Get actions (e.g., GetUser, GetGroup)
List* = all List actions

Step 7: Create Your Own Policy (Visual Editor)

Let’s build a custom policy.

  1. Policies → Create policy
  2. Choose Visual editor
  3. Service: IAM

4. Actions:

  • Check ListUsers (under List)
  • Check GetUser (under Read)

5. Resources: All resources

6. Click Next

7. Name: MyIAMReadPolicy

8. Click Create policy

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["iam:ListUsers", "iam:GetUser"],
      "Resource": "*"
    }
  ]
}
You just built a custom policy in 30 seconds!

Step 8: Clean Up

Remove unnecessary permissions.

  1. User groups → developers → Delete

2. Users → mino → Permissions

3. Remove IAMReadOnlyAccess (direct)

4. Confirm mino only has AdministratorAccess & IAMUserChangedPassword via admin group

Conclusion

Mastering IAM policy control isn’t just another checkbox on your AWS journey — it’s the difference between a secure, auditable, cost-controlled environment and one accidental click away from disaster. Every resource you launch, every dollar you spend, and every piece of data you store ultimately depends on who can do what. Strong IAM discipline enforces least privilege, prevents privilege creep, satisfies compliance requirements, and protects your organization from both insider mistakes and external attacks. In the cloud, security isn’t an afterthought — it’s built through deliberate, repeatable policy decisions. You now hold the keys to that process. Use them wisely, review them regularly, and never settle for “good enough.” Great cloud architecture begins and ends with great IAM governance. You’ve earned the right to own it — now go make security your superpower. 🚀
