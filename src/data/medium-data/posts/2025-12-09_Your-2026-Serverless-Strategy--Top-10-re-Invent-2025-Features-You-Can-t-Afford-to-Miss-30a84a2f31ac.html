1. CloudFront introduces flat rate price plan

Problem

Pay-per-use pricing, unlike fixed pricing, means you only pay for the services you actually consume, which eliminates wasted spending. However, this model can lead to surprisingly high bills if you experience an unexpected event like a massive surge in website traffic or a hacking attack (DDoS), which is a common and frustrating problem for many using services like AWS.

Solution

Enter Amazon Web Services’ latest innovation: flat-rate pricing plans for Amazon CloudFront, designed to bundle essential CDN and security services into predictable, no-overage monthly fees.

Key Features: What’s Included in the Bundle?

Flat-rate pricing plans include the following features for a simple monthly price:

  • CloudFront CDN
  • AWS WAF and DDoS protection
  • Bot management and analytics
  • Amazon Route 53 DNS
  • Amazon CloudWatch Logs ingestion
  • Serverless edge compute
  • Monthly Amazon S3 storage credits

Pricing Breakdown and Limits

Each tier is a monthly subscription that auto-renews until you say stop. No overages mean you won’t get dinged for exceeding caps — instead, performance might throttle gently after hitting limits, with email alerts at 50%, 80%, and 100% usage. Monitor everything in the console for peace of mind.

2. API Gateway supports response streaming

Problem

The main issue with traditional APIs, especially for interactive tools like chatbots or when dealing with large files, is that the system must wait for the entire response to be fully created by the server before it can send even the first piece of data to the user. This delay, called high Time-To-First-Byte (TTFB), makes applications feel slow and unresponsive, and it also creates problems for big responses that hit the default 10 MB size limit or the 29-second timeout of the API service.

Solution

The solution is to use Amazon API Gateway response streaming, which changes the way data is sent by immediately streaming the response back to the user piece-by-piece as soon as it is generated, instead of waiting for the whole thing. This new capability greatly improves how fast the application feels (like seeing text appear instantly in a chat) and completely solves the size limit problem, allowing developers to handle very large responses or tasks that can run for up to 15 minutes.

3. Lambda doesn’t need NAT Gateway anymore!

Problem

The core problem in modern networking is the depletion of available IPv4 addresses as the internet continues to grow, which is leading to resource scarcity. Furthermore, when Lambda functions inside a Virtual Private Cloud (VPC) needed to talk to the public internet, they were forced to use an IPv4-only network setup that required an expensive NAT Gateway, adding extra cost and complexity to the network design.

Solution

The solution is to enable IPv6 networking (dual-stack) for AWS Lambda functions. This change addresses the address shortage and, more importantly, allows the Lambda functions to use a free egress-only internet gateway instead of the costly NAT Gateway when they need internet access from a private subnet. This switch to IPv6 helps reduce operational costs, simplifies the network architecture, and future-proofs applications for the evolving internet.

Note

For Lambda functions that interact exclusively with AWS services through internal traffic, IPv6 offers limited benefits. For example, in an architecture where a Lambda function processes requests from Amazon API Gateway and queries a database hosted on Amazon Relational Database Service (Amazon RDS), no architectural change is expected. 4. Lambda ups max payload size for async invocations to 1MB

Problem

Previously, when sending event data to an AWS Lambda function for asynchronous processing — like sending logs, telemetry, or complex data structures from services like Amazon S3 or EventBridge — the maximum size of that data (the payload) was limited to only 256 KB. This small limit forced developers working with larger pieces of data, such as detailed user profiles or machine learning outputs, to use complex workarounds like splitting the data into smaller chunks or storing the data externally and just sending a reference link, which added unnecessary complexity to their application architecture.

Solution

AWS Lambda has resolved this by increasing the maximum payload size for asynchronous invocations from 256 KB to 1 MB. This four-fold increase allows developers to send much richer and more complex event data in a single go, eliminating the need to split or externalize data. Now, a single event can contain comprehensive information, which significantly simplifies the design of modern event-driven, serverless applications.

5. Lambda Managed Instances

Problem

The main issue was that some advanced workloads needed specialized compute resources (like powerful new CPU types) or required the significant cost savings available through long-term EC2 commitment plans (like Savings Plans). Because standard AWS Lambda couldn’t access these specific EC2 features or pricing, customers were forced to abandon the simple, serverless approach and manually manage their own EC2 virtual servers, which introduced a heavy, unwanted burden of dealing with patching, scaling, and infrastructure maintenance.

Solution

AWS solved this by introducing Lambda Managed Instances, which allow developers to select specific Amazon EC2 instance types and apply commitment-based pricing for cost optimization. The best part is that AWS handles all the operational complexity: setting up the instances, OS patching, load balancing, and auto-scaling are all managed automatically. This means users get the full flexibility and power of EC2 for their workloads while keeping the simplicity and low operational overhead of the serverless Lambda experience.

6. Lambda Durable Functions

Problem

When building complex, multi-step applications like payment processes, AI workflows, or human approval chains, developers faced a significant challenge because standard Lambda functions are short-lived and don’t remember their state. To manage these long-running tasks, they had to write a lot of extra code to save the workflow’s progress, handle retries on failure, and figure out how to pause the function for long periods without continuously paying for idle compute time while waiting for an external event.

Solution

AWS introduced Lambda durable functions, which allows you to write complex, sequential workflow code directly inside a standard Lambda function. Using a new Durable Execution SDK, the function automatically manages its own state, adds checkpoints to gracefully recover from failures with automatic retries, and can suspend execution for up to one year when waiting for external events (like an approval) without incurring any charges for idle compute time.

7. S3 Tables support intelligent-tiering and replication

Problem

Data teams struggled with two main manual burdens: managing the rising cost of storage because data they rarely used was sitting in expensive tiers, and the difficulty of setting up and constantly maintaining consistent copies of their data tables (like Apache Iceberg tables) across different geographic locations for disaster recovery or global access, which required complex, error-prone architectures.

Solution

The solution introduces two new capabilities: Intelligent-Tiering automatically monitors how often the table data is used and moves it to the most cost-effective storage tier — saving money without any manual work. It also provides Replication Support, which fully automates the process of creating and maintaining up-to-date, consistent copies of the tables across different AWS Regions and accounts, making global data management and disaster recovery much simpler.

8. Lambda supports native tenant isolation

Problem

When building multi-tenant applications like Software-as-a-Service (SaaS) platforms, developers had to ensure strict isolation between different customers (tenants) to prevent data leaks. Traditionally, they achieved this by either deploying a completely separate Lambda function for every single tenant, which quickly became a huge management headache, or by building their own complex, custom isolation logic into a shared function, which was difficult to secure and maintain.

Solution

AWS introduced a new tenant isolation mode directly into AWS Lambda. By simply enabling this mode and providing a tenant identifier in the invocation request, Lambda automatically uses a separate execution environment for each individual tenant, even if they are using the same function code. This built-in feature ensures that tenants’ data and cached variables are strictly isolated from one another, providing the required security and separation without any extra development or operational complexity.

9.Lambda adds provisioned mode for SQS ESM

Problem

When using AWS Lambda to process messages from an Amazon SQS queue, applications with very high traffic or sudden spikes faced limitations because the standard method of scaling up the Lambda polling resources was too slow. This slow scaling and a lower maximum capacity meant that during peak periods, customers couldn’t process events quickly enough, which caused message backlogs in the queue and led to unpredictable or unacceptably high processing latency for mission-critical, event-driven applications.

Solution

The solution is a new provisioned mode for the Lambda-SQS Event Source Mapping (ESM), which provides dedicated event pollers that remain ready to process messages. This new mode enables resources to scale three times faster and offers a 16 times higher maximum concurrency than before, with support for up to 20,000 concurrent requests. By allowing customers to set a guaranteed minimum capacity, this feature ensures predictable, low-latency performance and effectively handles sudden, high-volume traffic spikes.

API Gateway integration with ALB (without NLB)

Problem

Before this update, connecting an external-facing Amazon API Gateway REST API to an internal, private Application Load Balancer (ALB) required an extra, intermediary Network Load Balancer (NLB) to bridge the connection. This extra component increased the application’s complexity, cost, and network latency, forcing developers to manage, configure, and pay for an unnecessary piece of infrastructure just to connect two core AWS services within their private network.

Solution

The solution is to support direct private integration between API Gateway REST APIs and private ALBs using VPC Link v2. This enhancement removes the need for the intermediary NLB, which instantly simplifies the architecture, reduces operational costs (by eliminating NLB hourly charges), and improves performance by reducing network hops and latency. Additionally, a single VPC Link can now connect to multiple ALBs, making it easier to scale complex microservices architectures.
