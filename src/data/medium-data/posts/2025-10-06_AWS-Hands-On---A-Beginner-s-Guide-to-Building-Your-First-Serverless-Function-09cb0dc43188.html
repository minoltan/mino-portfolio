Introduction

Hey there, cloud explorers! Ready to roll up your sleeves and dive into AWS Lambda? In this post, we’re walking through a practical, step-by-step intro to creating and testing a Lambda function using the AWS Console. Think of this as your guided tour to serverless computing — complete with code, triggers, and monitoring tips. Whether you’re studying for an AWS certification or just curious about serverless, this hands-on guide will get you comfortable with Lambda’s basics. Let’s fire up the console and get started!

Why Practice with Lambda?

AWS Lambda is the cornerstone of serverless, letting you run code without managing servers. It’s event-driven, auto-scaling, and cost-efficient (thanks to a generous free tier). In this tutorial, we’ll:

  • Create a simple “Hello World” Lambda function.
  • Explore how it responds to events.
  • Test and debug it using the AWS Console.
  • Peek at configurations, permissions, and monitoring.

No coding expertise required — just follow along, and you’ll see Lambda’s magic in action!

Step 1: Navigating the Lambda Console

Hop into the AWS Lambda Console (search for “Lambda” in the AWS Management Console). If you see a welcome screen, check out the /begin URL for a slick UI demo. It’s a great way to visualize how Lambda works:

  • Function Overview: Lambda supports multiple languages like Node.js, Python, Java, .NET, Ruby, or even custom runtimes (e.g., Go or Rust).
  • Event-Driven Magic: The demo shows a Lambda function responding to events like streaming analytics, mobile app data, or S3 uploads. Click around, and you’ll see Lambda scale seamlessly — cogs multiply as events pile up, proving its auto-scaling prowess.
  • Cost Insight: The free tier covers 1 million requests and 400,000 GB-seconds monthly. More invocations? Costs scale, but you can estimate based on your workload.

This interactive UI is a fun way to grasp Lambda’s event-driven, scalable nature before diving into code.

Step 2: Creating Your First Lambda Function

Let’s build a “Hello World” function from scratch:

  1. Start in the Console: Click Create a Function.
  2. Choose a Blueprint: Select the “hello-world” blueprint for simplicity. Pick Python as the runtime (any version works; we’ll stick with the default).
  3. Name Your Function: Call it HelloWorld for clarity.
  4. Set Up Permissions:
  • Lambda needs an execution role (like an IAM role for EC2, but for serverless).
  • Choose Create a new role with basic Lambda permissions. This auto-generates a role with CloudWatch Logs access.

5. Create the Function: Hit Create Function, and boom — your function is ready!

The console loads your new function’s dashboard, complete with a code editor showing the default Python code.

Step 3: Understanding the Code

The blueprint gives you a simple Python function (lambda_function.py):

  • Handler: The entry point (lambda_handler) is called when the function is invoked. It takes an event (input data as JSON) and context (runtime info).
  • Logic: The default code expects a JSON event with keys like key1, key2, key3. It returns key1’s value (e.g., value1).

Here’s a peek at the code structure (simplified):

import json

print('Loading function')


def lambda_handler(event, context):
    #print("Received event: " + json.dumps(event, indent=2))
    print("value1 = " + event['key1'])
    print("value2 = " + event['key2'])
    print("value3 = " + event['key3'])
    return event['key1']  # Echo back the first key value
    #raise Exception('Something went wrong')

No need to be a Python pro — this just shows how Lambda processes input and returns output.

Step 4: Testing Your Function

Let’s see it in action:

  1. Create a Test Event:
  • Click Test in the console.
  • Use the default hello-world template, which provides a JSON input like:
{
  "key1": "value1",
  "key2": "value2",
  "key3": "value3"
}
  • Name the test event (e.g., HelloWorldEvent) and save it.

2. Run the Test: Click Test again. The function executes, and you’ll see:

  • Success: Output like {"statusCode": 200, "body": "value1"}.
  • Logs: Execution details in CloudWatch Logs, showing the input and result.

3. Trigger a Failure (Optional):

  • Edit the test JSON to remove key1 (e.g., delete “key1”: “value1”).
  • Run the test again. It fails because the code expects key1, demonstrating how errors surface.
  • Restore the original JSON and retest — it’s back to success!

You can save the test event to reuse it anytime, making iterative testing a breeze.

Step 5: Monitoring and Debugging

Lambda’s integration with Amazon CloudWatch makes monitoring easy:

  • Invocations and Metrics: The Monitoring tab shows stats like invocation count and duration. It takes a bit to populate, so give it a few runs.
  • CloudWatch Logs: Click View CloudWatch Logs to see detailed logs:
  • Success logs show the input (e.g., value1 = one) and execution details.
  • Error logs (like our failed test) pinpoint issues for debugging.
  • Log Streams: Each function run creates a log stream, perfect for tracking issues in real time.

Step 6: Configuring Your Function

The Configuration tab is where you fine-tune your function:

General Settings:

  • Memory: Adjust from 128 MB to 10 GB. More RAM boosts CPU and network performance.
  • Timeout: Set a max runtime (up to 15 minutes) to avoid runaway functions.
  • Ephemeral Storage: Allocate temporary storage for processing.

Permissions

  • The execution role (e.g., AWSLambdaBasicExecutionRole) allows writing to CloudWatch Logs.
  • Need to access S3 or DynamoDB? Edit the IAM role to add permissions (e.g., s3:PutObject).

Triggers:

  • No triggers yet, but you can add one (e.g., S3 for file uploads).
  • The console lists AWS services (S3, SNS, SQS) and partner events as trigger options. For S3, you’d pick a bucket and event type (e.g., ObjectCreated).

Step 7: Exploring Triggers (The Serverless Superpower)

Triggers are what make Lambda event-driven. In the console, the Triggers tab shows a menu of possibilities:

  • Amazon S3: React to file uploads (e.g., create thumbnails).
  • API Gateway: Build REST APIs that invoke Lambda.
  • CloudWatch Events/EventBridge: Schedule tasks or react to AWS events.
  • SNS/SQS: Process notifications or queue messages.

For now, we’re sticking with manual tests, but triggers are where Lambda shines in production.

Step 8: Managing Concurrency Settings

Concurrency is key to Lambda’s scalability, but it needs careful management to avoid throttling or performance hiccups. Let’s explore how to configure it in the Lambda Console:

  1. Access Concurrency Settings:
  • In the Configuration tab, find the Concurrency sub-tab on the left.
  • By default, your function uses the unreserved account concurrency (1,000 concurrent executions, shared across all Lambda functions in your account per region).
Note : The standard default maximum is still 1,000, but AWS implements the lower initial limit to prevent new users from incurring massive, unexpected costs due to runaway or misconfigured functions. You can easily raise the limit to 1,000 (or higher) by submitting a quota increase request via the AWS Service Quotas console.

2. Setting Reserved Concurrency:

  • Edit the concurrency settings and set a reserved concurrency limit (e.g., 20) for your function.
  • Impact: This reserves 20 concurrent executions for HelloWorld, leaving 980 for other functions in your account.
  • Why Do This? It prevents one function from hogging the account-wide limit, ensuring fair resource allocation.
  • Testing Throttling: For fun, set reserved concurrency to 0. Run a test — you’ll get a 429 Too Many Requests error (“Invoke API action failed due to exceeding the rate”). This simulates throttling, helping you understand how apps behave under limits.
  • Fixing It: Revert to unreserved concurrency or set a reasonable limit (e.g., 20). Retest, and your function should execute successfully.

3. Provisioned Concurrency for Cold Starts:

  • What’s a Cold Start? When a new Lambda instance starts, it loads code and initializes (e.g., sets up dependencies or database connections), causing a slight delay for the first request.
  • Solution: Use provisioned concurrency to keep a “warm pool” of pre-initialized instances, reducing latency.
  • How to Set It:
  • In the Concurrency tab, add a Provisioned Concurrency Configuration.
  • Specify a number (e.g., 5 instances) for a specific version or alias of your function. Note: You can’t apply it to the $LATEST version — you must publish a version first (covered in future lectures).
  • Cost Alert: Provisioned concurrency isn’t free; it incurs charges based on the number of warm instances. Choose a number that balances cost and performance.
  • Why It Matters: Eliminates cold start delays, critical for latency-sensitive apps (e.g., user-facing APIs).

Exam Tip: Know the difference between reserved concurrency (caps function scaling) and provisioned concurrency (reduces cold starts). Also, understand that concurrency limits are account-wide unless reserved.
