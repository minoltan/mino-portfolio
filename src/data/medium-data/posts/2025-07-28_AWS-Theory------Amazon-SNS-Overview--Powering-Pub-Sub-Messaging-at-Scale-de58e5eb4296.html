Introduction

In modern cloud architectures, real-time communication and event-driven designs are essential to building scalable and reliable applications. Amazon Simple Notification Service (SNS) is a fully managed pub/sub (publish/subscribe) messaging service that enables applications to send notifications to multiple subscribers instantly. Whether it’s sending SMS alerts, triggering Lambda functions, or delivering messages to multiple SQS queues, SNS simplifies event broadcasting and decouples components efficiently. When combined with Amazon SQS, it unlocks powerful fan-out patterns, ensuring reliable message delivery, ordering (with FIFO), and system resilience.

What if you want to send one message to many receivers?

  • The “event producer” only sends message to one SNS topic
  • As many “event receivers” (subscriptions) as we want to listen to the SNS topic notifications
  • Each subscriber to the topic will get all the messages (note: new feature to filter messages)
  • Up to 12,500,000 subscriptions per topic
  • 100,000 topics limit

1. SNS integrates with a lot of AWS services

Many AWS services can send data directly to SNS for notifications

2. Amazon SNS — Security

  • Encryption:
    • In-flight encryption using HTTPS API
    • At-rest encryption using KMS keys
    • Client-side encryption if the client wants to perform encryption/decryption itself
  • Access Controls: IAM policies to regulate access to the SNS API
  • SNS Access Policies (similar to S3 bucket policies)
    • Useful for cross-account access to SNS topics
    • Useful for allowing other services ( S3…) to write to an SNS topic

Real World : Simple SNS: Real-Time SMS Alerts for Banking Transactions

Use Case:
A bank sends SMS notifications to customers for every transaction (debit/credit).
Implementation:

  • Transaction system → SNS Topic → SMS Subscriptions (per customer).
    Why Needed?
  • Customers need immediate alerts for security (fraud detection) and transparency.
  • SNS handles high throughput (millions of SMS/day) and scales automatically.
    Alternatives Without SNS:
  • Building an in-house SMS gateway (expensive, unreliable).

SNS + SQS: Fan Out

  • Push once in SNS, receive in all SQS queues that are subscribers
  • Fully decoupled, no data loss
  • SQS allows for: data persistence, delayed processing and retries of work
  • Ability to add more SQS subscribers over time
  • Make sure your SQS queue access policy allows for SNS to write
  • Cross-Region Delivery: works with SQS Queues in other regions

Application: S3 Events to multiple queues

Real World : SNS + SQS: Fan-Out for E-Commerce Order Processing

Use Case:
An e-commerce site processes orders by triggering multiple downstream systems (inventory, analytics, shipping).
Implementation:

  1. Order service → SNS Topic → Fan-out to:
  • SQS Queue 1: Inventory system (deduplicates orders).
  • SQS Queue 2: Analytics service (logs data).
  • Lambda: Sends customer email confirmations.
    Why Needed?

2. Decouples order service from slow/error-prone subsystems (e.g., inventory DB).

3. No data loss: SQS persists messages if a service fails.
Alternatives Without Fan-Out:

4. Order service calls each subsystem directly (tight coupling, cascading failures).

Amazon SNS — FIFO Topic

  • FIFO = First In First Out (ordering of messages in the topic)
  • Similar features as SQS FIFO:
    • Ordering by Message Group ID (all messages in the same group are ordered)
    • Deduplication using a Deduplication ID or Content Based Deduplication
  • Can have SQS Standard and FIFO queues as subscribers
  • Limited throughput (same throughput as SQS FIFO)

Real world : SNS FIFO Topic: Ride-Sharing Trip Updates

Use Case:
A ride-sharing app (e.g., Uber) updates drivers/passengers with ordered trip events (“Driver assigned”, “En route”, “Completed”).
Implementation:

  1. Trip event service → SNS FIFO Topic (with MessageGroupID=TripID) → Subscribers:
  • Mobile apps (via SNS Mobile Push).
  • Billing system (via SQS FIFO).
    Why Needed?

2. Order matters: Users must see “Driver assigned” before “En route”.

3. FIFO prevents race conditions (e.g., billing before trip ends).
Alternatives Without FIFO:

4. Events arrive out of order, causing app confusion/billing errors.

SNS FIFO + SQS FIFO: Fan Out

  • In case you need fan out + ordering + deduplication

Real World : Financial Trade Settlement

Use Case:
A stock trading platform processes trades in strict order (buy/sell requests for the same stock).
Implementation:

  1. Trade service → SNS FIFO Topic → Fan-out to:
  • SQS FIFO Queue 1: Settlement system (ensures trades settle in order).
  • SQS FIFO Queue 2: Risk analysis (calculates exposure per stock).
    Why Needed?

2. Order + Deduplication: Critical for stock trades (e.g., selling the same shares twice is catastrophic).

3. Fan-out: Multiple systems need the same ordered data.
Alternatives:

4. Single-threaded processing (slow, no parallelism).

SNS — Message Filtering

  • JSON policy used to filter messages sent to SNS topic’s subscriptions
  • If a subscription doesn’t have a filter policy, it receives every message

Real World : Hospital Patient Monitoring

Use Case:
A hospital IoT system sends patient vitals (heart rate, blood pressure) to different departments based on severity.
Implementation:

  1. IoT devices → SNS Topic → Subscriptions with filters:
  • Filter 1: "heartRate > 100" → Emergency team (SMS).
  • Filter 2: "bloodPressure > 140" → Cardiologist (Email).
  • No filter: Log all data to SQS for records. Why Needed?

2. Relevance: Emergency team only sees critical alerts (no noise).

3. Cost Savings: Avoids sending every message to every endpoint.
Alternatives Without Filtering:

4. All departments receive all data (overwhelming, expensive).

Conclusion

Amazon SNS plays a critical role in building event-driven, scalable, and decoupled architectures. Its ability to fan out messages to multiple endpoints, integrate seamlessly with other AWS services, and handle millions of messages per second makes it an ideal choice for real-time notification and event distribution use cases. When paired with SQS, it provides durability, delayed processing, and flexibility to handle failures gracefully. Whether you need to send SMS alerts, process e-commerce orders, or manage IoT events, SNS ensures your messages are delivered quickly, reliably, and at scale — powering modern cloud applications with ease.
